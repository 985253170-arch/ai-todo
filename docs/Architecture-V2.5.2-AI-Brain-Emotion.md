## 一、背景与问题

### 1.1 当前 AI 的"固定按钮响应器"困境

V2.4 和 V2.5 上线后，AI Todo 具备了两个重要的 AI 能力：

| 版本 | 能力 | 交互模式 |
|------|------|----------|
| V2.4 | 任务级 AI 辅助（4 个固定按钮） | 按钮 → 单次输出 |
| V2.5 | 任务执行陪伴模式（5 个固定反馈） | 按钮 → 多轮逐步推进 |

两个能力都已线上验证通过，但暴露了一个共同问题：

**AI 被固定按钮和死板规则锁死了。**

具体表现为：

| # | 问题 | 表现 |
|---|------|------|
| 1 | AI 不知道用户在任务序列的位置 | 用户做到第 3 步和第 1 步，AI 的语气和策略完全一样 |
| 2 | AI 不知道前面完成了什么、后面还剩什么 | 无法给出"还剩两步，坚持一下"或"第一步完成，开个好头"这种有位置感的反馈 |
| 3 | AI 的 4 个 Assist 按钮过于机械 | "怎么开始"永远给 1-3 个起步动作模板，"拆小一点"永远给 3-5 个小步骤模板——不问任务类型、不问位置 |
| 4 | Companion 的鼓励虽然有 4 步结构，但仍然模板化 | 每次"鼓励我一下"都给承认感受→降低压力→缩小范围→给动作，像背稿子 |
| 5 | AI 语气偏冷、偏模板、偏"帮助文档" | 不像行动教练，更像 FAQ 机器人 |
| 6 | AI 输出有时过度保守，有时又过度代写 | Prompt 约束"不替用户完成"是对的，但也导致 AI 不敢给具体建议 |

### 1.2 为什么现在做

| 理由 | 说明 |
|------|------|
| **V2.5.1 已提供了关键基础设施** | `task-execution.ts` 中的 `getCurrentTaskIndex`、`getTaskExecutionStatus`、`isTaskLocked` 已经能精确定位用户在任务序列中的位置——但 AI 完全不知道这些信息 |
| **V2.4/V2.5 的 Prompt 已经线上验证** | Prompt 的基本结构和安全约束是有效的，不需要推翻——只需要升级 |
| **改动面可以非常小** | 不改数据库、不改 AI client、不改 Parser 核心逻辑、不改 UI 布局——主要改动在 Prompt 和 API 请求上下文 |
| **小改动大收益** | 让同样的按钮、同样的 API、同样的 UI 背后，AI 变得"有脑子和有温度" |

### 1.3 核心矛盾

```
当前 AI 能力           vs           产品期望
─────────────────────────────────────────────
固定按钮响应器                      有判断力的行动教练
模板化输出                          自然、像人、有温度
不知道步骤位置                      懂当前执行阶段
单次独立请求                        有上下文感知
机械式鼓励                          有温度的陪伴
过度保守或过度代写                  精准辅助，不越界
```

## 二、V2.5.2 目标

### 2.1 一句话

**让 AI Todo 的 AI 从"固定按钮响应器"升级为"有判断力、有温度、懂当前执行阶段的行动教练"。**

### 2.2 五个核心目标

| # | 目标 | 说明 |
|---|------|------|
| 1 | **AI 理解任务序列上下文** | AI 知道"当前是第几步/共几步"，前面做了什么，后面还剩什么 |
| 2 | **AI 语气更自然、更有温度** | 减少模板感，更像人和人之间的行动对话 |
| 3 | **AI 情感表达有边界** | 有温度但不鸡汤、不心理诊断、不制造依赖 |
| 4 | **AI 辅助更精准** | 根据任务类型、步骤位置、用户状态给出更针对性的帮助 |
| 5 | **固定按钮背后变聪明** | 保留现有按钮（降低用户认知负担），背后 Prompt 变智能 |

### 2.3 不是目标

| # | 不是目标 | 说明 |
|---|----------|------|
| 1 | 不是做自动 Agent | AI 不自动执行任何操作 |
| 2 | 不是做完整聊天系统 | 不做自由聊天输入框 |
| 3 | 不是推翻 V2.4/V2.5 | 在现有基础上增量升级 |
| 4 | 不是做页面重构 | 属于 V3.0 |
| 5 | 不是做长期记忆 | 属于后续版本 |

### 2.4 核心原则

```
Human-in-the-Loop 不动摇：
  用户决定是否完成 → 用户点击完成 → 用户决定是否继续 → 用户决定是否开始新目标

AI 能力边界不写死：
  V2.5.1 只在交互层加约束，V2.5.2 只在 Prompt 和上下文层升级，不限制未来 AI 能力扩展
```

## 三、产品定位与 AI 角色

### 3.1 AI Todo 不是普通 Todo List

| 普通 Todo List | AI Todo |
|---------------|---------|
| 任务平级，随便勾选 | 任务有序，依次执行 |
| 每天新清单 | 跨天持续行动路径 |
| 工具属性 | 教练属性 |
| 用户自己规划 | AI 生成有序行动步骤 + 陪执行 |
| AI 可无 | AI 是核心差异化 |

### 3.2 AI Todo 的 AI 不是普通聊天机器人

| 普通聊天机器人 | AI Todo 的 AI |
|---------------|---------------|
| 什么都能聊 | 只在任务上下文中工作 |
| 无限对话 | 有限交互（按钮驱动） |
| 无责任边界 | 明确的教练伦理边界 |
| 可替代用户做事 | 只辅助，不替代 |
| 追求对话长度 | 追求行动推进 |

### 3.3 AI 的角色定义

AI Todo 的 AI 扮演以下角色：

| 角色 | 职责 | 场景 |
|------|------|------|
| **行动教练** | 帮用户看清"现在做什么、怎么开始" | 用户面对任务无从下手 |
| **陪伴者** | 在用户执行过程中持续陪伴、不催促 | 用户一步一步推进任务 |
| **拆解助手** | 把大任务拆成小动作，降低执行门槛 | 任务太大、太模糊 |
| **执行引导者** | 基于任务序列位置，引导用户按路径推进 | 顺序执行中的每一步 |
| **情绪缓冲器** | 在用户卡住/气馁/拖延时，降低压力、缩小任务 | 用户情绪阻力 |

### 3.4 AI 不是

| # | 不是 | 为什么 |
|---|------|--------|
| 1 | 不是自动执行 Agent | 不自动勾选、不自动创建、不自动发送 |
| 2 | 不是心理咨询师 | 不做心理诊断、不挖情绪根源 |
| 3 | 不是代写机器 | 不生成完整文章/邮件/报告/代码 |
| 4 | 不是无限聊天机器人 | 没有自由聊天入口，交互由按钮驱动 |
| 5 | 不是项目管理机器人 | 不做甘特图、不做排期、不做依赖管理 |

## 四、现有 AI 能力分析

### 4.1 V2.4 task-assist 当前能力边界

**文件**：[src/prompts/task-assist.ts](../src/prompts/task-assist.ts) · [src/app/api/task-assist/route.ts](../src/app/api/task-assist/route.ts)

#### 4.1.1 当前能力

| 按钮 | AI 行为 | 输入信息 | 输出格式 |
|------|---------|----------|----------|
| 怎么开始 | 给 1-3 个起步动作 | taskTitle + goal + actionType | "你现在只需要做这几步：1. ... 2. ... 3. ..." |
| 拆小一点 | 拆成 3-5 个小步骤 | taskTitle + goal + actionType | "可以先拆成这些小步骤：1. ... 2. ... 3. ..." |
| 5 分钟版本 | 给一个 5 分钟极简版 | taskTitle + goal + actionType | "5 分钟版本：... 具体做法：..." |
| 我卡住了 | 分析卡点 + 给下一步 | taskTitle + goal + actionType | "你可能卡在：... 下一步：..." |

#### 4.1.2 四个固定按钮是否过窄

**结论：按钮本身不窄，但按钮背后的 AI 推理太窄。**

| 维度 | 当前状态 | 问题 |
|------|----------|------|
| 按钮覆盖的场景 | ✅ 起步/拆解/极简/卡点——4 个核心场景覆盖到位 | 不需要改按钮 |
| 按钮对应的 Prompt | ❌ 每个按钮有固定输出模板 | AI 像填模板，不像思考 |
| AI 上下文 | ❌ 只知道 taskTitle 和 goal | 不知道用户在序列中的位置、不知道前一步/后一步 |
| AI 语气 | ❌ 偏冷、偏"帮助文档" | 不像教练，像 FAQ |

#### 4.1.3 task-assist Prompt 是否过度限制 AI

**当前 System Prompt**：

```
1. 只返回纯文本，不返回 JSON，不返回代码块，不输出 Markdown 长文。
2. 输出必须简短、具体、可执行，让用户读完就能立刻开始做。
3. 不超过 5 个步骤。
4. 不说空泛鸡汤，不使用"相信自己""你一定可以"这类表达。
5. 不自动替用户完成任务，不修改任务，不生成新任务。
6. 不让用户安装工具、购买工具或访问外部网站，除非当前任务明确需要。
7. 不要使用复杂术语。
```

**分析**：

| 规则 | 评价 | 建议 |
|------|------|------|
| 规则 1（纯文本） | ✅ 保留 | 纯文本是最稳定的交互格式 |
| 规则 2（简短可执行） | ✅ 保留 | 核心价值 |
| 规则 3（≤5 步骤） | ⚠️ 合理但有争议 | 对于拆解类按钮合理，但对"怎么开始"可能太机械 |
| 规则 4（不鸡汤） | ✅ 保留 | 但需要补充"可以有温度"的许可 |
| 规则 5（不代完成） | ✅ 保留 | Human-in-the-Loop 底线 |
| 规则 6（不推工具） | ✅ 保留 | 保持简单 |
| 规则 7（不复杂术语） | ✅ 保留 | 保持可读性 |

**总体判断**：当前 Prompt 的安全约束是正确且必要的，但它是一个"禁止清单"而非"能力指南"。AI 被告诉"不能做什么"但没被告诉"能做到什么程度、可以多有人味"。

### 4.2 V2.5 companion 当前能力边界

**文件**：[src/prompts/task-companion.ts](../src/prompts/task-companion.ts) · [src/app/api/task-companion/route.ts](../src/app/api/task-companion/route.ts)

#### 4.2.1 当前能力

| 反馈信号 | AI 行为 | 输入信息 | 输出约束 |
|----------|---------|----------|----------|
| start | 给第一步，降低门槛 | taskTitle + goal + currentStep + stepHistory | 80-150 字，≤3 动作 |
| done | 认可完成 + 给下一步 | 同上 | 同上 |
| stuck | 承认卡住 + 拆更小 | 同上 | 同上 |
| too_hard | 承认难度 + 降级方案 | 同上 | 同上 |
| encourage | 4 步鼓励结构 | 同上 | 同上 |

#### 4.2.2 五个反馈按钮是否过窄

**结论：5 个按钮覆盖了核心陪伴场景，但和 V2.4 一样——按钮背后的 AI 推理太窄。**

| 维度 | 当前状态 | 问题 |
|------|----------|------|
| 按钮覆盖的场景 | ✅ start/done/stuck/too_hard/encourage 覆盖到位 | 不需要改按钮 |
| AI 上下文 | ⚠️ 有 stepHistory（比 assist 好），但无序列上下文 | 不知道"第几步/共几步" |
| AI 语气 | ⚠️ 比 assist 好——有 4 步鼓励结构 | 但仍然模板化，每次都按同一个框架输出 |
| AI 温度 | ⚠️ 鼓励规则有效但不自然 | 需要更像人，更少像背稿子 |

#### 4.2.3 task-companion Prompt 是否过度模板化

**当前 Companion System Prompt** 的核心结构：

```
核心规则（8 条禁止/约束）
→ 用户反馈处理（5 种信号的处理模板）
→ 鼓励规则（4 步结构模板）
→ 收尾判断
→ 输出格式
```

**分析**：

| 维度 | 当前状态 | 问题 |
|------|----------|------|
| 结构 | ✅ 完整、清晰 | 但像"操作手册"，AI 按手册执行 |
| 鼓励规则 | ✅ 4 步结构有效防止鸡汤 | 但 AI 每次都 1→2→3→4 线性执行，缺乏变化 |
| 用户反馈处理 | ✅ 5 种信号各有模板 | 模板化导致输出缺乏针对性和温度 |
| 缺少 | ❌ 序列上下文 | AI 不知道用户做到第几步 |
| 缺少 | ❌ 情感表达的"许可" | Prompt 只说了"不能做什么"，没说"可以多有人味" |

### 4.3 Parser 对纯文本输出的限制是否合理

**文件**：[src/lib/task-assist-parser.ts](../src/lib/task-assist-parser.ts) · [src/lib/task-companion-parser.ts](../src/lib/task-companion-parser.ts)

**分析**：

| 维度 | 当前状态 | 评价 |
|------|----------|------|
| 输出类型 | 纯文本（string） | ✅ 正确——纯文本最稳定、最易展示、最易复制 |
| JSON 兼容 | assist-parser 做了 JSON unwrap fallback | ✅ 合理——防御性处理 |
| [DONE] 检测 | companion-parser 按行检测 `[DONE]` | ✅ 合理 |
| 截断长度 | assist: 500 字, companion: 300 字 | ✅ 合理 |
| Markdown 清理 | 两端都清理 `#` `**` `__` 等 | ✅ 合理 |

**结论**：Parser 的设计合理。纯文本输出的好处远大于结构化输出（用户不需要 JSON，前端不需要渲染复杂格式）。V2.5.2 **不改 Parser 核心逻辑**。

### 4.4 task-execution 的 current/locked 信息是否可以成为 AI 上下文

**文件**：[src/lib/task-execution.ts](../src/lib/task-execution.ts)

**绝对可以，而且这正是 V2.5.2 的核心价值。**

当前 `task-execution.ts` 提供了：

| 函数 | 信息 | 对 AI 的价值 |
|------|------|-------------|
| `getCurrentTaskIndex(tasks)` | 当前是第几步（0-based index） | AI 可以说"你在第 3 步，还剩 2 步" |
| `getTaskExecutionStatus(index, tasks)` | completed / current / locked | AI 知道当前任务的状态 |
| `isTaskLocked(index, tasks)` | 某任务是否 locked | AI 不会建议用户跳过 locked 任务 |
| `hasIncompleteTasks(tasks)` | 是否有未完成任务 | AI 可以判断"还剩多少" |
| `shouldCarryOverTaskGroup(tg)` | 是否跨天继承 | AI 可以感知"继续昨天的任务" |

这些信息**完全来自前端推导**，不需要 API 访问数据库——只需要前端在请求 AI 时把这些信息传过去。

### 4.5 当前 AI 不知道什么（逐项确认）

| # | 问题 | 答案 |
|---|------|------|
| 1 | AI 是否知道用户处于第几步？ | ❌ **不知道**。assist API 只传 taskTitle + goal + actionType。companion API 只传 taskTitle + goal + currentStep + stepHistory。都不传 currentIndex / totalTasks。 |
| 2 | AI 是否知道前一步 / 后一步？ | ❌ **不知道**。API 请求中没有 previousTaskTitle / nextTaskTitle 字段。 |
| 3 | AI 是否能表达适当情绪温度？ | ⚠️ **有限**。companion 有鼓励规则，但模板化。assist 完全无温度。 |
| 4 | AI 是否会过度代写？ | ⚠️ **有时**。assist 的"拆小一点"和"5 分钟版本"偶尔给出接近完整方案的内容。Prompt 有约束但不够精细。 |
| 5 | AI 是否会过度保守？ | ⚠️ **有时**。"不替用户完成"的约束有时导致 AI 不敢给具体建议。 |
| 6 | AI 是否知道任务列表中有 locked 任务？ | ❌ **不知道**。API 完全不感知顺序执行状态。 |
| 7 | AI 是否会建议用户跳过任务？ | ⚠️ **可能**。由于不知道 locked 状态，AI 可能给 locked 任务提出推进建议。但 locked 任务不显示 AI 入口（V2.5.1 已修复），所以实际不会触发。这个问题已被 V2.5.1 的 UI 层解决。 |

## 五、AI 大脑层设计

### 5.1 概念：AI Brain Context Layer

V2.5.2 引入一个轻量的 **AI Brain Context Layer**（AI 大脑上下文层），不新增服务、不新增数据库、不新增 API Route。

它是一个概念层，由两个具体实现组成：

```
┌────────────────────────────────────────────────┐
│              AI Brain Context Layer              │
│                                                  │
│  ┌──────────────────────┐  ┌──────────────────┐ │
│  │ ai-brain-context.ts  │  │ 升级后的 Prompt   │ │
│  │ (纯函数，构造上下文)  │  │ (使用上下文)      │ │
│  └──────────────────────┘  └──────────────────┘ │
│                                                  │
│  输入：tasks, taskIndex, goal, userSignal       │
│  输出：结构化的 AI 上下文对象                     │
│  用途：Assist 和 Companion 共用                   │
└────────────────────────────────────────────────┘
```

### 5.2 Task Context（任务序列上下文）

这是 V2.5.2 最核心的新信息。AI 从未见过的上下文：

```typescript
interface TaskSequenceContext {
  /** 当前任务在任务列表中的位置（1-based，人类可读） */
  currentStepNumber: number;        // 如：3（第 3 步）
  /** 任务总数 */
  totalSteps: number;               // 如：5
  /** 已完成任务数量 */
  completedSteps: number;           // 如：2
  /** 当前任务执行状态 */
  executionStatus: "current";       // 一定是 current（只有 current 任务才能触发 AI）
  /** 前一步任务标题（可能不存在） */
  previousTaskTitle?: string;       // 如："整理会议纪要"
  /** 后一步任务标题（可能不存在） */
  nextTaskTitle?: string;           // 如："回复客户邮件"
  /** 是否第一步 */
  isFirstStep: boolean;
  /** 是否最后一步 */
  isLastStep: boolean;
  /** 剩余步骤数 */
  remainingSteps: number;           // 如：2
}
```

**这些信息全部来自前端已有数据，零成本获取。**

### 5.3 User Signal（用户信号）

保留现有按钮体系，但让 AI 能感知"用户为什么按这个按钮"：

```typescript
interface UserSignalContext {
  /** 用户点击的按钮类型 */
  signal: "how_to_start" | "break_down" | "five_minute" | "im_stuck"
        | "start" | "done" | "stuck" | "too_hard" | "encourage";
  /** 用户是否卡住了（从 signal 推导） */
  isStuck: boolean;                 // signal === "im_stuck" || signal === "stuck"
  /** 用户是否觉得太难 */
  isOverwhelmed: boolean;           // signal === "too_hard"
  /** 用户是否需要鼓励 */
  needsEncouragement: boolean;      // signal === "encourage"
  /** 用户是否刚完成一步 */
  justCompletedStep: boolean;       // signal === "done"
  /** 用户是否刚开始 */
  isJustStarting: boolean;          // signal === "start" || signal === "how_to_start"
}
```

### 5.4 Coach Policy（教练策略）

嵌入在 Prompt 中的隐性策略，不单独抽成代码：

```
教练策略（嵌入 System Prompt）：

1. 每次只推动当前这一步
   - 不给完整计划，不展望后续步骤
   - 用"现在只需要做..."而不是"你需要做 1. 2. 3. 4. 5."

2. 允许解释为什么做这一步
   - "这一步很重要，因为..."
   - 但不超过一句话

3. 允许给更小的动作
   - "如果觉得这一步还是太大，可以先只做..."
   - 把"打开文档"拆成"打开电脑 → 打开软件 → 新建文件"

4. 允许给模板，但不能替用户完成最终成果
   - ✅ "周报通常包含这三块：本周完成 / 下周计划 / 需要协调。你可以先写'本周完成'部分。"
   - ❌ "以下是你本周的周报：[完整周报内容]"

5. 允许情绪回应，但不能心理诊断
   - ✅ "这一步确实容易让人觉得无从下手，很多人都有同感。"
   - ❌ "你听起来很焦虑，这种焦虑可能来自于..."

6. 根据步骤位置调整策略
   - 第一步：降低门槛，小动作启动
   - 中间步：保持节奏，认可推进
   - 最后一步：提醒收尾，鼓励完成
   - 唯一步：不用强调顺序，聚焦任务本身
```

### 5.5 Output Style（输出风格）

从"帮助文档"风格升级为"行动教练"风格：

| 维度 | 当前风格 | 目标风格 |
|------|----------|----------|
| **温度** | 冷、中性、模板化 | 温暖、自然、像人和人对话 |
| **长度** | 80-150 字（companion）、不定（assist） | 保持简短，但允许根据任务复杂度略微浮动 |
| **人称** | 第三人称 / 无称呼 | "你"——直接对用户说话 |
| **结构** | 模板化（"你现在只需要做这几步：1. 2. 3."） | 自然语言，减少编号，用"先...然后...最后..." |
| **模板感** | 每次格式完全一致 | 每次表达不同，但结构清晰 |
| **鸡汤感** | 无（成功避免了） | 持续保持无鸡汤，但增加认同感 |

**风格对比示例**：

当前：
> 你现在只需要做这几步：
> 1. 打开文档工具
> 2. 新建一个文件
> 3. 标题写"项目周报"

目标：
> 先从最简单的一步开始——打开你常用的文档工具，新建一个文件，标题写上"项目周报 - 本周"。就这一件事，做完就可以先停下来。

区别：
- 不是"你需要做"而是"先从...开始"
- 不是编号列表而是自然语句
- 明确说"做完就可以先停下来"——降低心理负担
- 有温度但不鸡汤

## 六、情感表达边界

### 6.1 总则

**AI 可以有温度，但不能变鸡汤。AI 可以共情，但不能做心理诊断。**

### 6.2 可以表达的情感

| 情感维度 | 许可范围 | 示例 |
|----------|----------|------|
| **认同** | 承认任务的难度和用户的感受 | "这一步确实容易卡住，不是你一个人的问题。" |
| **鼓励** | 基于行动的轻量鼓励 | "第一步完成，已经开了个好头。" |
| **降低压力** | 明确说"不需要一次做完" | "不需要现在就做到完美，先有一个版本出来就行。" |
| **认可进展** | 认可用户完成了一步 | "很好，这一步完成。离目标又近了一点。" |
| **陪伴感** | 让用户感觉有人在旁边 | "我陪你把这一步拆开看看。" |

### 6.3 不能表达的情感

| # | 禁止 | 原因 | 示例（❌ 不能说） |
|---|------|------|------------------|
| 1 | 不做心理诊断 | 不是心理咨询工具 | "你看起来很焦虑"、"你可能在逃避" |
| 2 | 不做情绪分析 | 不给用户贴标签 | "你属于完美主义倾向"、"你有拖延症" |
| 3 | 不写长篇鸡汤 | 用户需要下一步动作 | "相信自己！你一定可以的！💪" |
| 4 | 不说空泛鼓励 | 缺乏可操作性 | "加油"、"坚持就是胜利" |
| 5 | 不制造依赖 | 不让用户觉得"不用 AI 就不行" | "没有我你可能做不好这个" |
| 6 | 不说教 | 不教育用户"应该怎样" | "你应该学会时间管理"、"正确的方法是..." |
| 7 | 不制造焦虑 | 不暗示后果 | "如果你不完成这一步，后面会..." |
| 8 | 不夸张赞美 | 不夸大进展 | "太棒了！你真是天才！" |

### 6.4 鼓励分级标准

鼓励的深度根据场景分级，不是每次都走完整 4 步结构：

| 场景 | 鼓励深度 | 结构 |
|------|:--:|------|
| 用户刚完成一步 | 轻量 | 认可完成 + 自然过渡到下一步 |
| 用户主动请求鼓励 | 标准 | 承认感受 → 降压力 → 缩任务 → 给动作（4 步） |
| 用户连续卡住 | 深度 | 4 步 + 提供替代方案 / 可选路径 |
| 用户即将完成最后一步 | 轻量 | 认可即将完成 + 提醒收尾 |

## 八、固定按钮与自由输入策略

### 8.1 评估：按钮不是问题

当前固定按钮的设计**不是错误**。理由：

| # | 理由 | 说明 |
|---|------|------|
| 1 | 按钮降低了用户认知负担 | 用户不需要想"我要怎么问 AI"——按钮已经给出了选择 |
| 2 | 按钮防止了聊天化 | 没有输入框，用户不会把 AI 当聊天机器人 |
| 3 | 按钮覆盖了核心场景 | 起步/拆解/极简/卡点/完成/鼓励——覆盖了任务执行的主要需求 |
| 4 | 按钮便于移动端操作 | 比打字快，适合手机场景 |

**按钮的问题是"按钮背后太死板"，不是"按钮本身有错"。**

### 8.2 五种候选方案

#### 方案 A：只升级 Prompt（零 UI 变更）

- 不改 UI
- 不改 API 请求格式
- 只改 `task-assist.ts` 和 `task-companion.ts` 的 Prompt
- 让 AI 在现有约束下变聪明变温暖

✅ 最小风险
❌ 不传序列上下文 → AI 仍然不知道步骤位置

#### 方案 B：升级 Prompt + 传入任务序列上下文（推荐）

- API 请求增加可选字段：`currentStepNumber` / `totalSteps` / `previousTaskTitle` / `nextTaskTitle`
- 前端从 `tasks[]` + `taskIndex` 提取这些信息
- Prompt 使用这些信息生成有位置感的输出
- 不改 UI 布局、不改按钮

✅ AI 真正变聪明
✅ 风险可控
✅ 改动集中在 Prompt + API 请求体

#### 方案 C：新增轻量自由输入（"补充一句"）

- 在 Assist / Companion 面板中新增一个可选的小输入框
- 用户可以在按钮之外补充一句话（如"这个我上周做过"、"时间不够"）
- AI 把这句话作为额外上下文
- 但不做成聊天框——仍然是一次性请求

✅ 给了用户更多表达空间
❌ 引入输入框 → 增加 UI 复杂度
❌ 增加聊天化风险
❌ 增加 Prompt 复杂度（需要处理额外自由文本）

#### 方案 D：按钮 + 可选补充一句（方案 B + 方案 C 的轻量版）

- 在按钮旁边加一个很小的可选输入："（可选）说一下当前情况"
- 不超过 50 字
- 只作为额外上下文，不改变交互模式

#### 方案 E：完整聊天系统

❌ 直接排除——违反产品定位，属于不做范围

### 8.3 推荐策略

**V2.5.2 采用方案 B**（升级 Prompt + 传入任务序列上下文）：

1. **保留现有按钮**：4 个 Assist 按钮 + 5 个 Companion 反馈按钮——一个不改
2. **不做自由输入**：不新增聊天框、不新增输入框
3. **按钮背后变智能**：同一个按钮在不同任务类型、不同步骤位置上，AI 给出不同风格和深度的输出
4. **序列上下文由前端自动传入**：用户无感，不需要额外操作

**方案 D（按钮 + 可选补充一句）留到 V2.5.3 或 V2.6 评估**，在方案 B 验证有效后再考虑。

## 九、与 V2.5.1 顺序执行的关系

### 9.1 V2.5.1 为 V2.5.2 铺的路

| V2.5.1 提供的能力 | V2.5.2 如何使用 |
|-------------------|----------------|
| `getCurrentTaskIndex(tasks)` | AI 知道"当前是第几步" |
| `getTaskExecutionStatus(index, tasks)` | AI 知道"这是 current 任务" |
| `isTaskLocked(index, tasks)` | AI 知道哪些任务不能碰 |
| `TaskItem` 已有 `taskIndex` prop | 前端可直接提取序列信息传给 API |
| `TaskList` 已有 `tasks` 数组 + `index` | 前端可直接计算 `totalSteps` / `previousTaskTitle` / `nextTaskTitle` |

### 9.2 V2.5.2 不破坏 V2.5.1

| V2.5.1 规则 | V2.5.2 行为 |
|-------------|------------|
| locked 任务不可勾选 | AI 入口只在 current 任务显示 → AI 不会收到 locked 任务请求 |
| 用户不能跳过前置任务 | AI 不会建议用户跳过 locked 任务（因为根本不会为 locked 任务触发 AI） |
| 顺序执行：当前 step → 完成 → 下一个解锁 | AI 感知到步骤推进，可以给出位置相关的反馈 |
| 跨天未完成任务继承 | AI 可以感知 carryover 上下文 |

### 9.3 V2.5.2 增强但不越界

```
V2.5.1（交互层约束）          V2.5.2（AI 层增强）
─────────────────────        ─────────────────────
locked 不可操作               AI 知道有 locked 任务 → 但不提
current 是唯一可操作          AI 聚焦 current
AI 入口只在 current           AI 收到序列上下文
顺序自然推进                  AI 能感知推进
```

**关键原则**：
- AI 可以知道后面还有什么任务，但**不能让用户跳过去**
- AI 可以鼓励用户"先完成当前这一步"，但**不能催促"快点做完"**
- AI 可以感知"还剩 2 步"，但**不能制造焦虑"你还有很多没做"**

## 十三、风险与缓解

### 13.1 P0 — 阻塞

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|----------|
| **P0-1** | Prompt 升级后 AI 输出变长、变空洞 | AI 辅助质量下降 | 保留"简短、具体、可执行"约束 + maxTokens 不变 + parser 截断不变 |
| **P0-2** | AI 情感表达过度，像心理咨询 | 产品定位崩塌 | Prompt 保留禁止心理诊断的硬约束 + 明确"你不是心理咨询师" |
| **P0-3** | AI 基于序列上下文鼓励用户跳过 locked 任务 | 破坏 V2.5.1 顺序执行 | Prompt 增加规则："locked 任务不可被提及和建议" |
| **P0-4** | API 请求体新增字段导致旧客户端报错 | 兼容性问题 | 所有新字段设为可选（`?:`），API route 做 undefined 兜底 |
| **P0-5** | 前端传错 currentStepNumber | AI 理解错步骤位置，给出误导建议 | 序列上下文从 `TaskList` 的 `tasks.map` 中直接计算，不经过 state，确保准确性 |
| **P0-6** | AI 过度代写 | Human-in-the-Loop 被破坏 | Prompt 保留"不替用户完成完整任务"的硬约束 |

### 13.2 P1 — 严重影响体验

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|----------|
| **P1-1** | AI 变得太自由，输出不可控 | 用户体验不一致 | 保留全部安全约束 + 保留输出长度限制 + temperature 保持 0.4 |
| **P1-2** | AI 太热情，给用户压力 | "教练"变成"催促者" | Prompt 增加："你是陪伴者，不是催促者。用户有自己的节奏。" |
| **P1-3** | 序列上下文让 AI 过度关注"还剩多少" | 制造完成焦虑 | Prompt 规则："不要强调剩余步骤数，除非用户主动问" |
| **P1-4** | Prompt 太长导致 AI 忽略关键约束 | 安全约束失效 | 把安全约束放在 Prompt 最前面 + 用"禁止"列表强化 |
| **P1-5** | 上下文字段增加导致 token 消耗增加 | API 成本略增 | 序列上下文 ~100 tokens——影响极小。maxTokens 不变控制上限 |

### 13.3 P2 — 可接受

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|----------|
| **P2-1** | 纯文本输出偶尔带格式标记 | 展示微瑕 | parser 已有 Markdown 清理，问题不大 |
| **P2-2** | 不同 DeepSeek 版本对 Prompt 理解有差异 | 输出质量波动 | 上线后监控并迭代 Prompt |
| **P2-3** | 方案 B 需要改 API 请求体 | 改动略大于方案 A | API route 改动 ~20 行/文件，风险可控 |
| **P2-4** | 未来的 V2.5.3 可能还需要再调 | 迭代成本 | 这是正常的——Prompt 调优是持续过程 |

## 十五、验收标准

### 15.1 AI 语气与温度

| # | 验收项 | 验证方式 |
|---|--------|----------|
| **T1** | AI 语气更自然，不像背模板 | 多次触发同一按钮，输出应有表达变化 |
| **T2** | AI 能体现当前任务序号 | 输出中出现"第一步"、"最后一步"等位置感知（如果序列上下文传入） |
| **T3** | AI 不输出空泛鸡汤 | 多次"鼓励我一下"，不出现"你一定可以的"、"相信自己"、"你很棒" |
| **T4** | AI 不做心理诊断 | 多次陪伴，不出现"你看起来焦虑"、"你可能在逃避"等诊断语句 |
| **T5** | AI 不过度代写 | 不输出完整周报/邮件/文章 |

### 15.2 AI 辅助能力

| # | 验收项 | 验证方式 |
|---|--------|----------|
| **A1** | 卡住时能给更小动作 | "我卡住了" → AI 给出不多于 2 个动作 |
| **A2** | 太难时能降级任务 | "太难了" → AI 给出简化方案 |
| **A3** | 鼓励时有温度但不过度 | "鼓励我一下" → 有温度 + 有具体动作 + 无鸡汤 |
| **A4** | AI 不建议跳过 locked 任务 | 不会出现"你可以先做后面的" |
| **A5** | AI 不自动勾选任务 | 不会调用 toggle API |
| **A6** | AI 不自动创建任务 | 不会调用 generate API |
