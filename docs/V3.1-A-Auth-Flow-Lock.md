## 2. 已有账号验证码登录

```text
输入邮箱
→ sendOtp intent = sign-in
→ shouldCreateUser = false
→ 输入六位验证码
→ verifyOtp
→ 已设置密码：进入 AppShell
→ 未设置密码：进入首次设置密码门禁
```

- 未知邮箱不得在登录流程中静默创建账号。
- 页面应以温柔文案引导用户进入显式注册页。
- 发送成功后，同一认证卡片切换到验证码输入状态：脱敏邮箱、6 位验证码输入、重新发送倒计时和更换邮箱入口。
- 验证码固定为 6 位数字；错误、过期、网络失败和重复请求均须通过统一 Auth facade 映射为温和、可行动的提示。

---

## 3. 显式注册

```text
从登录页进入 RegisterPage
→ sendOtp intent = sign-up
→ shouldCreateUser = true
→ 输入六位验证码
→ verifyOtp
→ Session 建立
→ 首次设置密码
→ 进入 AppShell
```

- 注册与登录共用 OTP 能力和验证码视觉交互，但必须传递不同 intent。
- Session 建立后立即进入首次设置密码门禁；不可通过系统返回绕过。
- 已有账号的注册冲突遵循 Architecture 锁定的安全、温和引导；不得通过额外探测枚举账号。

## 4. 首次设置密码与密码登录

### 4.1 首次设置密码

OTP 登录、注册 OTP 验证或启动恢复后，若 `password_set !== true`：

```text
authenticated-needs-password
→ 首次设置密码门禁
→ 设置成功
→ authenticated
→ AppShell
```

首次设置密码不是可跳过推荐项。用户仅可明确选择“退出当前账号”，且退出使用 local signOut。

### 4.2 密码登录

```text
密码登录成功
→ 直接 authenticated
→ AppShell
```

`password_set` metadata 仅为产品体验标记：密码登录成功后可 best-effort 补写 `password_set:true`；补写失败不阻断进入 AppShell、不强制再次设置密码，也不得将用户降级为 `authenticated-needs-password`。

邮箱验证码登录和邮箱密码登录长期并存。

## 5. 退出当前账号

“退出当前账号”固定为 local signOut：

```ts
auth.signOut({ scope: "local" })
```

- 仅退出当前浏览器 Session；
- 同一浏览器中共享该 Session 的标签页通过 `SIGNED_OUT` 收敛为 guest；
- 其他浏览器 Profile 的独立 Session 不受影响；
- 其他设备上的独立 Session 不受影响；
- 本阶段不实现“退出所有设备”，不得默认使用 global scope 或未指定 scope 的 `auth.signOut()`。

---

## 6. 页面与 Service 边界

- 页面不得直接调用 Supabase；只通过统一 auth service facade 接收 action 与状态；
- 保留 `authService.mock.ts`，新增 Mock adapter 与 Real adapter，不删除或覆盖 Mock source；
- Session 状态由 [page.tsx](../apps/mobile-app/app/page.tsx) 的统一认证层管理；
- 客户端不得暴露 service role key；不得擅自修改数据库 schema、RLS 或根 API Route；
- Auth 页面维持既有 AuthShell、单一视觉结构和无 BottomTabBar 的约束；不新增第二套认证页面或子路由。

## 7. 明确不做

V3.1-A 不做：

- 忘记密码或密码重置页面；
- 手机号、OAuth、微信、QQ、Apple、Google 登录；
- MFA、多账号切换、复杂账号中心、管理员体系；
- 会员、支付、客服聊天；
- 数据库 schema、RLS、根 API Route 修改；
- 删除 `authService.mock.ts`；
- 重做 Auth 视觉或新增 Auth 子路由；
- 真实任务、历史、成长、AI 或任务同步。
