## 一、背景与问题

### 1.1 V2.5.3 达成了什么

V2.5.3 完成了 Promp 帮助边界修正——方案 A（只改 Prompt），只修改了 2 个 Prompt 文件：

| 成果 | 说明 |
|------|------|
| AI 不再默认让用户去知乎/百度/小红书搜索 | 安全红线第 8 条 + 帮助边界分节明确禁止 |
| AI 可以主动给材料、框架、清单、低风险草稿 | 11 类材料分类写入 Prompt |
| AI 不替用户完成最终责任动作 | "不能替用户完成什么"分节 7 条规则 |
| 帮助边界从"太窄"修正为"精准" | 五类答案边界（Type 1-5） |

V2.5.3 让 AI 从"搜索引擎代理"升级为"材料提供者"——这是**质的飞跃**。

### 1.2 V2.5.3 暴露的新问题

V2.5.3 上线后，AI 的行为模式发生了根本变化：

**AI 现在会说**：
> "不用去搜，我先给你 5 个常见面试问题。你现在只选第 1 个，写三行：项目背景、你做了什么、结果是什么。写出来告诉我。"

**但问题来了**：用户写出来后，"告诉"AI 的通道不存在。

当前 TaskCompanionPanel 只有 4 个固定按钮：

```
[我完成了]  [我卡住了]
[太难了]    [鼓励我一下]
```

这 4 个按钮**无法表达**：
- "我写了一句话，你看看行不行"（草稿反馈）
- "我只完成了一半"（部分进展）
- "我不知道这样算不算完成"（验收请求）
- "这是我的草稿"（贴一小段内容）
- "我今天只有 3 分钟"（时间约束）
- "这个任务太大了"（粒度反馈）

### 1.3 核心矛盾

```
V2.5.3 让 AI 能"给材料"        →  但用户无法把"材料使用结果"反馈给 AI
AI 说"写出来告诉我"             →  但界面上没有"告诉 AI"的通道
AI 说"写完这一行就可以先停"      →  但用户停在哪、写成什么样，AI 完全不知道
AI 说"只选第 1 个准备"          →  但用户选没选、准备成什么样，AI 无法判断
```

**V2.5.3 把 AI 的输出能力修好了，但用户→AI 的输入通道仍然是 4 个固定按钮。**

这就是 V2.6 要解决的核心问题。

## 三、核心产品判断

### 判断 1：当前固定按钮陪伴模式已经到达上限

V2.5.2 引入陪伴模式时，5 个固定按钮（start / done / stuck / too_hard / encourage）覆盖了基本场景。但 V2.5.3 让 AI 从"给指令"升级为"给材料"后，用户的行为也变了——用户不再只是"做任务"，而是会"写草稿"、"写一半"、"不确定够不够"。

**4 个固定按钮无法表达用户在做材料相关任务时的真实状态。**

固定按钮是 V2.5.2 时代的产品设计——那时 AI 只给指令，用户只需要反馈"做了没"。现在 AI 给材料、给框架、给示例，用户需要的反馈通道远比 4 个按钮丰富。

### 判断 2：AI 现在能给材料，但用户无法把真实反馈交给 AI

这是一个**单向通道问题**：

```
V2.5.2：AI → 指令 → 用户自己去搜索 → 用户回到 App 点"我完成了"
V2.5.3：AI → 材料 → 用户在 App 内使用材料 → 用户无法反馈材料使用结果
```

V2.5.3 的改进让用户留在了 App 内（不再跳去搜索），但也产生了新的需求——用户需要一个方式把"材料使用结果"告诉 AI。

### 判断 3：4 个固定按钮无法表达的 7 种真实反馈

| # | 用户想表达的 | 固定按钮能表达吗？ |
|---|------------|:--:|
| 1 | "我写了一句话，你看看行不行" | ❌ |
| 2 | "我只完成了一半" | ❌（点"我完成了"不准确，点"我卡住了"也不准确） |
| 3 | "我不知道这样算不算完成" | ❌ |
| 4 | "这是我的草稿" | ❌ |
| 5 | "我今天只有 3 分钟" | ❌ |
| 6 | "这个任务太大了" | ❌（点"太难了"部分接近，但不够精确） |
| 7 | "我做了一部分，剩下不太确定" | ❌ |

### 判断 4：因此必须引入"任务内受控反馈框"

这不是做聊天软件。这是给当前任务增加一条"用户→AI"的反馈通道。

**反馈框的能力边界**：
- ✅ 让用户输入当前进展、卡点、草稿、时间约束、完成度判断
- ❌ 不做全局自由聊天
- ❌ 不做脱离当前任务的闲聊
- ❌ 不保存长期对话记忆

### 判断 5：这个输入框不是 ChatGPT 式聊天框

关键区别：

| ChatGPT 式聊天 | 任务内受控反馈框 |
|--------------|---------------|
| 全局对话 | 绑定当前任务 |
| 无限轮次上下文 | 当前任务 + 最近几步 |
| 可聊任何话题 | 只能反馈当前任务执行状态 |
| 历史对话长期保留 | 任务完成后清空 |
| 用户可要求 AI 做任何事 | 安全红线仍然生效 |

### 判断 6：它是当前任务的反馈通道，用于执行、验收、鼓励、调整

反馈框的四个核心用途：
1. **执行反馈**："我做了这一步，你看看"
2. **质量验收**："这样算完成吗？"
3. **鼓励融入**：不再需要单独点"鼓励我一下"
4. **难度调整**："太大了"、"今天只有 3 分钟"

### 判断 7："鼓励我一下"按钮可以删除

**理由**：
- V2.5.3 已经把"主动鼓励"内置到了 stuck / too_hard / done / start 中
- V2.6 有了反馈输入框后，用户输入"I'm stuck"或"写不出来"时，AI 自然应该回应鼓励+材料+动作
- 独立的"鼓励我一下"按钮把鼓励变成了用户主动索取的行为——不自然
- 鼓励应该是 AI 对用户状态的回应，不是用户按按钮才能得到的"功能"

## 五、V2.6：任务内受控反馈框与 AI 验收机制

### 5.1 名称

**V2.6：任务内受控反馈框与 AI 验收机制**

### 5.2 一句话目标

让用户能把当前进展、草稿、卡点反馈给 AI，让 AI 基于反馈继续推进当前任务，并像老师一样验收当前小步质量。

### 5.3 必须包含的能力

#### 5.3.1 删除"鼓励我一下"按钮

**当前状态**（`TaskCompanionPanel.tsx`）：

```typescript
const SIGNAL_BUTTONS = [
  { signal: "done", label: "我完成了" },
  { signal: "stuck", label: "我卡住了" },
  { signal: "too_hard", label: "太难了" },
  { signal: "encourage", label: "鼓励我一下" },  // ← 删除此项
];
```

**目标状态**：

- `encourage` 不再是独立按钮
- 鼓励融入到：
  - `stuck` → AI 自然回应鼓励 + 材料 + 更小动作
  - `too_hard` → AI 自然回应鼓励 + 降级方案
  - `done` → AI 自然轻量认可
  - 用户输入反馈 → AI 基于内容给行动导向鼓励

**注意**：`encourage` 的 Companion Prompt 支持可以保留——因为用户可能在反馈框中输入"我需要鼓励"，AI 仍需响应。但从 UI 层移除独立按钮。

#### 5.3.2 增加任务反馈输入框

**位置**：`TaskCompanionPanel` 内，AI 消息下方，按钮区域上方

**建议结构**：

```
┌─────────────────────────────────────┐
│  AI 消息区域                         │
│  "不用去搜，我给你 5 个常见问题..."    │
├─────────────────────────────────────┤
│  反馈输入框                          │
│  ┌─────────────────────────────────┐│
│  │ 写下你现在做到哪了 / 卡在哪里 /   ││
│  │ 贴一小段草稿...                  ││
│  └─────────────────────────────────┘│
│  [发送给 AI]                         │
├─────────────────────────────────────┤
│  [我完成了]  [我卡住了]  [太难了]    │
│  [复制当前步骤]  [退出陪伴]          │
└─────────────────────────────────────┘
```

**输入框设计原则**：

| 维度 | 约束 |
|------|------|
| 类型 | `<textarea>` 或 `<input>`，单行或短多行 |
| placeholder | "写下你现在做到哪了 / 卡在哪里 / 贴一小段草稿" |
| 发送按钮文案 | "发送给 AI" |
| 字符限制 | 建议 200-300 字（短文本，不是长文） |
| 发送行为 | 发送后清空输入框，触发 AI 请求 |
| 键盘行为 | Enter 发送（Shift+Enter 换行） |
| 未输入状态 | 输入框为空时，发送按钮可禁用或允许空发送（作为"继续"信号） |

**输入限制（产品约束，不是技术约束）**：

- ✅ 短文本（200-300 字上限）
- ✅ 当前任务上下文内
- ❌ 不做全局聊天
- ❌ 不保存长期对话记忆（任务完成后清空）
- ❌ 不保留跨任务对话历史

#### 5.3.3 AI 基于用户输入继续推进

用户输入 → AI 识别意图 → 给出对应回应：

| 用户输入类型 | 示例 | AI 回应 |
|------------|------|---------|
| 给草稿 | "我叫张三，在XX公司做了3年数据分析..." | AI 评价草稿："这句不错，这句可以加个数据。下一步：补一行你的项目结果。" |
| 说卡住 | "我不知道怎么写项目成果" | AI 识别卡点："成果可以从三个方向写：效率提升、收入增长、问题解决。你的项目属于哪种？" |
| 说只有 3 分钟 | "今天只有 3 分钟" | AI 给 3 分钟版："只看第 1 个问题，只写一句话。3 分钟够。" |
| 不确定是否完成 | "这样算完成吗？" | AI 给验收判断（见 5.3.4） |
| 部分进展 | "我写了开头，但不满意" | AI 给修改建议："开头可以更直接，试试这个句式..." |

#### 5.3.4 AI 验收当前小步质量

这是 V2.6 的核心新增能力——AI 不仅要推下一步，还要能判断"这一步算不算完成"。

**AI 可以说**：

| 验收结论 | 示例 |
|---------|------|
| ✅ 通过 | "这一步基本可以过。你现在可以勾选完成，下一步是..." |
| ⚠️ 还差一点 | "还差一点。你写的项目背景没问题，但缺了'你做了什么'这一句。补一句就行。" |
| 📝 先补 | "先补一句结果就够。不需要写完整报告，就一句话——'结果是 XX 提升了 Y%。'" |
| ❌ 不算完成 | "这一步还不算完成，因为缺少 X。你现在只需要补这一个点，其他的可以先不管。" |

**验收边界**：
- AI 能判断质量：可以、不够、缺什么
- AI 能给具体修改建议：补什么、怎么补
- AI 不能自动勾选任务——用户必须自己点击完成
- AI 不能编造用户经历来"帮"用户通过验收
- 最终完成权始终在用户手中

#### 5.3.5 AI 给鼓励和修改建议

**鼓励规则**（延续 V2.5.3 的主动鼓励机制）：
- 鼓励必须行动导向
- 不空泛鸡汤（❌ "你一定可以的"）
- 不心理诊断（❌ "你看起来很焦虑"）
- 可以给小段修改建议
- 不替用户写完整最终稿

**修改建议示例**：
> "开头没问题。第二句可以更具体——不说'做了很多项目'，说'主导了XX项目，结果是 YY'。试着重写这一句。"

| 8 | 不做多轮对话历史 UI（聊天气泡列表） | 只保留当前步骤 + AI 最后回应 |

## 六、V2.7：任务难度与数量动态调整

### 6.1 名称

**V2.7：任务难度与数量动态调整**

### 6.2 一句话目标

当用户多次反馈"太难 / 卡住 / 没时间"时，AI 不只是安慰，而是能建议降低今天任务难度和数量——并让用户一键接受调整。

### 6.3 为什么 V2.7 不能合并到 V2.6？

| 理由 | 说明 |
|------|------|
| **V2.6 是 V2.7 的前置** | 没有反馈输入框，AI 无法准确判断用户是否"多次卡住"——只能靠按钮计数，但按钮信息量不够 |
| **V2.7 的改动面更大** | 涉及任务状态扩展、任务组调整逻辑、可能的数据字段设计 |
| **V2.7 需要 V2.6 的用户反馈数据** | 用户反馈框里的"太难了""太大了""今天没时间"才是真正的调整触发信号 |
| **先验证 V2.6 的用户反馈体验** | 如果用户不喜欢在 App 里打字，V2.7 的调整触发策略需要重新设计 |

**结论**：V2.6 验证"用户→AI 反馈通道"，V2.7 利用这个通道做"任务自适应调整"。

### 6.4 必须包含的能力

#### 6.4.1 识别调整触发条件

| 触发条件 | 检测方式 |
|---------|---------|
| 连续多次 too_hard | 同一任务内 2+ 次 too_hard |
| 连续多次 stuck | 同一任务内 2+ 次 stuck |
| 用户输入"今天没时间" | 反馈框文本识别 |
| 用户输入"这个太大了" | 反馈框文本识别 |
| 当前任务反复无法通过验收 | AI 多次给"还差一点"验收结论 |
| 同一任务组内多任务卡住 | 跨任务的 stuck/too_hard 模式 |

#### 6.4.2 AI 给出调整建议

| 调整类型 | 示例 |
|---------|------|
| 降低当前任务难度 | "这个任务确实大。我们把它降级为最小完成版：只需要..." |
| 把当前任务改成最小完成版 | "完整版先不管。今天只需要做：..." |
| 建议今天只保留 1-2 个核心任务 | "今天已经卡了好几次了。建议只保留前 2 个任务，其余明天再做。" |
| 建议其余任务明天继续 | "剩下的 3 个任务可以标记为明天继续，你今天已经尽力了。" |
| 建议生成降级版任务 | "这个任务我帮你想了一个降级版，只需要 5 分钟。" |

**AI 不能**：
- 自动删除任务 ❌
- 自动勾选任务 ❌
- 绕过 locked 顺序执行 ❌
- 替用户决定放弃目标 ❌
- 说"这个目标太难了，换个目标吧" ❌

#### 6.4.3 增加"接受调整"按钮

```
┌─────────────────────────────────────┐
│  AI 调整建议                         │
│  "今天卡了好几次。建议今天只保留      │
│   前 2 个任务，后 3 个移到明天。"     │
│                                      │
│  [接受调整]  [不用，继续]             │
└─────────────────────────────────────┘
```

**关键约束**：
- AI 不自动改任务——用户必须点击"接受调整"
- Human-in-the-Loop 必须保留
- 用户点击"不用，继续"后，AI 继续推进当前任务
- 调整是建议，不是命令

#### 6.4.4 任务标记能力

接受调整后，任务可以被标记为：

| 标记 | 含义 | UI 表现 |
|------|------|---------|
| **明日继续** | 今天不做，明天自动带入 | 弱化显示，移到列表底部 |
| **降级版** | 替换为更简单版本 | 任务标题更新，但保留原版引用 |
| **暂缓** | 本周内再做 | 从今日列表移除，不删除 |
| **保留但不要求今天完成** | 保留可见，但不催促 | 灰色显示，无完成压力 |

#### 6.4.5 明确边界

| AI 不能 | 原因 |
|---------|------|
| 自动删除任务 | 用户可能还想做 |
| 自动勾选任务 | Human-in-the-Loop |
| 绕过 locked 顺序执行 | V2.5.1 规则 |
| 替用户决定放弃目标 | 尊重用户目标 |
| 在用户拒绝调整后反复建议 | 尊重用户选择 |

### 6.5 推荐技术方向

> ⚠️ 以下为方向判断，不是具体执行方案。

| 层 | 可能需要 | 说明 |
|----|:--:|------|
| 任务状态字段 | 🔄 可能新增 | `adjustment_status`: "明天继续" / "降级版" / "暂缓" |
| 任务组调整逻辑 | 🔄 可能新增 | `adjustTaskGroup()` 函数 |
| UI 层 | 🔄 可能新增 | "接受调整"按钮 + 调整建议卡片 |
| API | 🔄 可能新增或扩展 | 保存调整后的任务组 |
| 数据库字段 | 🔄 可能新增 | tasks 表可能新增 `adjustment` 字段 |
| Prompt | 🔄 改 | companion 新增调整触发和输出规则 |
| `task-execution.ts` | 🔄 可能改 | 任务状态判断可能扩展 |

**V2.7 必须在 V2.6 稳定后再做。**

## 八、版本依赖关系

```
V2.5.3 ✅ ──→ V2.6 🔜 ──→ V2.7 📋 ──→ V3.0A 🏗️
  │              │              │              │
  │              │              │              │
  ▼              ▼              ▼              ▼
AI 能主动    用户能反馈    AI 能调整    App 能承载
给材料       草稿/卡点     任务难度     所有能力
```

### 依赖关系详解

#### 依赖 1：V2.5.3 → V2.6

**V2.5.3 是 V2.6 的前置条件。**

理由：
- V2.5.3 让 AI 知道可以主动给材料（不再只是给指令）
- 只有 AI 先能给材料，用户才有"材料使用结果"需要反馈
- 如果 AI 还是说"去知乎搜索"，用户不需要反馈框——因为在 App 内没有"使用材料"的行为

**没有 V2.5.3 的 V2.6 会怎样**：
- 用户输入"我写了一句话，你看看行不行"
- AI 回答"请打开浏览器搜索"——反馈框变成摆设

#### 依赖 2：V2.6 → V2.7

**V2.6 是 V2.7 的前置条件。**

理由：
- V2.7 需要判断"用户是否多次卡住、太难、没时间"
- 固定按钮只能给 4 种信号，信息量不够做准确判断
- 用户反馈框里的"太大了""今天只有 3 分钟""我做不动了"才是真正的调整触发信号
- 没有反馈框，V2.7 的调整触发只能靠按钮计数——太粗糙

**没有 V2.6 的 V2.7 会怎样**：
- 用户点一次"太难了"，AI 就开始建议调整——过度反应
- 或者用户点了 5 次"太难了"，AI 无动于衷——按钮无法表达"为什么难"

#### 依赖 3：V2.7 → V3.0A

**V2.7 是 V3.0A 的体验增强前置。**

理由：
- V3.0A 可以先做 UI（App Shell + 底部导航），不依赖 V2.7
- 但最理想的 V3.0A 是把 V2.6/V2.7 的新陪伴卡片一起整合进去
- 如果 V3.0A 在 V2.6/V2.7 之前做，App Shell 里装的还是旧版固定按钮陪伴面板——刚做完就要再改一次

**推荐的策略**：
- V3.0A 等 V2.6/V2.7 完成后再做（理想路径）
- 或者在 V2.6 完成后先做 V3.0A 的纯 UI 重组部分（不含陪伴卡片新形态），V2.7 完成后再把调整卡片整合进去（折中路径）

#### 依赖 4：V3.0A 不应该在陪伴卡片新结构未明确前贸然重构

理由：
- 否则可能把"4 个固定按钮 + 鼓励我一下"的旧体验搬进新 App Shell
- App Shell 的目标是承载最好的执行体验——如果陪伴面板还是旧版，App Shell 的价值打折扣

## 十、产品边界

### 10.1 AI Todo 是什么（不变）

- 手机端优先的 AI 行动教练
- 帮助用户把模糊目标转化为今天能做的小任务
- 降低开始行动的心理压力
- 提高真实完成率

### 10.2 AI Todo 不是（不变）

| 不是 | 说明 |
|------|------|
| 聊天软件 | 不追求对话体验，追求执行结果 |
| 项目管理工具 | 不追求多项目、多阶段管理 |
| 自动化 Agent | 不自动替用户做事 |
| 心理治疗工具 | 不做心理诊断和情感陪伴 |
| 搜索引擎 | 不追求覆盖所有知识领域 |

### 10.3 V2.6-V3.0A 的新边界

| 维度 | V2.5.3 边界 | V2.6-V3.0A 新边界 |
|------|-----------|----------------|
| 用户→AI 通道 | 4 个固定按钮 | 固定按钮 + 受控短反馈文本 |
| AI 对用户输出的理解 | 只知道按钮类型 | 知道草稿内容、卡点原因、时间约束 |
| AI 验收能力 | 无（只能推下一步） | 可以判断"够不够""缺什么" |
| 任务调整 | AI 无调整能力 | AI 可以建议调整，用户接受后执行 |
| 界面结构 | 单页堆叠 | App Shell + 底部导航 |

### 10.4 如何避免产品变成普通聊天软件？

| 机制 | 说明 |
|------|------|
| **任务绑定** | 反馈框只在 TaskCompanionPanel 内，只服务当前任务 |
| **上下文限制** | 只传最近 5 步历史，不保留长期对话 |
| **不存对话** | 不把用户输入存入数据库，任务完成后清空 |
| **安全红线** | 所有 V2.5.3 的安全红线仍然生效 |
| **无全局入口** | 没有底部导航的"聊天"Tab，没有全局聊天按钮 |
| **任务完成即结束** | 用户勾选任务完成后，面板关闭，对话结束 |
| **输入框定位** | 是"任务反馈"不是"聊天输入"——placeholder 明确表达用途 |
| **UI 不是聊天界面** | 不做聊天气泡列表，不做对话历史滚动，只保留当前 AI 输出 |

### 10.5 如何避免 AI 替用户完成任务？

所有 V2.5.3 的安全红线 + 帮助边界规则在 V2.6 和 V2.7 中继续生效。具体见第十四节。

### 10.6 如何继续保持任务完成权在用户手中？

| 机制 | 说明 |
|------|------|
| **用户自己勾选** | 不论 AI 说"可以过了"还是"还差一点"，勾选永远是用户手动操作 |
| **AI 不能自动勾选** | 安全红线明确禁止 |
| **验收是建议不是命令** | AI 说"可以过了"，用户可以不同意 |
| **调整需用户确认** | V2.7 的调整必须用户点击"接受调整"才生效 |
| **拒绝权** | 用户可以点"不用，继续"拒绝 AI 的调整建议 |

## 十一、技术风险

### 11.1 V2.6 风险

| # | 风险 | 严重度 | 缓解 |
|---|------|:--:|------|
| 1 | 用户反馈框变成"要求 AI 完成整件事" | P0 | Prompt 安全红线保留；输入框 placeholder 引导正确用法 |

## 十五、路线文档必须回答的 10 个问题

### Q1：为什么 V2.5.3 之后不能直接做 V3.0A？

V2.5.3 修好了 AI "能说什么"，但修好之后立刻暴露了新问题：用户没有通道把反馈给 AI。如果直接跳到 V3.0A（App Shell 重组 UI），结果是把"单向通道 + 固定按钮"的旧体验搬进新框架——UI 好看了，但核心体验问题没解决。应该先用 V2.6 修好"用户→AI 通道"，用 V2.7 修好"AI 自适应调整"，再把完整体验装入 V3.0A 的 App Shell。

### Q2：为什么固定按钮不够？

固定按钮只能表达 4 种状态（完成/卡住/太难/要鼓励），无法表达用户真实的 7 种反馈：草稿内容、部分进展、验收请求、时间约束、粒度反馈、修改结果、实际卡点原因。当 AI 开始给材料（而不是只给指令）后，用户的行为变复杂了——4 个按钮覆盖不了。

### Q3：为什么需要任务内受控反馈框？

因为 V2.5.3 让 AI 从"搜索引擎代理"变成了"材料提供者"。用户现在会在 App 内使用 AI 给的材料——写草稿、填框架、套模板——然后需要把使用结果反馈给 AI。没有反馈框，用户就只能"用了材料没法说"，AI 也只能"给了材料不知道效果"。

### Q4：为什么鼓励按钮应该删除？

V2.5.3 已经把主动鼓励内置到了 stuck / too_hard / done / start 中。独立的"鼓励我一下"按钮把鼓励变成了用户主动索取的行为——体验不自然。有了反馈输入框后，用户的真实表达（"写不出来""好难"）本身就是请求支持——AI 自然回应鼓励即可。

### Q5：为什么 AI 验收很重要？

当前 AI 只会推"下一步"，但不会判断"上一步够不够"——这导致用户不确定自己是不是真的"完成了"。有了验收能力，AI 可以像老师一样说"可以过了"或"还差一句"或"缺了 X"——这让"完成"有了质量标准，而不是用户自己猜。

### Q6：为什么动态调整任务难度与数量要放到 V2.7，而不是 V2.6？

V2.6 建立用户→AI 的反馈通道，V2.7 利用这个通道做任务调整。没有反馈通道，调整触发只能靠按钮计数——信息量不够，容易误判。而且 V2.7 涉及任务状态扩展、可能的数据字段变更——改动面比 V2.6 大，需要等 V2.6 稳定后再做。

### Q7：为什么 V3.0A 要等 V2.6/V2.7 明确后再落地？

V3.0A 是 UI 重组，它依赖于 V2.6/V2.7 已经定型的陪伴卡片形态和任务列表形态。在不知道"新陪伴卡片里有什么""任务调整建议长什么样"之前设计 App Shell，等于把旧形态搬进新框架。先定义内容，再设计容器。

### Q8：如何避免产品变成普通聊天软件？

8 道防线：任务绑定（只在 TaskCompanionPanel 内）、上下文限制（只传最近 5 步）、不存对话（任务完成后清空）、安全红线（全部保留）、无全局聊天入口（没有聊天 Tab）、任务完成即结束（面板关闭）、输入框定位是"反馈"不是"聊天"（placeholder 明确）、UI 不是聊天界面（不做气泡列表）。

### Q9：如何避免 AI 替用户完成任务？

所有 V2.5.3 的安全红线和帮助边界规则在 V2.6/V2.7 中继续生效。新增的验收能力只给"质量判断"不给"自动完成"。调整建议需要用户点击"接受"才生效。代码 Review 逐条检查 Prompt 中"不能替用户完成什么"分节的规则。

### Q10：如何继续保持任务完成权在用户手中？

勾选永远是用户手动操作。AI 验收是建议不是命令。AI 调整需用户确认。用户始终可以拒绝。Human-in-the-Loop 是最高优先级——这条在代码 Review 中每次必查。
