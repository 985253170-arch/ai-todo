### 4.1 V2.6 只做

| # | 内容 | 说明 |
|---|------|------|
| 1 | 任务内受控反馈框 | `<textarea>` 或 `<input>`，在 `TaskCompanionPanel` 内，AI 消息下方 |

| 7 | AI 可建议"这一步基本可以过 / 还差一点 / 你可以自己勾选完成" | 验收是建议，不是命令 |

## 7. 版本依赖关系

### 依赖 1：V2.5.3 → V2.6（硬依赖）

**V2.5.3 是 V2.6 的前置条件。**

| 理由 | 说明 |
|------|------|
| AI 必须先能主动给材料 | V2.5.3 让 AI 不再说"去搜索"，而是直接给材料 |
| 只有 AI 给材料，用户才有"材料使用结果"需要反馈 | 如果 AI 还是说"去知乎搜索"，用户在 App 内没有反馈需求 |
| 没有 V2.5.3，V2.6 的反馈框是摆设 | 用户输入"我写了一句话，你看看"，AI 回答"请去浏览器搜索"——反馈框白做 |

**违规后果**：如果在 V2.5.3 未完成前做 V2.6，AI 无法理解用户反馈的材料使用场景，反馈框无意义。

### 依赖 2：V2.6 → V2.7（硬依赖）

**V2.6 是 V2.7 的前置条件。**

| 理由 | 说明 |
|------|------|
| 没有反馈通道，无法判断用户是否多次卡住 | 4 个固定按钮信息量不够，按钮计数太粗糙 |
| 用户反馈框里的文本才是真正的调整触发信号 | "太大了""今天只有 3 分钟""我做不动了"——按钮无法表达 |
| V2.7 的调整触发依赖用户反馈的语义信息 | 不是靠按钮次数，是靠"为什么难"的内容判断 |
| V2.6 需要先验证用户反馈体验 | 如果用户不喜欢在 App 里打字，V2.7 的调整触发策略需要重新设计 |

**违规后果**：如果在 V2.6 未完成前做 V2.7，调整触发只能靠按钮计数——要么过度反应（点一次就建议调整），要么迟钝（点了 5 次无动于衷）。

### 依赖 3：V2.6 + V2.7 → V3.0A（强建议依赖）

**V2.6 / V2.7 是 V3.0A 的体验前置。**

| 理由 | 说明 |
|------|------|
| V2.6 改变了陪伴面板形态 | 新增输入框、删除 encourage 按钮、新增 AI 验收 |
| V2.7 改变了任务列表形态 | 调整建议卡片、"接受调整"按钮、任务标记 |
| 不能在不知道新形态时设计 App Shell | 否则把旧形态搬进新框架，刚做完又要改 |
| V3.0A 的目标是承载最好的执行体验 | 如果陪伴面板还是旧版 4 按钮，App Shell 价值打折扣 |

**违规后果**：V3.0A 在 V2.6/V2.7 之前做 → App Shell 里装旧版陪伴面板 → V2.6/V2.7 做完后必须再改一次 App Shell → 浪费一次 UI 重组。

**折中路径**（如果时间压力大）：V2.6 完成后可以先做 V3.0A 的纯 UI 重组（不含陪伴卡片新形态），V2.7 完成后再把调整卡片整合进去。但**最理想路径**是 V2.6 → V2.7 → V3.0A 严格顺序。

### 依赖 4：V3.0A 不应在陪伴卡片新结构未明确前贸然重构（硬约束）

**V3.0A 在启动前必须确认 V2.6 架构已完成设计（执行方案阶段即可，不需要代码上线），否则会出现"新外壳 + 旧体验"。**

### 依赖关系图

```
V2.5.3 ✅ ──→ V2.6 🔜 ──→ V2.7 📋 ──→ V3.0A 🏗️
  │              │              │              │
  │  硬依赖       │  硬依赖       │  强建议依赖    │
  │              │              │              │
  ▼              ▼              ▼              ▼
AI 能主动    用户能反馈    AI 能调整    App 能承载
给材料       草稿/卡点     任务难度     所有能力
```

## 9. 产品边界锁定

### 9.1 AI Todo 是什么（不可变）

- 手机端优先的 **AI 行动教练**，不是 Todo List
- 帮助用户把模糊目标转化为**今天能做的小任务**
- 降低开始行动的心理压力
- 提高真实完成率

### 9.2 AI Todo 不是什么（不可变）

| 不是 | 说明 |
|------|------|
| 聊天软件 | 不追求对话体验，追求执行结果 |
| 项目管理工具 | 不追求多项目、多阶段管理 |
| 自动化 Agent | 不自动替用户做事 |
| 心理治疗工具 | 不做心理诊断和情感陪伴 |
| 搜索引擎 | 不追求覆盖所有知识领域 |

### 9.3 产品边界硬约束（V2.6-V3.0A 期间不可破坏）

| # | 边界 | 说明 |
|---|------|------|
| 1 | AI Todo 是行动教练，不是聊天软件 | 最高优先级产品边界 |
| 2 | 允许当前任务内受控短反馈输入框 | V2.6 新增，仅此而已 |
| 3 | 不允许全局自由聊天框 | 没有聊天 Tab、没有全局聊天入口 |
| 4 | 输入框只服务当前任务、当前步骤、当前草稿、当前卡点 | 不服务跨界上下文 |
| 5 | 不新增聊天 Tab | 底部导航永远不出现"聊天" |
| 6 | 不做长期对话记忆 | 对话不存数据库，任务完成后清空 |
| 7 | 任务完成后，反馈上下文结束 | 面板关闭 = 对话结束 |
| 8 | AI 可以验收，但不能完成 | 验收是建议，勾选是用户的动作 |
| 9 | AI 可以建议"你可以自己勾选完成"，但不能自动勾选 | Human-in-the-Loop 最高优先级 |
| 10 | 任务完成权始终在用户手里 | 所有版本、所有功能都必须遵守 |

### 9.4 如何避免产品变成普通聊天软件（8 道防线）

| # | 防线 | 机制 |
|---|------|------|
| 1 | 任务绑定 | 反馈框只在 `TaskCompanionPanel` 内，只服务当前任务 |
| 2 | 上下文限制 | 只传最近 5 步历史（`stepHistory`），不保留长期对话 |
| 3 | 不存对话 | userFeedback 不写入数据库，任务完成后清空 |
| 4 | 安全红线 | 所有 V2.5.3 安全红线在 V2.6/V2.7/V3.0A 中继续生效 |
| 5 | 无全局入口 | 不做底部导航的"聊天"Tab，不做全局聊天按钮 |
| 6 | 任务完成即结束 | 用户勾选任务完成后，陪伴面板关闭，对话结束 |
| 7 | 输入框定位是"反馈"不是"聊天" | placeholder 明确表达："写下你现在做到哪了 / 卡在哪里 / 贴一小段草稿" |
| 8 | UI 不是聊天界面 | 不做聊天气泡列表、不做对话历史滚动、只保留当前 AI 输出 |

### 9.5 Human-in-the-Loop 保障（不可破坏）

| # | 保障 |
|---|------|
| 1 | 任务完成：用户手动勾选 checkbox，AI 不能自动勾选 |
| 2 | AI 验收：AI 只能建议"可以过了"，不能自动标记完成 |
| 3 | AI 调整（V2.7）：用户必须点击"接受调整"才生效 |
| 4 | 用户拒绝权：用户可以点"不用，继续"拒绝调整建议 |
| 5 | 用户始终是最终决策者 |

## 10. 对 V3.0 旧判断的修正

### 10.1 当前旧判断

`docs/Architecture-V3.0-Web-App-Separation.md` 中存在以下旧判断，**必须修正**：

**位置 1：第 4 节"不做事项"第 11 条**

> ❌ 旧判断："不做完整自由聊天系统"
>
> ✅ 修正为："不做全局自由聊天系统；允许当前任务内受控短反馈输入框"

**位置 2：第 13 节"AI 陪伴面板结构"**

> ❌ 旧判断："不出现通用输入框：使用固定反馈按钮（start / done / stuck / too_hard / encourage）"
>
> ✅ 修正为："不出现全局自由聊天框；允许当前任务内受控短反馈输入框。使用反馈按钮（done / stuck / too_hard）+ 任务内受控反馈输入框"

**位置 3：第 17 节"风险"第 5 条**

> ❌ 旧判断："不新增自由输入框，保持任务上下文 + 固定反馈按钮"
>
> ✅ 修正为："不新增全局自由输入框；任务内反馈框需确保不变成聊天"

### 10.2 修正说明

这不是普通聊天框：

1. **这是当前任务反馈通道**——用户告诉 AI "写到哪了、卡在哪、差多少"
2. **它服务执行、验收、鼓励、调整**——四个核心用途，不是闲聊
3. **V3.0A 执行方案必须按这个新判断处理**——陪伴卡片设计必须基于 V2.6 新形态

### 10.3 修正时机

- V3.0 架构文档（`Architecture-V3.0-Web-App-Separation.md`）的修正**不在 V2.6/V2.7 范围内**
- 修正应在 **V3.0A 执行方案阶段**（即 V2.6/V2.7 完成后、V3.0A 实施前）统一修订
- 届时基于本锁定文档的新判断进行修订
