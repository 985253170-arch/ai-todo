# AI Todo V2.0 产品规划 PRD

## 1. 产品长期定位

AI Todo 的长期定位不是传统 Todo List，而是一个手机端优先的 AI 行动教练。

它的核心价值是帮助用户把“模糊目标”转化为“今天能做的小任务”，并通过历史完成情况不断调整任务难度、数量和节奏，降低开始行动的心理压力，提高真实完成率。

产品主线：

目标 → AI 拆解 → 执行 → 记录 → 复盘 → 智能调整

V2.0 的重点不是做复杂项目管理，而是让用户每天更容易开始、更容易完成、更容易持续。

## 2. V1.0 已完成能力总结

当前 AI Todo 已具备以下基础能力：

- 输入一个目标
- AI 生成 3-8 条当天可执行任务
- 展示任务列表
- 勾选任务完成状态
- 本地 localStorage 保存与恢复
- Supabase 云端保存与恢复
- 登录 / 登出
- 匿名 device_id 任务归属
- 登录 user_id 任务归属
- 匿名 device_id → user_id 迁移
- PWA 基础支持
- 手机端基础适配
- AI 行动教练风格 UI

V1.0 已经完成了“今天目标 → AI 拆解 → 执行保存”的闭环。

## 3. V2.0 核心目标

V2.0 的核心目标是从“单日任务工具”升级为“持续行动教练”。

主要目标：

1. 记录用户每天生成和完成的任务。
2. 展示历史行动记录。
3. 统计任务完成情况。
4. 让 AI 基于历史表现给出轻量复盘。
5. 为下一次任务生成提供个性化调整依据。
6. 保持手机端使用简单、低压力。

V2.0 成功标准：

- 用户能看到自己过去做过什么。
- 用户能知道自己最近完成得怎么样。
- AI 能根据用户完成情况调整建议。
- 用户不会感觉产品变复杂。

## 4. V2.0 不做范围

V2.0 明确不做：

- 团队协作
- 项目管理
- 多任务组复杂管理
- 甘特图
- 日历提醒
- 社交分享
- 排行榜
- 会员系统
- 支付
- 权限分级
- 复杂数据报表
- 多人空间
- 目标 OKR 系统
- 长周期项目管理工具

V2.0 仍然聚焦个人、当天、轻量行动。

## 5. 核心用户场景

场景 1：早上规划今天

用户打开手机，输入“我要准备面试”，AI 拆成今天能完成的几件小事。用户不需要自己规划。

场景 2：晚上查看完成情况

用户看到今天完成了 4 / 6 个任务，知道自己推进了目标，也知道哪些任务没完成。

场景 3：第二天继续

用户再次输入类似目标，AI 根据昨天完成情况减少任务数量，或把任务拆得更轻。

场景 4：低能量状态

用户最近连续完成率低，AI 给出更轻量建议，例如“今天先做 2 件最小任务”。

场景 5：阶段性复盘

用户查看最近 7 天，知道自己在哪些目标上持续行动，哪些目标经常中断。

## 6. 用户主流程

主流程：

1. 用户打开 AI Todo。
2. 输入今天想推进的目标。
3. AI 生成今日任务。
4. 用户执行并勾选任务。
5. 系统保存当天任务记录。
6. 用户第二天打开，可以看到历史记录和完成统计。
7. AI 根据历史完成情况调整任务生成策略。
8. 用户形成持续行动循环。

登录用户流程：

1. 登录账号。
2. 当前 device 匿名任务迁移到 user_id。
3. 后续任务记录归属 user_id。
4. 多设备读取同一账号历史和统计。

未登录用户流程：

1. 使用 device_id 保存任务。
2. 可查看当前设备历史。
3. 登录后迁移匿名任务。

## 7. 功能模块设计

V2.0 建议包含 5 个模块：

1. 今日行动计划
   - 保持 V1.0 当前主界面
   - 输入目标
   - AI 生成任务
   - 勾选完成
   - 清空 / 重新生成 / 开始新一天

2. 历史记录
   - 展示过去每天的任务组
   - 按日期倒序
   - 显示目标、完成数、总任务数
   - 点击可查看当天任务详情

3. 完成统计
   - 最近 7 天完成率
   - 最近 30 天完成率
   - 总完成任务数
   - 连续行动天数
   - 最常出现的目标关键词，可后置

4. AI 复盘
   - 今日完成后给一句反馈
   - 每周给轻量总结
   - 根据完成率调整任务数量建议

5. 生成策略调整
   - 高完成率：任务可以略多或略深入
   - 低完成率：任务更少、更具体、更容易开始
   - 经常未完成：优先生成 2-3 个最小任务

## 8. 历史记录设计

历史记录目标：

让用户看到自己每天推进了什么，而不是做复杂归档。

建议字段：

- 日期
- goal
- tasks
- completedCount
- totalCount
- completionRate
- createdAt
- updatedAt
- user_id / device_id

历史记录展示：

- 默认展示最近 7 天
- 支持展开最近 30 天
- 每条记录是一个简洁卡片：
  - 日期
  - 目标
  - 已完成 4 / 6
  - 完成率 67%

历史记录不做：

- 手动编辑历史
- 多分类管理
- 标签系统
- 搜索筛选，V2.0 可暂不做

## 9. 任务完成统计设计

V2.0 统计应保持轻量。

核心指标：

1. 今日完成率
   - completedCount / totalCount

2. 最近 7 天完成率
   - 最近 7 天所有任务完成数 / 总任务数

3. 连续行动天数
   - 有生成任务且至少完成 1 条任务的连续天数

4. 总完成任务数
   - 用户累计完成的任务数量

5. 最近表现状态
   - 稳定行动
   - 有点吃力
   - 刚刚开始

避免复杂指标：

- 不做燃尽图
- 不做项目进度预测
- 不做复杂趋势图
- 不做排行榜

## 10. AI 复盘与反馈设计

AI 复盘分三层：

1. 今日反馈

触发条件：

- 用户完成全部任务
- 用户完成部分任务
- 用户一天结束后再次打开

示例：

- “今天你完成了 4 个任务，已经把目标往前推进了一步。”
- “今天任务完成率偏低，明天可以从更小的一步开始。”

2. 下次生成调整

AI Prompt 可加入历史上下文：

- 最近 7 天平均完成率
- 最近一次任务数量
- 最近未完成任务数量
- 当前目标

调整策略：

- 完成率高：生成 5-7 条任务
- 完成率中：生成 3-5 条任务
- 完成率低：生成 2-3 条更轻任务

3. 周复盘

每 7 天生成一次轻量总结：

- 本周推进过哪些目标
- 完成情况如何
- 下周建议怎么调整

注意：

AI 复盘必须短，不要说教，不要制造压力。

## 11. 移动端体验方向

V2.0 必须手机端优先。

设计原则：

- 第一屏仍然是今日行动计划
- 历史和统计不能抢主流程
- 底部或卡片内提供轻量入口
- 文案温和，不制造焦虑
- 任务卡片触控区域足够大
- 统计用简单数字和短句，不做复杂图表

建议结构：

首页：

- Header
- 今日行动计划
- 今日完成情况
- 最近 7 天简要统计
- 历史记录入口

历史页或历史区：

- 日期卡片列表
- 点击展开任务详情

统计区：

- 最近 7 天完成率
- 连续行动天数
- 总完成任务数
- AI 简短反馈

## 13. API 能力规划

建议新增 API：

1. GET `/api/task-groups/history`

用途：获取历史任务组。

参数：

- limit：返回数量
- cursor：分页游标，可后置

返回字段：

- success：请求是否成功
- data：历史任务组数组
- data.id：任务组 ID
- data.goal：用户目标
- data.tasks：任务数组
- data.completedCount：已完成任务数
- data.totalCount：任务总数
- data.createdAt：创建时间
- data.updatedAt：更新时间

2. GET `/api/task-groups/stats`

用途：获取最近 7 / 30 天统计。

返回字段：

- success：请求是否成功
- data.totalCompleted：总完成任务数
- data.sevenDayCompletionRate：最近 7 天完成率
- data.activeDayStreak：连续行动天数

3. POST `/api/task-groups/review`

用途：AI 复盘。

请求字段：

- taskGroupId：任务组 ID

返回字段：

- success：请求是否成功
- data.feedback：AI 复盘反馈

4. generate-tasks API 增强

在后端读取历史统计，不让前端传可信统计。

增强后：

- 当前 goal
- 最近完成率
- 最近任务数量
- 最近未完成情况

用于调整 AI 生成策略。

## 14. 与 Phase 11 的兼容关系

Phase 11 已完成：

- 登录 / 登出
- user_id 归属
- device_id 匿名归属
- device_id → user_id 迁移

V2.0 必须延续这套规则：

- 已登录：所有历史、统计、复盘都按 user_id 查询
- 未登录：按 device_id 查询
- 前端不能传 userId
- userId 只能来自服务端 session
- service_role 只在服务端 API 使用
- 登录后迁移匿名历史，不能重复创建数据

兼容重点：

- Phase 12 历史记录必须 session-aware
- Phase 13 统计必须 session-aware
- Phase 14 AI 复盘必须 session-aware
- 登出后不能显示 user_id 历史和统计

Phase 12 验收标准：

- 未登录只看 device_id 历史

Phase 13 验收标准：

- 显示最近 7 天完成率
- 显示总完成任务数
- 显示连续行动天数
- 统计按 user_id / device_id 隔离

- 手机端展示清晰

Phase 14 验收标准：

- AI 失败时页面不崩溃

## 17. 风险点

1. 产品复杂度风险

历史、统计、复盘容易把产品变成复杂管理工具。

控制方式：

- 首页仍以今日任务为主
- 历史和统计做轻量入口
- 不做复杂筛选和图表

2. AI 反馈压力风险

如果 AI 复盘像批评用户，会降低使用意愿。

控制方式：

- 文案温和
- 不评价人格
- 只反馈行动
- 强调下一步

3. 数据归属风险

登录 / 登出 / 匿名迁移后，历史和统计容易串数据。

控制方式：

- API 全部 session-aware
- userId 只来自服务端
- 本地缓存带 scope
- 登出后清空 user 数据

4. 统计准确性风险

跨天、时区、重复生成任务会影响统计。

控制方式：

- 明确以用户本地日期为统计口径
- Phase 13 前定义 date 归属规则
- 避免早期复杂跨时区逻辑

5. AI 成本风险

每次复盘或智能调整都调用 AI 会增加成本。

控制方式：

- 复盘按需触发
- 统计本地或数据库计算
- Prompt 保持短上下文
- 周复盘后置

## 18. 最终推荐开发路线

推荐路线：

1. Phase 12：历史记录
2. Phase 13：完成统计
3. Phase 14：AI 复盘
4. Phase 15：智能任务调整

原因：

- 历史记录是统计和复盘的基础。
- 统计先于 AI 调整，否则 AI 没有稳定依据。
- 复盘先于智能调整，可以先验证用户是否喜欢 AI 教练反馈。
- 最后再让 AI 生成策略自动变化，风险最低。

V2.0 最小闭环：

用户今天生成任务并完成 → 明天能看到昨天记录 → 看到最近完成情况 → AI 给一句复盘 → 下次任务更适合自己。

这条路线最符合 AI Todo 的长期定位：不是管理更多任务，而是帮助用户持续完成真正重要的小行动。
