# V3.0：Web 与手机应用界面分离架构方案

> 状态：架构设计阶段
> 定位：将 AI Todo 从"一个响应式网页"升级为"Web 产品入口 + 手机应用工作台"双界面体系
> 依赖：V2.5.3 AI 主动辅助与帮助边界修正 ✅
> 下一文档：Execution-Plan-V3.0-Web-App-Separation.md

---

## 1. 背景与问题

### 1.1 当前状态

AI Todo 当前只有一个核心功能页面 `/app`，所有功能堆在单页内：GoalInput、TaskList、StatsBar、HistoryPanel、TaskReviewPanel、AI Assist Panel、AI Companion Panel 全部在 `/app` 中渲染。

### 1.2 存在的问题

1. **功能堆叠**：目标输入、任务列表、统计、历史、复盘全堆在一个长页面，视觉和使用体验差
2. **网页端和手机端共用同一套页面思路**：响应式适配只是把同一个页面缩小/重排，没有为手机端设计独立的信息架构
3. **PWA 不等于 App**：PWA 虽然已能"添加到桌面"，但它仍然只是网页壳，不等于完成 App 级信息架构。App 需要底部导航、单屏单任务、触控优先的交互设计
4. **用户已有手机应用界面结构设计图**：后续实现必须尊重 App 独立结构，不能继续走"一个响应式页面"的老路
5. **信息架构不利于高频使用**：用户每天打开 App 需要先滚动过 GoalInput 和一堆已完成任务，才能看到当前要做的任务

## 2. 核心判断

```
Web 和 App 必须分开设计。
不是简单响应式适配。
不是把网页缩小成手机。
不是把 App 做成另一个 Landing Page。
```

Web 端和 App 端有不同的信息架构、不同的导航方式、不同的用户目标。它们可以共享同一套 API 和数据库，但 UI 层必须独立设计。

## 3. V3.0 目标

1. 建立 Web / App 双界面边界
2. 保留 Web 作为产品介绍、登录、桌面访问入口
3. 将核心使用体验迁移为手机 App Shell
4. 降低主页面信息堆叠
5. 让"今日行动 → 当前任务 → AI 陪伴 → 完成反馈"成为手机端主路径
6. 不过度重构——先重组 UI 层，不改 Hook 和 API

## 4. V3.0 不做事项

| # | 不做 | 原因 |
|---|------|------|
| 1 | 不重写 Auth | OTP + Password 体系稳定，不动 |
| 2 | 不重写数据库 | 当前 schema 满足需求 |
| 3 | 不重写 AI API | generate-tasks / review / assist / companion 全套稳定 |
| 4 | 不重写任务生成逻辑 | task-generation prompt 和 task-parser 不动 |
| 5 | 不新增支付 | 产品早期不商业化 |
| 6 | 不做团队协作 | 个人工具定位不变 |
| 7 | 不做复杂项目管理 | 不是 Jira/Asana |
| 8 | 不直接上架 App Store | PWA 先行 |
| 9 | 不做 React Native / Flutter | V3.0 仍是 Web 技术栈 |
| 10 | 不做原生推送 | 留在后续评估 |
| 11 | 不做完整自由聊天系统 | AI 陪伴保持当前的任务上下文 + 固定反馈按钮 |

## 5. Web 端定位

Web 端负责"产品发现和账号管理"，不承担主要任务执行体验。

### Web 端职责

- Landing Page（产品介绍、价值主张）
- 登录 / 注册
- 忘记密码 / 重置密码
- 桌面端轻量使用（保留 `/app` 的桌面访问能力）
- 未来：设置 / 帮助 / 隐私政策页面

### Web 端明确不做的

- 不做 Web 端的 AI 陪伴深度体验
- 不做 Web 端的任务执行主流程
- 不在 Web 首页堆砌功能

## 6. 手机 App 端定位

手机 App 端是 AI Todo 的主产品体验，负责"每日行动执行"。

### App 端职责

- 今日行动（当前目标 + 当前任务优先）
- 当前任务执行（AI 帮我一下 / 开始陪我做）
- 历史记录
- 统计
- 我的 / 设置

### App 端体验原则

- 每屏只做一件事
- 当前任务永远在最显眼位置
- AI 陪伴是任务层能力，不是全局聊天
- 底部导航固定，一键切换

## 7. Web 与 App 的信息架构差异

| 维度 | Web | 手机 App |
|------|-----|---------|
| 主要目的 | 介绍产品 / 登录入口 / 桌面访问 | 高频执行任务 |
| 导航方式 | 顶部导航 / 单页滚动 | 底部 Tab 导航 |
| 信息密度 | 可展示更多说明和概览 | 每屏只做一件事 |
| AI 入口 | 解释 AI 能力 | 深度嵌入任务执行 |
| 任务展示 | 可概览全部任务 | 当前任务优先 |
| 历史/统计 | 可大屏浏览 | 精简视图 |
| 情绪体验 | 专业可信 | 温暖陪伴 |

## 8. 路由与页面结构建议

### 方案 A：沿用 `/app`，内部切换 App Shell

```
/              Web Landing
/login         Web Login
/app           Mobile-first App Shell（内部视图切换）
```

- 优点：改动小，不拆子路由
- 缺点：Web/App 边界不够清晰

### 方案 B：新增子路由

```
/              Web Landing
/login         Web Login
/app/today     手机 App 今日行动
/app/history   手机 App 历史
/app/stats     手机 App 统计
/app/me        我的
```

- 优点：结构清晰，每个视图独立 URL
- 缺点：改动稍大，路由数增加

### 推荐

**V3.0 先采用方案 A**（App Shell 内部多视图），不一次性大规模重构所有路由。`/app` 内部使用状态驱动视图切换（Today / Records / Me），顶部 URL 不变。

原因：
- 改动面可控，风险低
- 不需要新增路由文件
- 后续可渐进式拆分子路由
- 执行方案阶段再根据真实代码决定是否拆子路由

## 9. App Shell 设计原则

- **手机端优先**：以 375-430px 宽度为设计基准
- **单屏只突出一个主动作**：不在一屏内堆叠多个功能模块
- **底部导航固定**：始终可见，一键切换视图
- **顶部只保留状态和当前页标题**：不堆砌操作按钮
- **不再把目标输入、任务列表、统计、历史、复盘全部堆在一个页面**
- **AI 陪伴是任务层能力，不是全局聊天入口**：陪伴面板只在当前任务上下文中出现
- **不出现"双重界面"**：不能在任务页面内部再塞一个完整的 App 壳

## 10. 底部导航设计

### 推荐：MVP 三 Tab

```
今日    记录    我的
```

- **今日**：当前目标 + 任务列表 + AI 辅助/陪伴入口（核心）
- **记录**：历史记录 + 统计概览
- **我的**：账号信息 + 设置 + 退出登录

### 为什么三个而不是四个

- 降低首次使用认知负担
- "历史"和"统计"天然关联，合并为一个 Tab 更自然
- 未来可根据使用数据拆分为四 Tab

### 备选：四 Tab

```
今日    历史    统计    我的
```

如果执行方案阶段判断拆分更合理，可升级为四 Tab。

## 11. 今日行动页结构

"今日"Tab 是用户打开 App 看到的第一屏，信息层级如下：

```
┌─────────────────────────────┐
│  顶部层                      │
│  - 温馨鼓励标题（如"今天是新的一天"）│
│  - 今日状态（已完成 X / 共 Y 项）  │
│  - 当前目标摘要（一句话）     │
├─────────────────────────────┤
│  任务层                      │
│  - 当前任务卡片（最优先）     │
│    ├ 任务标题                │
│    ├ 为什么做这一步（可选）   │
│    ├ [AI 帮我一下] [开始陪我做]│
│    └ [完成] 按钮             │
│  - 下一步预览（弱化展示）     │
│  - locked 任务（更弱化）      │
├─────────────────────────────┤
│  完成状态                    │
│  - 全部完成后显示提示/复盘入口│
├─────────────────────────────┤
│  底部层                      │
│  [今日]  [记录]  [我的]      │
└─────────────────────────────┘
```

### 关键约束

- 页面打开后，当前任务（解锁态）应在首屏可见，不需要滚动
- 已完成任务默认收起或折叠
- 不能出现"双重界面"——任务卡片内部不能再套一层完整的 App Shell
- GoalInput 在任务列表为空时显示，有任务时默认收起（点击可展开修改目标）

## 12. 任务执行页结构

当用户进入"执行当前任务"状态（点击 AI 帮我一下或开始陪我做）时：

```
┌─────────────────────────────┐
│  ← 返回今日                  │
│  当前任务标题                 │
│  为什么做这一步（一句）       │
├─────────────────────────────┤
│  一个最小动作                │
│  （AI 给出的当前步骤）        │
├─────────────────────────────┤
│  [AI 帮我一下] [开始陪我做]   │
│  （任务级入口按钮）           │
├─────────────────────────────┤
│  AI 面板（展开时）            │
│  - AI Assist：步骤/模板/清单  │
│  - AI Companion：逐步陪伴    │
│  - 固定反馈按钮，无自由输入框  │
├─────────────────────────────┤
│  [完成这一步]                │
│  [返回今日]                  │
└─────────────────────────────┘
```

### 关键约束

- 执行页不脱离今日页的上下文
- AI 面板覆盖在当前任务卡片下方，不是独立全屏
- 执行完成后回到今日页，显示下一步

## 13. AI 陪伴面板结构

- 陪伴面板是当前任务内的辅助层，不是聊天软件
- **不出现通用输入框**：使用固定反馈按钮（start / done / stuck / too_hard / encourage）
- 每次只推进当前一步
- AI 可以给材料、框架、清单，但不替用户提交最终结果
- 面板内只显示当前轮次的 AI 输出，不堆砌历史对话
- [DONE] 标记出现时，提示用户可以勾选完成

## 14. 历史 / 统计 / 我的页面结构

### 记录 Tab（历史 + 统计）

```
┌─────────────────────────────┐
│  统计概览（顶部卡片）         │
│  最近 7 天 | 最近 30 天      │
│  总完成数 · 连续行动天数     │
├─────────────────────────────┤
│  历史记录（下方列表）         │
│  按日期分组                  │
│  - 日期                      │
│  - 目标摘要                  │
│  - 任务完成情况              │
│  - 可展开查看详情            │
│  - 懒加载分页（上拉加载更多） │
└─────────────────────────────┘
```

### 我的 Tab

```
┌─────────────────────────────┐
│  账号信息                    │
│  - 邮箱                      │
│  - 登录状态                  │
├─────────────────────────────┤
│  设置                        │
│  - 修改密码（如有密码）       │
│  - 设置密码（如未设置）       │
├─────────────────────────────┤
│  退出登录                    │
└─────────────────────────────┘
```

## 15. PWA 与未来原生 App 的关系

- **当前 PWA 可以继续存在**：V3.0 不删除 PWA 支持，App Shell 本身就是 PWA 友好结构
- **V3.0 不做原生 App**：不写 React Native / Flutter / Swift / Kotlin 代码
- **先把手机端信息架构做成 App-like**：底部导航、单屏单任务、触控优先
- **未来如果做原生 App**：当前 App Shell 可作为迁移蓝图——每个 Tab 视图对应一个原生 Screen，API 层完全复用
- **不要因为 PWA 已存在就忽略 App UI 独立设计**：PWA 是技术手段，App Shell 是信息架构——这是两件事

## 17. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 改动面过大 | 一次重构太多文件，容易引入 bug | 拆成 V3.0A / V3.0B / V3.0C 子阶段 |
| 破坏现有任务逻辑 | UI 重组影响任务顺序执行、locked/unlocked | 先只重组 UI 层，不改 Hook 和 API |
| 双重界面再次出现 | AppShell 内部又套了一层 App 风格容器 | 明确 App Shell 只能有一层，Code Review 重点检查 |
| Web 与 App 入口混乱 | 用户不知道用 `/` 还是 `/app` | 保留 `/` 和 `/login` 为 Web，`/app` 为 App Shell，Landing CTA 指向 `/login` |
| AI 陪伴变聊天软件 | 新 UI 把陪伴面板做成全屏聊天 | 不新增自由输入框，保持任务上下文 + 固定反馈按钮 |
| 移动端和桌面端体验冲突 | Tailwind 响应式断点难以同时覆盖 | 用 CSS class 或条件渲染区分 Mobile Shell vs Desktop View |
| 底部导航与 PWA safe-area 冲突 | 底部导航被系统手势条遮挡 | 使用 `env(safe-area-inset-bottom)` padding |

- [ ] 今日行动页只突出当前目标和当前任务
- [ ] 当前任务卡片在首屏可见，无需滚动

## 19. 后续 Execution Plan 要点

V3.0 的下一步**不是直接让 Codex 写代码**，而是：

1. **Claude Code 先核验真实代码结构**：读取 `MainWorkspace.tsx`、`app/page.tsx`、所有相关组件，确认当前渲染树和状态管理拓扑
2. **Claude Code 输出 `Execution-Plan-V3.0-Web-App-Separation.md`**：基于真实代码结构，写出精确到组件/Props/Hook 的执行方案
3. **ChatGPT 审查执行方案**：评估改动画、风险、子阶段拆分
4. **Codex 按执行方案实现**
5. **Claude Code Review**：逐文件对比执行方案
6. **ChatGPT 最终验收**

### 建议子阶段拆分

| 子阶段 | 内容 | 风险 |
|:---:|------|:---:|
| V3.0A | App Shell + 底部导航 + TodayView（只重组 UI，不改逻辑） | 低 |
| V3.0B | RecordsView + ProfileView + 历史/统计迁移 | 低 |
| V3.0C | 移动端细节优化（safe-area、触控、滚动） | 低 |

**V3.0A 是 MVP**：做完后用户就能在手机上看到底部导航和今日行动独立视图，且不影响桌面端使用。
