## 一、路线重排背景

### 1.1 当前项目状态

AI Todo 已完成以下阶段：

| 版本 | 内容 | 状态 |
|------|------|:--:|
| V1.0 | 基础 Todo + AI 生成 + Supabase 同步 | ✅ 已归档 |
| V2.0 (Phase 12-15) | 历史 → 统计 → AI 复盘 → 智能调整 | ✅ 闭环 |
| V2.1 | Magic Link → Email + Password | ✅ 已完成 |
| V2.1-Follow-up | 自定义 SMTP（阿里云邮件推送） | ✅ 已完成 |
| V2.1B | OTP + Password 混合账号体系 | ✅ 已完成 |
| V2.2A | `/` `/login` `/app` 三页面路由拆分 + Auth 路由守卫 | ✅ 已完成 |
| V2.2B | `/login` 登录页视觉升级 | ✅ 已完成 |
| 自定义域名 | aitodoai.cn / www.aitodoai.cn | ✅ 线上验收通过 |

### 1.2 为什么重排

用户体验后确认：当前 `/app` 页面确实仍然不够美观，且存在功能堆叠问题。

**但新的产品决策是：不再优先推进 V2.2C / V2.2D 的页面美化和移动端优化。**

原因：

1. **页面优化重要，但不是当前最大产品验证点。** 继续美化可能消耗大量阶段资源，却不能验证 AI Todo 的核心差异化。
2. **当前更需要验证"用户是否真的需要任务级 AI 辅助"。** 这是 AI Todo 作为"AI 行动教练"区别于普通 Todo List 的关键。
3. **页面美化和多页面拆分容易打乱后续核心能力验证节奏。** 早期页面重构可能过早锁定信息架构，增加后续改造成本。
4. **安全底座必须先补齐。** 防滥用、忘记密码、错误提示等会影响真实用户使用，是后续 AI 能力的基础。

### 1.3 核心决策

```
页面优化、美观统一、移动端大优化、App Shell、多页面拆分
    → 全部后移到 V3.0

当前优先：
    V2.3 安全增强 → V2.4 AI 辅助执行 MVP → V2.5 AI 辅助增强 → V2.6 沉淀
```

## 四、V2.3：安全增强

### 4.1 定位

先补齐账号安全和反滥用能力。不改变任务核心流程，不做页面美化，不做 AI 辅助执行。

**一句话**：安全底座先稳，再开放更多能力。

### 4.2 候选范围

| # | 功能 | 说明 | 优先级 |
|---|------|------|:---:|
| 1 | **忘记密码** | 用户输入邮箱 → 发送重置密码邮件 → 设置新密码 | P1（必做） |
| 2 | **重置密码页面** | `/reset-password` 或 Supabase 回调页面，输入新密码 + 确认密码 | P1（必做） |
| 3 | **Cloudflare Turnstile** | 轻量防机器人方案，登录/注册表单加入人机验证 | P1（必做） |
| 4 | **登录/注册错误提示优化** | 错误信息脱敏（不暴露"用户不存在" vs "密码错误"），统一为"邮箱或密码错误" | P2 |
| 5 | **Supabase Auth 安全配置复查** | 检查 rate limit、session 过期时间、refresh token 策略、邮件模板安全性 | P2 |
| 6 | **邮件模板和回调链路复查** | 确认重置密码邮件模板、Confirm signup 邮件模板的文案和回调 URL 正确 | P2 |

### 4.3 明确不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 不做复杂账号中心 | 最小安全原则，不建用户资料编辑页 |
| 2 | 不做第三方 OAuth（Google/GitHub/微信） | 配置复杂、维护成本高、当前用户量不需要 |
| 3 | 不做会员系统 / 付费 | 产品早期，不引入商业化 |
| 4 | 不做团队权限 / RBAC | 个人工具定位 |
| 5 | 不做页面重构 | 全部留在 V3.0 |
| 6 | 不做 AI 辅助执行 | 属于 V2.4 |
| 7 | 不做 Turnstile 之外的验证码方案 | 保持简单 |

### 4.4 不改

- 不改 `useAuth` 核心登录/登出逻辑
- 不改 OTP + 密码混合账号体系
- 不改 Supabase `signInWithOtp` / `verifyOtp` / `signInWithPassword` 调用
- 不改 `/app` 任何组件
- 不改任务核心流程
- 不改 API Route（除 V2.3 新增的安全相关接口）

## 五、V2.4：任务级 AI 辅助执行 MVP

### 5.1 定位

AI 辅助用户完成任务，但不替用户完成任务。

这是 AI Todo 从"AI 拆任务 + 监督完成"升级为"AI 辅助推进任务"的**第一版**。基于现有 `/app` 单页 TaskItem / TaskList 接入，不依赖未来 V3.0 页面拆分。

**一句话**：用户看到每条任务时，可以点"AI 帮我推进"，AI 给出步骤、模板、检查清单或建议，用户自己执行。

### 5.2 MVP 范围

| # | 功能 | 说明 |
|---|------|------|
| 1 | **任务级"AI 帮我推进"入口** | 每个 TaskItem 卡片提供一个入口按钮/图标，打开任务级辅助面板 |
| 2 | **任务级辅助面板** | 面板绑定当前 `taskId` / `taskGroupId` / `goal`，显示在当前任务卡片下方或侧边 |
| 3 | **自由输入** | 用户输入需求（如"帮我写邮件模板"、"拆成更细的步骤"、"给我检查清单"） |
| 4 | **意图自动识别** | 后端根据用户输入 + task 上下文自动识别意图类型 |
| 5 | **AI 返回结构化输出** | 步骤 / 模板 / 初稿 / 检查清单 / 下一步建议 |
| 6 | **复制结果** | 支持一键复制 AI 输出内容 |
| 7 | **重新生成** | 不满意可以重新生成 |
| 8 | **最多一次轻量追问** | 允许一次追问（如"再详细一点"、"换个风格"），不形成长对话 |
| 9 | **用户自己执行** | AI 只提供建议和内容，用户自己判断、修改、执行、勾选完成 |

### 5.3 交互流程

```
┌──────────────────────────────────────┐
│  ☐ 撰写项目周报                      │  ← TaskItem
│     [AI 帮我推进 →]                  │  ← 入口按钮
├──────────────────────────────────────┤
│  ┌────────────────────────────────┐  │
│  │ 你想怎么推进这个任务？          │  │  ← 自由输入
│  │ [帮我写周报模板]  [发送]       │  │
│  └────────────────────────────────┘  │
│  ┌────────────────────────────────┐  │
│  │ AI 输出：                       │  │
│  │ ## 周报模板                     │  │
│  │ ### 本周完成                    │  │
│  │ - ...                           │  │
│  │ ### 下周计划                    │  │
│  │ - ...                           │  │
│  │                                 │  │
│  │ [📋 复制] [🔄 重新生成]        │  │
│  └────────────────────────────────┘  │
│  [追问：再精简一点]  [发送]         │  ← 最多一次追问
└──────────────────────────────────────┘
```

### 5.4 明确不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 不做全局聊天机器人 | 只在 TaskItem 上下文内，不脱离任务 |
| 2 | 不做长期聊天记录 | V2.6 只做轻量行为记录，不做完整对话存储 |
| 3 | 不做动态快捷建议 API | 属于 V2.5 |
| 4 | 不做 Streaming（SSE） | V2.5 评估是否加入；V2.4 用标准 request-response |
| 5 | 不做数据库聊天表 | 不存储完整对话 |
| 6 | 不让 AI 自动勾选完成 | Human-in-the-loop，完成权始终在用户 |
| 7 | 不让 AI 自动提交/发送/发布 | 不做任何自动化执行 |
| 8 | 不做跨任务上下文 | 每次辅助只针对当前任务 |
| 9 | 不做页面重构 | 基于现有 `/app` 单页 |

### 5.5 Human-in-the-Loop 边界

| 谁做 | 什么 |
|------|------|
| **用户** | 判断任务是否完成、勾选完成、执行关键操作、做最终决定 |
| **AI** | 建议步骤、生成模板、提供检查清单、给出初步方案、回答"怎么做" |

**AI 永远不替用户完成任务，只帮用户看清怎么做。**

## 六、V2.5：AI 辅助执行增强

### 6.1 定位

在 V2.4 MVP 验证有效后，增强 AI 辅助执行的体验和输出质量。

**一句话**：让 AI 辅助更聪明、更快捷、更自然。

### 6.2 候选范围

| # | 功能 | 说明 | 优先级 |
|---|------|------|:---:|
| 1 | **动态快捷建议** | 根据 task 标题和内容，自动生成 2-3 个建议入口（如"拆成更细步骤"、"写邮件模板"、"生成检查清单"），减少用户打字 | P2 |
| 2 | **意图识别增强** | 更精准地识别用户意图类型，减少"抱歉我不理解" | P2 |
| 3 | **输出类型优化** | 优化步骤/模板/清单/初稿/指令五种输出类型的格式和可操作性 | P2 |
| 4 | **更好的错误兜底** | AI 输出异常时的友好提示和重试引导 | P2 |
| 5 | **更好的复制体验** | 按输出段落分段复制、Markdown 渲染预览 | P3 |
| 6 | **支持更自然的一次追问** | 追问更流畅，上下文保留更好 | P2 |
| 7 | **Streaming 评估** | 评估是否加入 SSE streaming（`text/event-stream`）替代标准 request-response，提升即时反馈感 | P3（非默认必做） |

### 6.3 明确不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 不变成无限聊天 | 仍然最多 1 次追问 |
| 2 | 不做全自动 Agent | Human-in-the-loop 不变 |
| 3 | 不脱离当前任务上下文 | 每次辅助仍绑定 taskId |
| 4 | 不做跨任务长期记忆 | 留在后续版本评估 |
| 5 | 不做页面重构 | 属于 V3.0 |

## 七、V2.6：AI 辅助执行沉淀与轻量记录

### 7.1 定位

只沉淀轻量行为数据，帮助后续复盘、优化 AI 辅助质量。**不做完整聊天系统。**

**一句话**：记录"谁在哪个任务上用了什么 AI 帮助、有没有用"，不记录"说了什么"。

### 7.2 候选范围

| # | 字段 | 说明 |
|---|------|------|
| 1 | `taskId` | 关联到哪个任务 |
| 2 | `taskGroupId` | 关联到哪个任务组 |
| 3 | `assistIntent` | 意图类型（如 `breakdown_steps` / `write_template` / `checklist` / `general_help`） |
| 4 | `createdAt` | 请求时间 |
| 5 | `wasCopied` | 用户是否复制了结果 |
| 6 | `wasRegenerated` | 用户是否重新生成 |
| 7 | `wasHelpful` | 用户是否标记有帮助（可选轻量反馈） |
| 8 | `shortSummary` | 极短摘要（如"周报模板"，≤100 字） |

### 7.3 明确不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 不保存完整长对话 | 只存轻量行为数据 |
| 2 | 不做复杂知识库 | RAG 等留在后续评估 |
| 3 | 不做长期用户画像 | 不做 user profile AI 建模 |
| 4 | 不做跨任务大记忆系统 | 不做向量存储、不做 embedding |
| 5 | 不做自动化执行链路 | 不记录"AI 自动做了什么"——因为 AI 不做自动执行 |
| 6 | 数据库 schema 变更 | 如需新增表/字段，必须经单独架构评审 |

## 八、V3.0：正式 App 架构重构

页面优化、美观统一、App Shell、多页面拆分全部放到 V3.0。

**一句话**：基于已验证的核心能力，重新设计产品的视觉、页面结构和移动端体验。

### 8.2 V3.0 候选范围

| # | 模块 | 说明 |

| 2 | 不在 V2.4 做 | AI 辅助 MVP 阶段 |
| 3 | 不在 V2.5 做 | AI 辅助增强阶段 |
| 4 | 不在 V2.6 做 | AI 辅助沉淀阶段 |

### 8.4 V3.0 的设计原则

1. **吸收真实反馈**：V3.0 的信息架构应基于 V2.4-V2.6 用户真实使用行为，而非凭空设计
2. **服务核心能力**：页面结构服务于"AI 拆任务 → AI 辅助执行 → 复盘"核心流程
3. **移动端优先**：底部导航、触控友好、单手操作
4. **渐进式迁移**：不一次性重写，逐步从现有 `/app` 迁移到 App Shell

## 九、为什么这样排——决策理由

### 9.1 为什么暂停 V2.2C / V2.2D

| # | 理由 | 详细说明 |
|---|------|---------|
| 1 | **页面优化不是当前最大产品验证点** | `/app` 虽然不够美观，但功能完整可用。现在投入 2-3 轮对话做 UI 美化，不能验证 AI Todo 的核心差异化——"AI 能不能真的帮用户完成任务" |
| 2 | **继续美化可能消耗大量时间** | V2.2C 涉及 15+ 组件，V2.2D 涉及 3 个页面 + 全局 CSS。全部做完预计 3-5 轮对话，但这段时间可以用来验证一个更关键的能力 |
| 3 | **当前更需要验证任务级 AI 辅助** | AI Todo 和普通 Todo List 的区别不在 UI 好不好看，而在于 AI 是不是真的"教练"。V2.4 的任务级 AI 辅助是核心差异化的第一步 |
| 4 | **过早页面重构可能锁定错误架构** | 在没有 AI 辅助执行真实使用数据的情况下做多页面拆分，可能设计出不符合用户行为的信息架构，V3.0 时还得再改 |

### 9.2 为什么先做 V2.3

| # | 理由 | 详细说明 |
|---|------|---------|
| 1 | **安全是 AI 能力的基础** | 开放更多 AI 能力前，必须先防滥用。没有 Turnstile 的 AI 辅助 API 容易被脚本刷 |
| 2 | **忘记密码是真实用户刚需** | 真实用户一定会忘记密码。现在没有重置密码流程，用户只能重新注册 |
| 3 | **安全配置影响用户信任** | 错误提示脱敏、rate limit 配置、session 安全等直接影响用户对产品的信任 |
| 4 | **改动面小、风险可控** | V2.3 预计只涉及 3-6 个文件（useAuth 微调 + 新增重置密码页面 + Turnstile 集成），不会大面积影响现有功能 |

### 9.3 为什么 V2.4 优先于页面大重构

| # | 理由 | 详细说明 |
|---|------|---------|
| 1 | **AI 辅助执行是核心差异化** | AI Todo 的价值主张是"AI 行动教练"，不是"好看的 Todo List"。V2.4 直接体现这个价值主张 |
| 2 | **现有 `/app` 已经可以承载 MVP** | TaskItem / TaskList 已有稳定的组件结构和状态管理，AI 辅助面板可以直接挂载到 TaskItem 下方——不需要等多页面架构完成 |
| 3 | **先验证能力，再根据行为重构页面** | V2.4-V2.6 收集的真实使用数据（哪些任务类型最常请求 AI 帮助、用户使用频次、追问率、复制率）可以直接指导 V3.0 的信息架构设计 |
| 4 | **降低"做完没人用"的风险** | 如果先花大量时间做 V3.0 页面架构，再发现 AI 辅助执行的设计方向错了，前面做的页面重构可能白做 |

### 9.4 为什么 V3.0 再做页面架构

| # | 理由 | 详细说明 |
|---|------|---------|
| 1 | **吸收 V2.4-V2.6 的真实使用反馈** | 知道用户实际怎么用 AI 辅助后，再设计页面结构，而不是凭空猜测 |
| 2 | **避免过早设计错误的信息架构** | 例如：AI 辅助面板在 TaskItem 中的位置、历史记录中是否需要显示 AI 辅助记录、统计中是否需要 AI 辅助使用数据——这些都需要真实数据来回答 |
| 3 | **页面重构应服务于已验证的核心能力** | V3.0 不是为了好看而重构，而是为了让已验证的核心能力（AI 拆任务 + AI 辅助执行 + 复盘）在更好的信息架构中呈现 |
| 4 | **V2.4-V2.6 期间 `/app` 仍然可用** | 当前单页工作台虽然不够美观，但功能完整、lint 通过、build 成功、线上稳定。用户可以在上面正常使用 AI 辅助执行 |
