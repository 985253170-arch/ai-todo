## 第二部分 产品目标

### 2.1 产品总目标

AI Todo 的产品总目标是帮助用户将模糊目标拆解为具体、轻量、可执行的任务，并通过任务完成记录和基础激励反馈，帮助用户降低开始行动的难度。

第一版产品不追求复杂功能，而是优先验证一个核心问题：

用户是否愿意通过 AI 来帮助自己拆解任务，并在生成的任务列表中持续执行。

因此，V1.0 的重点不是做一个功能完整的任务管理系统，而是完成一个最小可用版本，让用户能够从“输入目标”顺利走到“执行任务”。

### 2.2 V1.0 核心目标

V1.0 需要完成以下核心目标：

第一，用户可以输入一个模糊目标。

例如：

我要学习 Python

我要准备面试

我要做一个小项目

我要减肥

我要提升英语能力

第二，系统可以调用 AI，将目标拆解为具体任务。

AI 返回的任务需要尽量满足以下标准：

任务清晰

步骤具体

难度适中

当天可以执行

不要一次生成过多任务

第三，用户可以查看 AI 生成的任务列表。

任务列表需要以清晰、简单的方式展示，让用户一眼知道接下来要做什么。

第四，用户可以勾选任务完成状态。

用户完成任务后，可以点击任务前方的勾选框，将任务标记为已完成。

第五，任务数据可以保存。

用户刷新页面后，已经生成的任务和完成状态不应丢失。

### 2.3 MVP 目标

本项目第一版采用 MVP 思路开发。

MVP 的意思是最小可用产品。它不是最终版本，而是用最少功能验证核心需求是否成立。

AI Todo V1.0 的 MVP 目标为：

用户输入一个目标后，AI 能够生成一组可执行任务；用户可以勾选任务完成状态；任务和完成状态能够被保存。

如果这个闭环可以正常运行，说明 V1.0 达到基础目标。

MVP 的完整流程如下：

用户打开网页

↓

输入一个目标

↓

点击 AI 拆分按钮

↓

AI 生成任务列表

↓

用户查看任务

↓

用户勾选完成任务

↓

系统保存任务状态

### 2.4 用户价值目标

V1.0 希望为用户提供三个核心价值。

第一，降低开始行动的难度。

用户不需要自己拆解复杂目标，只需要输入一句话，AI 就能给出今日可执行的任务建议。

第二，提高任务清晰度。

相比“我要学习 Python”这种模糊目标，AI 生成的任务会更加具体，例如“安装 Python 环境”“学习变量概念”“完成 2 道练习”。

第三，增强完成反馈。

用户可以通过勾选任务获得基础完成反馈，知道自己今天完成了哪些内容，从而获得一定的成就感。

### 2.5 产品体验目标

V1.0 的体验目标是简单、清晰、低门槛。

页面不追求复杂设计，也不追求大量功能，而是保证用户第一次打开页面时，可以快速理解如何使用。

具体体验目标如下：

用户进入首页后，3 秒内能看懂产品用途

用户不需要学习成本即可输入目标

用户点击按钮后可以获得任务结果

任务列表清晰易读

勾选完成操作简单直接

整体页面适配电脑和手机浏览器

### 2.6 技术实现目标

虽然 PRD 主要关注产品需求，但 V1.0 仍需要明确基础技术目标，方便后续开发验收。

V1.0 技术实现目标如下：

项目可以在本地正常运行

首页可以正常加载

输入框可以接收用户输入

按钮可以触发 AI 任务拆解

AI 返回内容可以转换为任务列表

任务完成状态可以被修改

任务数据可以被保存

页面刷新后任务状态不丢失

项目可以部署到线上访问

### 2.7 第一版成功标准

V1.0 是否成功，不看功能数量，而看核心流程是否跑通。

只要满足以下条件，即可认为第一版成功：

第一，用户可以输入一个目标。

第二，AI 可以生成 3 到 8 条可执行任务。

第三，用户可以勾选任务完成状态。

第四，刷新页面后，任务和完成状态仍然存在。

第五，整个流程没有明显报错。

第六，用户能理解这个产品是用来帮助自己拆解目标和执行任务的。

### 2.8 暂不追求的目标

为了避免第一版范围过大，V1.0 暂不追求以下目标：

不追求复杂 UI 设计

不追求完整用户系统

不追求多用户协作

不追求日历提醒

不追求任务分类体系

不追求复杂数据统计

不追求移动 App

不追求商业化付费

不追求完整 AI 陪伴系统

这些能力可以作为后续版本规划，但不应影响 V1.0 的开发进度。

### 2.9 阶段性目标总结

AI Todo V1.0 的阶段性目标可以总结为一句话：

用最简单的方式验证“AI 能否帮助用户把模糊目标拆成可执行任务，并让用户愿意按照任务列表完成行动”。

只要这一点成立，后续就可以继续扩展任务追踪、个性化规划、激励反馈、长期目标管理等高级能力。

## 第三部分 目标用户与用户画像

### 3.1 目标用户定位

AI Todo V1.0 的目标用户不是所有待办工具用户，而是那些“有目标，但缺少拆解能力和执行规划能力”的用户。

第一版产品优先面向以下三类用户：

大学生

刚毕业求职者

AI 学习者 / 编程学习者

选择这三类用户作为第一批目标用户，主要基于以下原因：

第一，他们经常面对较大的目标。

例如学习一门技能、准备一场面试、完成一个项目、提升英语能力等。这类目标通常不是一天完成的，需要被拆解成多个小步骤。

第二，他们往往缺少经验。

小白用户知道自己想变好，但不知道如何开始，也不知道任务应该拆到什么程度才算可执行。

第三，他们对 AI 工具有接受度。

学生、求职者和 AI 学习者更愿意尝试新工具，也更容易理解 AI 拆解任务的价值。

第四，他们的使用场景足够明确。

这类用户通常会在学习、求职、项目练习、自我提升等场景中使用产品，方便 V1.0 聚焦核心功能。

### 3.2 核心用户画像一：大学生

#### 基础特征

大学生通常有学习、考试、课程作业、技能提升等需求，但在执行过程中容易出现拖延和计划混乱。

他们可能会设定目标，例如：

我要复习英语四级

我要准备期末考试

我要学习 Python

我要完成课程论文

我要做一个作品集项目

#### 核心痛点

大学生的主要痛点不是没有目标，而是不知道如何把目标拆成每天可以完成的小任务。

例如，用户知道自己要学习 Python，但不知道今天应该先看视频、写代码、做练习，还是搭建环境。

目标过大时，用户容易产生压力，最终出现拖延。

#### 使用场景

大学生可能在以下场景使用 AI Todo：

晚上准备第二天学习计划

考试前拆分复习任务

学习一门新技能时规划步骤

做课程作业前拆解任务

准备实习或作品集时制定计划

#### 对产品的期待

大学生希望产品能够：

帮自己把大目标拆成小任务

告诉自己今天先做什么

任务不要太复杂

最好能有一点激励反馈

操作简单，不需要学习成本

### 3.3 核心用户画像二：刚毕业求职者

#### 基础特征

刚毕业求职者通常面临简历优化、岗位投递、面试准备、项目复盘等任务。他们压力较大，但又缺少系统规划能力。

他们可能会设定目标，例如：

我要准备数据分析面试

我要优化简历

我要投递 20 个岗位

我要准备自我介绍

我要学习 Excel 和 SQL

#### 核心痛点

求职者的痛点是任务多、压力大、优先级不清楚。

他们知道自己要找工作，但不知道每天应该优先做什么。很多人会在“改简历、看岗位、学技能、刷题、准备面试”之间来回切换，导致效率较低。

#### 使用场景

求职者可能在以下场景使用 AI Todo：

每天早上制定求职任务

面试前拆分准备事项

投递前整理岗位要求

复盘面试失败原因

规划一周求职行动

#### 对产品的期待

求职者希望产品能够：

根据目标生成清晰行动清单

帮助自己分清优先级

任务具体到可以马上做

减少焦虑感

让每天的求职过程更有秩序

### 3.4 核心用户画像三：AI 学习者 / 编程学习者

#### 基础特征

AI 学习者和编程学习者通常有强烈的学习意愿，但容易被复杂概念和工具链劝退。

他们可能会设定目标，例如：

我要学会用 AI 做项目

我要开发一个小程序

我要学习 Next.js

我要接入 OpenAI API

我要做一个个人网站

#### 核心痛点

这类用户最大的问题是路径不清晰。

他们经常不知道：

先学什么

后学什么

哪些内容现在不需要学

今天应该写什么代码

遇到问题时下一步该怎么查

如果没有清晰拆解，他们容易在教程、工具和概念之间迷路。

#### 使用场景

AI 学习者和编程学习者可能在以下场景使用 AI Todo：

开始一个新项目之前拆解开发步骤

学习新技术时制定每日计划

遇到复杂任务时拆成小任务

根据学习进度调整任务

记录已完成的学习内容

#### 对产品的期待

这类用户希望产品能够：

把复杂技术目标拆成小白能理解的步骤

告诉自己当前阶段不要学太多无关内容

每次只给出少量可执行任务

支持长期学习计划

能够记录学习进度

### 3.5 非目标用户

为了保证 V1.0 聚焦，以下用户暂时不是第一版重点服务对象：

第一，重度项目管理用户。

例如团队负责人、项目经理、企业协作用户。这类用户通常需要多人协作、权限管理、任务分配、看板、甘特图等复杂能力，不适合 V1.0 优先支持。

第二，专业效率工具用户。

例如已经深度使用 Notion、Todoist、TickTick、飞书等工具的高级用户。他们对工具灵活性、跨平台同步、自动化能力要求较高，V1.0 难以满足全部需求。

第三，企业用户。

企业用户通常涉及团队管理、数据安全、权限控制、组织架构等复杂需求，不适合作为个人练手项目的第一阶段目标。

第四，纯提醒需求用户。

如果用户只需要简单提醒，例如“晚上八点提醒我喝水”，传统提醒工具已经可以满足，不是本产品第一版重点场景。

### 3.6 用户核心需求总结

综合以上三类目标用户，AI Todo V1.0 需要优先满足以下核心需求：

第一，把模糊目标拆成具体任务。

用户输入一个较大的目标后，系统需要生成清晰、具体、可执行的任务列表。

第二，降低开始行动的心理压力。

任务不能太多，也不能太复杂。第一版应尽量控制任务数量，让用户感觉“我现在就可以开始做”。

第三，提供基础执行反馈。

用户完成任务后，可以勾选完成，并看到自己的完成情况。

第四，保存任务和完成状态。

用户刷新页面后，任务不应丢失，避免降低信任感。

第五，保持使用流程简单。

目标用户中包含大量小白用户，因此产品操作不能复杂。首页应尽量直接，让用户打开后就知道怎么用。

### 3.7 用户画像总结

AI Todo V1.0 的核心用户可以概括为：

有目标、有行动意愿，但缺少任务拆解能力和持续执行反馈的小白用户。

他们不是没有目标，而是目标太大、步骤太乱、反馈太少。

因此，AI Todo V1.0 应该优先帮助他们完成一件事：

把“我想做”变成“我现在可以做什么”。

## 第四部分 用户使用场景与用户流程

### 4.1 使用场景概述

AI Todo V1.0 的核心使用场景是：用户有一个想完成的目标，但不知道如何拆解成具体任务，于是打开产品，让 AI 帮助自己生成可执行的任务清单。

本产品第一版不追求覆盖所有任务管理场景，而是聚焦“目标拆解”这一高频入口。

典型场景包括：

用户准备学习一项新技能

用户准备面试或求职

用户想完成一个小项目

用户想制定当天计划

用户想把长期目标拆成当天任务

在这些场景中，用户的核心需求不是简单记录待办事项，而是希望系统告诉自己“现在应该先做什么”。

### 4.2 核心使用场景一：学习计划拆解

#### 场景描述

用户想学习一项新技能，但不知道第一步该做什么。

例如用户输入：

我要学习 Python

系统通过 AI 自动拆解为当天可执行任务。

#### 用户痛点

用户知道自己想学习 Python，但不知道学习顺序，也不知道今天应该完成哪些内容。如果直接搜索教程，容易被大量信息淹没，反而更难开始。

#### 产品解决方式

AI 根据用户输入的目标，生成少量清晰任务，例如：

安装 Python 环境

了解变量和数据类型

跟着教程写一个简单示例

完成 2 个基础练习

记录今天遇到的问题

#### 预期结果

用户看到任务后，能够马上开始执行，而不是继续停留在“我要学习”的模糊状态。

### 4.3 核心使用场景二：求职准备拆解

#### 场景描述

用户准备找工作或准备面试，但不知道如何安排当天任务。

例如用户输入：

我要准备数据分析岗位面试

系统通过 AI 自动拆解为面试准备清单。

#### 用户痛点

求职准备涉及简历、岗位 JD、项目复盘、自我介绍、面试题等多个部分。用户容易因为任务太多而焦虑，不知道优先做什么。

#### 产品解决方式

AI 将求职目标拆解为具体任务，例如：

阅读 3 个目标岗位 JD

提炼岗位要求中的高频技能

修改简历中与岗位相关的项目描述

准备 1 分钟自我介绍

模拟回答 3 个常见面试问题

#### 预期结果

用户可以按照任务清单逐步推进求职准备，减少焦虑感，提高行动效率。

### 4.4 核心使用场景三：小项目开发拆解

#### 场景描述

用户想完成一个小项目，但不知道完整开发流程。

例如用户输入：

我要做一个 AI 待办清单项目

系统通过 AI 自动拆解为项目开发任务。

#### 用户痛点

小白用户面对项目开发时，容易不知道先做需求、页面、数据库还是代码。任务边界不清楚，容易一上来就陷入技术细节。

#### 产品解决方式

AI 将项目目标拆解为阶段性任务，例如：

明确项目核心功能

绘制首页草图

创建项目目录

完成输入框和按钮

接入 AI 拆解接口

实现任务列表展示

保存任务数据

#### 预期结果

用户可以看到一个清晰的项目推进路径，知道当前阶段应该做什么。

### 4.5 核心使用场景四：每日计划生成

#### 场景描述

用户每天早上或晚上打开产品，输入当天目标，让 AI 生成今日任务。

例如用户输入：

我明天想高效学习 3 小时

系统通过 AI 生成一份轻量计划。

#### 用户痛点

用户有时间，但不知道如何合理安排。如果计划太满，容易执行失败；如果计划太模糊，又无法产生行动。

#### 产品解决方式

AI 将用户目标拆成适合当天完成的任务，例如：

复习昨天学习内容 20 分钟

学习一个新知识点 40 分钟

完成练习 60 分钟

整理错题或问题 30 分钟

总结今天收获 10 分钟

#### 预期结果

用户可以快速获得当天行动清单，并通过勾选任务获得完成反馈。

### 4.7 主流程说明

#### 第一步：进入首页

首页需要清晰展示产品用途，让用户知道可以在输入框中输入目标，并点击按钮生成任务。

#### 第二步：输入目标

V1.0 不要求用户按照固定格式输入，系统需要尽量支持自然语言表达。

#### 第三步：点击 AI 拆分按钮

用户点击按钮后，系统开始处理用户输入。

按钮点击后需要给出加载状态，例如：

AI 正在拆解任务...

这样可以避免用户误以为系统没有反应。

#### 第四步：系统调用 AI

系统将用户输入内容发送给 AI，并要求 AI 返回结构化任务列表。

AI 返回内容应尽量控制在 3 到 8 条任务之间，避免任务过多导致用户压力过大。

#### 第五步：展示任务列表

系统将 AI 返回的任务展示为待办清单。

每一条任务应包含：

任务标题

完成状态

勾选操作

V1.0 不要求任务包含截止时间、标签、优先级等复杂字段。

#### 第六步：用户勾选完成

用户完成任务后，可以点击任务前方的勾选框。

任务状态由未完成变为已完成。

已完成任务可以通过删除线、颜色变化或勾选状态进行区分。

#### 第七步：保存任务状态

系统需要保存任务内容和完成状态。

用户刷新页面后，任务列表和完成状态仍应存在。

V1.0 可以先使用本地存储或简单数据库实现保存，具体技术实现由技术方案文档决定。

### 4.8 异常流程

#### 异常情况一：用户没有输入内容

如果用户未输入任何内容就点击按钮，系统应提示：

请先输入一个目标

系统不应调用 AI 接口。

#### 异常情况二：输入内容过短

如果用户只输入一个字或无意义内容，例如：

啊

系统可以提示：

请输入更具体的目标，例如：我要学习 Python

#### 异常情况三：AI 生成失败

如果 AI 接口调用失败，系统应提示：

任务生成失败，请稍后重试

同时保留用户已经输入的目标，方便用户重新点击按钮。

#### 异常情况四：AI 返回内容格式异常

如果 AI 返回的内容无法被正常解析为任务列表，系统可以展示通用错误提示：

任务解析失败，请重新生成

后续技术实现中需要通过提示词约束 AI 返回固定格式，降低解析失败概率。

#### 异常情况五：网络异常

如果用户网络异常，系统应提示：

网络连接异常，请检查网络后重试

#### 异常情况六：保存失败

如果任务保存失败，系统应提示：

任务暂时无法保存，请稍后重试

但不应立即清空当前页面上的任务内容，避免用户数据丢失。

### 4.10 关键转化点

V1.0 需要重点关注以下关键转化点：

第一，用户是否愿意输入目标。

如果首页说明不清楚，用户可能不知道应该输入什么。因此首页需要提供示例提示。

第二，用户是否愿意点击 AI 拆分按钮。

按钮文案需要明确，例如“AI 拆分任务”比“提交”更清楚。

第三，AI 生成结果是否可执行。

如果任务太空泛，用户会失去信任。因此 AI 生成任务必须具体、轻量、可执行。

第四，用户是否愿意勾选任务完成。

勾选是用户产生执行反馈的关键动作。交互必须简单明显。

第五，用户刷新后是否还能看到任务。

如果刷新后任务丢失，会严重影响产品可信度。因此保存能力是 V1.0 的基础要求。

后续所有功能扩展，都应该围绕这个闭环增强，而不是打断这个闭环。

## 第五部分 功能需求清单

### 5.1 功能需求概述

AI Todo V1.0 的核心功能围绕一个完整闭环展开：

输入目标 → AI 拆解任务 → 展示任务列表 → 勾选完成 → 保存状态

第一版产品不追求复杂功能，而是优先保证核心流程完整、稳定、易理解。

V1.0 的功能可以拆分为以下 6 个模块：

首页展示模块

目标输入模块

AI 任务拆解模块

任务列表模块

任务完成状态模块

任务保存模块

其中，AI 任务拆解是产品的核心功能，任务列表展示和状态保存是基础支撑能力。

### 5.2 功能模块一：首页展示

#### 功能说明

首页是用户进入产品后的第一个页面，需要让用户快速理解产品用途，并引导用户输入目标。

首页应保持简洁，不加入过多复杂入口。

#### 页面内容

首页需要包含以下内容：

产品名称

一句话介绍

目标输入框

AI 拆分按钮

示例目标提示

任务列表展示区域

#### 推荐文案

产品名称：

AI Todo

一句话介绍：

把模糊目标拆成今天可以执行的任务。

输入框占位提示：

例如：我要学习 Python / 我要准备面试 / 我要做一个小项目

按钮文案：

AI 拆分任务

#### 功能规则

首页打开后，默认展示输入框和按钮。

如果还没有生成任务，任务列表区域可以展示空状态提示。

空状态文案建议：

输入一个目标，让 AI 帮你拆成可执行任务。

#### 验收标准

用户打开页面后，可以看到产品名称。

用户可以看到输入框。

用户可以看到 AI 拆分按钮。

用户可以理解这个产品是用来拆解任务的。

页面在电脑和手机浏览器中都能正常展示。

### 5.3 功能模块二：目标输入

#### 功能说明

用户可以在输入框中输入自然语言目标，系统根据该目标生成任务列表。

用户不需要按照固定格式输入。

#### 输入示例

用户可以输入：

我要学习 Python

我要准备数据分析面试

我要做一个 AI 待办清单项目

我要减肥

我要提升英语口语

#### 输入规则

V1.0 支持普通自然语言输入。

输入内容不能为空。

输入内容建议限制在 5 到 100 个字符之间。

如果输入为空，系统应提示：

请先输入一个目标。

如果输入过短，系统可以提示：

请输入更具体的目标，例如：我要学习 Python。

如果输入过长，系统可以提示：

目标描述太长，请简化为一句话。

#### 验收标准

用户可以在输入框中正常输入文字。

用户未输入内容时，点击按钮不会调用 AI。

用户未输入内容时，页面会给出明确提示。

用户输入有效目标后，可以继续生成任务。

### 5.4 功能模块三：AI 任务拆解

#### 功能说明

AI 任务拆解是 V1.0 的核心功能。

用户输入目标后，点击按钮，系统调用 AI，将用户目标拆解为具体、轻量、可执行的任务列表。

#### 触发方式

用户点击：

AI 拆分任务

系统开始调用 AI。

#### 加载状态

AI 生成过程中，按钮或页面需要展示加载状态。

推荐文案：

AI 正在拆解任务...

加载过程中，按钮应避免被重复点击，防止重复请求。

#### AI 生成要求

AI 返回任务需要满足以下要求：

数量控制在 3 到 8 条之间

每条任务应具体明确

每条任务应适合当天执行

每条任务不宜过长

避免生成空泛任务

避免一次性安排过重任务

尽量使用动词开头

#### 好的任务示例

针对目标：

我要学习 Python

好的任务结果：

安装 Python 开发环境

学习变量和数据类型

跟着教程写一个 print 示例

完成 2 道变量练习题

记录今天遇到的 1 个问题

#### 不好的任务示例

不推荐生成：

努力学习 Python

全面掌握 Python

成为编程高手

这些任务太空泛，不适合直接执行。

#### AI 返回格式建议

为了方便系统解析，AI 应返回结构化数据。

建议格式：

{

"tasks": [

{

"title": "安装 Python 开发环境"

},

{

"title": "学习变量和数据类型"

},

{

"title": "完成 2 道变量练习题"

}

}

#### 异常处理

如果 AI 调用失败，提示：

任务生成失败，请稍后重试。

如果 AI 返回格式异常，提示：

任务解析失败，请重新生成。

如果网络异常，提示：

网络连接异常，请检查后重试。

#### 验收标准

用户输入目标后，点击按钮可以触发 AI 生成。

AI 可以返回 3 到 8 条任务。

任务内容具体、清晰、可执行。

加载过程中按钮不会重复触发请求。

AI 失败时页面不会崩溃。

AI 失败时用户输入内容不会被清空。

### 5.5 功能模块四：任务列表展示

#### 功能说明

系统需要将 AI 生成的任务展示为待办清单，方便用户查看和执行。

#### 展示内容

每条任务至少包含：

勾选框

任务标题

完成状态

V1.0 暂不要求展示：

截止时间

优先级

标签

任务备注

子任务

任务负责人

#### 展示规则

任务列表按照 AI 返回顺序展示。

未完成任务默认展示为普通状态。

已完成任务展示为完成状态。

完成状态可以通过以下方式体现：

勾选框被选中

任务文字出现删除线

任务样式变弱

#### 空状态

如果当前没有任务，任务区域显示空状态文案：

还没有任务，输入一个目标试试。

#### 验收标准

AI 生成任务后，页面可以展示任务列表。

每条任务都有勾选框。

每条任务标题显示清楚。

没有任务时，页面有空状态提示。

任务列表在手机和电脑上都能正常查看。

### 5.6 功能模块五：任务完成状态

#### 功能说明

用户可以通过点击勾选框，将任务标记为完成或未完成。

这是 V1.0 的基础执行反馈功能。

#### 操作规则

用户点击未完成任务的勾选框后，任务变为已完成。

用户再次点击已完成任务的勾选框后，任务可以恢复为未完成。

#### 状态规则

任务状态分为两种：

未完成

已完成

V1.0 暂不支持：

进行中

已取消

已延期

已失败

已归档

#### 完成反馈

用户勾选任务后，页面可以显示简单完成进度。

例如：

已完成 3 / 5

该功能可以作为 V1.0 推荐功能，如果开发时间不足，可以延后。

#### 验收标准

用户可以将任务标记为已完成。

用户可以将已完成任务恢复为未完成。

已完成任务与未完成任务有明显视觉区别。

勾选状态刷新后不丢失。

### 5.7 功能模块六：任务保存

#### 功能说明

系统需要保存用户生成的任务和完成状态，避免刷新页面后数据丢失。

任务保存是 V1.0 的必要功能。

#### 保存内容

每条任务至少需要保存以下信息：

任务标题

完成状态

创建时间

如果保存整个任务组，还可以保存：

用户输入的原始目标

任务生成时间

任务列表

#### 保存方式

V1.0 可采用两种方案之一：

方案一：本地存储。

适合最早期练手版本，开发简单，不需要登录系统。

方案二：数据库存储。

适合后续扩展，可支持历史记录、多设备同步和用户系统。

V1.0 初期建议先使用本地存储，跑通核心流程后，再升级到数据库。

#### 保存规则

AI 生成任务后，系统应自动保存任务。

用户勾选任务完成状态后，系统应自动更新保存状态。

用户刷新页面后，系统应自动读取上一次保存的任务列表。

#### 验收标准

### 7.6 V1.0 推荐保存方式

V1.0 推荐优先使用本地存储保存数据。

也就是使用浏览器 localStorage 保存当前任务组。

这样做的原因是：

第一，开发复杂度低。

不需要一开始就接入数据库、登录系统和用户权限，适合小白练手。

第二，能够快速跑通核心闭环。

V1.0 的重点是验证“输入目标、AI 拆解、任务勾选、刷新恢复”这一流程，而不是做完整账号系统。

第三，方便后续升级。

当本地版本稳定后，可以再升级到数据库保存，例如 Supabase、PostgreSQL 或其他云数据库。

### 7.7 localStorage 保存规则

#### 保存时机

系统应在以下时机保存数据：

第一，AI 成功生成任务后，立即保存任务组数据。

第二，用户勾选任务完成状态后，立即更新保存数据。

第三，用户取消完成状态后，立即更新保存数据。

第四，用户清空任务后，删除本地保存数据。

#### 读取时机

系统应在页面初始化时读取本地保存数据。

如果存在已保存任务组，则自动恢复并展示。

如果不存在已保存任务组，则展示空状态。

#### 保存键名建议

localStorage 的键名建议为：

ai\_todo\_current\_task\_group

{

"id": "group\_001",

"goal": "我要学习 Python",

"tasks": [

{

"id": "task\_001",

### 7.8 数据更新规则

#### AI 生成新任务时

当用户输入目标并成功生成任务后，系统应创建新的任务组。

如果本地已经存在旧任务组，V1.0 可以直接用新任务组覆盖旧任务组。

V1.0 暂不保留历史任务组。

后续版本可以增加历史记录功能。

#### 用户勾选任务时

当用户勾选某条任务后，系统需要更新该任务的 completed 字段。

例如：

{

"id": "task\_001",

"title": "安装 Python 开发环境",

"completed": true

}

同时更新该任务和任务组的 updatedAt 字段。

#### 用户取消勾选时

当用户取消勾选已完成任务后，系统需要将 completed 字段改回 false。

同时更新 updatedAt 字段。

#### 用户清空任务时

如果 V1.0 实现“清空任务”功能，系统应删除当前任务组数据，并让页面回到空状态。

### 7.9 数据校验规则

为了避免页面异常，系统在读取本地保存数据时需要进行基础校验。

需要校验的内容包括：

数据是否存在

数据是否可以被正常解析

是否包含 goal 字段

是否包含 tasks 字段

tasks 是否为数组

每条任务是否包含 title 字段

completed 是否为布尔值

如果本地数据格式异常，系统应忽略异常数据，并展示空状态。

不应因为本地数据异常导致页面崩溃。

### 7.10 数据异常处理

#### 本地数据不存在

如果用户第一次打开产品，本地没有保存数据，页面展示空状态。

提示文案：

还没有任务，输入一个目标试试。

#### 本地数据损坏

如果 localStorage 中的数据无法解析，系统应清除异常数据，并展示空状态。

#### 保存失败

如果本地保存失败，系统可以提示：

任务暂时无法保存，请稍后重试。

但不应立即清空当前页面上的任务内容。

#### 读取失败

如果读取本地数据失败，系统展示空状态，不影响用户重新生成任务。

### 7.11 数据安全与隐私说明

V1.0 使用本地存储保存数据，数据仅保存在用户当前浏览器中。

这意味着：

数据不会自动同步到其他设备

更换浏览器后看不到原任务

清除浏览器缓存可能导致数据丢失

暂不涉及用户账号和云端数据库

V1.0 不应要求用户输入敏感信息，例如身份证号、银行卡号、密码、私人地址等。

如果后续版本加入登录和云端同步，需要单独补充隐私策略和数据安全设计。

### 7.12 后续数据库升级方向

虽然 V1.0 推荐使用 localStorage，但为了后续扩展，可以提前考虑数据库升级方向。

后续如果加入用户登录和历史记录，可以设计以下数据表：

#### 用户表 users

用于保存用户账号信息。

#### 任务组表 task\_groups

用于保存用户每次输入目标后生成的一组任务。

#### 任务表 tasks

用于保存每一条具体任务。

#### 未来可扩展字段

后续版本可以增加：

用户 ID

任务优先级

任务标签

任务预计耗时

任务完成时间

任务所属目标

AI 生成原因

任务难度

用户反馈

拖延原因

这些字段不在 V1.0 中实现，只作为未来扩展方向。

### 7.13 数据结构总结

AI Todo V1.0 的数据结构应保持简单。

第一版只需要围绕一个任务组保存数据：

任务组

├── 用户目标

├── 任务列表

│ ├── 任务标题

│ ├── 完成状态

│ ├── 创建时间

│ └── 更新时间

└── 任务组创建和更新时间

V1.0 的数据保存目标可以总结为一句话：

用户生成的任务和勾选状态，刷新页面后不能丢失。

## 第八部分 AI 生成规则与提示词要求

### 8.1 章节目标

AI Todo V1.0 的核心能力来源于大语言模型。

为了保证 AI 每次都能生成高质量、可执行的任务，而不是随机发挥，需要提前定义 AI 的行为规则。

本章规定 AI 应遵循的生成原则、输出规范和质量要求。

本章不涉及具体 Prompt 内容，而是定义 AI 应该完成什么工作、遵守什么规则。

后续技术方案将依据本章内容设计正式 Prompt。

## 8.2 AI 的职责

AI 在本产品中的角色不是聊天机器人。

而是：

**任务规划助手（Task Planner）**

AI 的唯一职责是：

根据用户目标，生成具体、可执行、合理的行动计划。

AI 不负责：

长篇聊天

闲聊

编写文章

回答百科知识

解决所有问题

AI 的核心任务只有一句话：

**帮助用户开始行动。**

## 8.3 AI 的核心原则

AI 每次生成任务时，都必须遵循以下原则。

### 原则一：目标理解

AI 应首先理解用户真正想完成什么。

例如：

用户输入：

我要学 Python

真正目标不是：

学 Python。

而是：

开始学习 Python，并建立学习路径。

AI 不应机械拆字。

而应理解真实意图。

### 原则二：拆解优先

AI 不直接给答案。

AI 优先拆步骤。

例如：

错误：

Python 是一种解释型语言……

正确：

安装 Python

↓

学习变量

↓

写 Hello World

↓

完成两个练习

### 原则三：可执行

生成内容必须能够立即行动。

例如：

错误：

努力学习

提高能力

坚持下去

正确：

安装 Python

观看变量视频20分钟

完成2道练习

用户看完以后：

应该知道：

我现在马上可以开始。

### 原则四：降低行动门槛

AI 不应该一次安排过多任务。

例如：

错误：

学习 Python 基础

学习函数

学习类

学习模块

学习数据库

学习网络

学习 Flask

……

正确：

今天：

安装环境

学习变量

完成两个练习

AI 应帮助用户开始，而不是制造压力。

### 原则五：适度鼓励

AI 可以给予简单鼓励。

例如：

今天完成这些任务即可，不需要追求一次学完。

但鼓励应简短。

避免空洞鸡汤。

## 8.4 任务生成规范

AI 返回任务时，应满足以下要求。

### 数量

建议：

3～8 条。

过少：

用户感觉价值不足。

过多：

用户容易放弃。

### 长度

每条任务：

建议 10～30 个字。

避免：

一大段文字。

### 表达方式

推荐：

动词开头。

例如：

安装 Python

学习变量

完成练习

整理笔记

避免：

关于 Python 的学习

变量知识理解

……

### 难度

AI 应按照：

简单

↓

普通

↓

稍难

的顺序安排。

不要第一步就安排困难任务。

### 时间控制

AI 应默认：

今天可以完成。

不应生成：

需要连续几个月完成的大目标。

## 8.5 AI 禁止行为

AI 不允许：

第一：

生成无法执行的话。

例如：

努力学习

保持热爱

成为高手

第二：

一次安排大量任务。

第三：

生成互相重复任务。

第四：

生成与目标无关任务。

第五：

回答百科知识。

例如：

Python 是……

产品目标不是百科。

## 8.6 特殊场景处理

如果用户输入：

我要减肥

AI 应拆：

步行8000步

喝2000ml水

减少含糖饮料

力量训练20分钟

而不是：

减肥需要长期坚持……

如果用户输入：

我要做一个APP

AI 应拆：

确定项目主题

完成需求文档

画页面草图

创建项目

……

而不是：

APP 开发流程包括……

## 8.7 输出格式规范

AI 返回内容必须采用统一格式。

推荐 JSON：

{

"tasks": [

{

"title": "安装 Python"

},

{

"title": "学习变量"

}

]

}

禁止：

Markdown。

禁止：

长篇自然语言。

禁止：

解释性文字。

## 8.8 AI 质量标准

每次 AI 返回内容，应满足以下检查。

✅ 是否真正理解目标？

✅ 是否拆成步骤？

✅ 是否今天可以完成？

✅ 是否没有废话？

✅ 是否没有重复？

✅ 是否容易执行？

只有全部满足，才能返回给用户。

## 8.9 后续升级方向

未来版本可以增加：

AI 根据历史任务自动调整计划。

AI 根据完成率减少或增加任务数量。

AI 根据用户执行习惯推荐最佳学习时间。

AI 自动生成每日复盘。

AI 自动分析拖延原因。

这些功能属于 V2、V3，不属于 V1.0。

## 8.10 本章总结

AI Todo 的核心竞争力不是使用 AI，而是定义 AI 的行为。

AI 在本产品中的定位始终只有一个：

**帮助用户把目标变成今天可以执行的行动。**

所有 Prompt、模型和技术实现，都应围绕这一目标展开。

## 第九部分 AI 决策策略

### 9.1 章节目标

AI Todo V1.0 的核心能力是根据用户输入目标生成任务列表。

但在真实使用过程中，用户输入的目标并不总是清晰、完整、合理。用户可能输入非常模糊的内容，也可能输入过大的目标、过短的内容、无意义内容，甚至输入与任务管理无关的内容。

因此，本章用于定义 AI 在不同输入场景下的处理策略，确保产品输出稳定、可控、符合用户预期。

本章重点回答以下问题：

什么情况下 AI 可以直接生成任务？

什么情况下需要提示用户输入更具体的目标？

什么情况下需要拒绝或停止生成？

AI 应该生成多少条任务？

AI 如何控制任务难度？

AI 如何判断任务是否适合今天执行？

### 9.2 AI 决策总原则

AI Todo V1.0 的 AI 决策应遵循一个核心原则：

能拆就拆，不能拆就引导用户补充，不做无关回答。

具体来说：

第一，如果用户输入的是明确目标，AI 应直接生成任务列表。

第二，如果用户输入过于模糊，但仍能判断大概方向，AI 应生成轻量任务，并尽量降低行动门槛。

第三，如果用户输入完全无意义，系统应提示用户重新输入目标。

第四，如果用户输入与任务拆解无关，AI 不应进入闲聊模式，而应引导用户输入目标。

第五，如果用户输入存在明显风险或不适合提供行动建议，系统应避免生成具体执行任务，并给出安全提示。

### 9.3 输入类型判断

AI 或系统需要将用户输入大致分为以下几类：

| **输入类型** | **示例** | **处理方式** |
| --- | --- | --- |
| 明确目标 | 我要学习 Python | 直接生成任务 |
| 模糊目标 | 我想变好 | 生成轻量通用任务，或提示更具体 |
| 过大目标 | 我要成为程序员 | 拆成今天的第一步 |
| 过短输入 | 学习 | 提示用户补充具体目标 |
| 无意义输入 | 啊啊啊 | 提示重新输入 |
| 闲聊输入 | 你好，你是谁 | 引导输入目标 |
| 高风险输入 | 涉及违法、自伤、伤害他人等 | 不生成执行任务，给出安全提示 |

### 9.4 明确目标处理策略

#### 判断标准

如果用户输入中包含明确动作和目标对象，系统可以认为这是明确目标。

例如：

我要学习 Python

我要准备数据分析面试

我要做一个 AI 待办清单项目

我要复习英语四级

#### 处理方式

AI 应直接生成 3 到 8 条任务。

任务需要具体、轻量、当天可以执行。

#### 示例

用户输入：

我要准备数据分析面试

AI 应返回类似：

{

"tasks": [

{

"title": "阅读 3 个数据分析岗位 JD"

},

{

"title": "整理岗位高频技能要求"

},

{

"title": "准备 1 分钟自我介绍"

},

{

"title": "复盘一个项目经历"

},

{

"title": "模拟回答 3 个常见问题"

}

]

}

### 9.5 模糊目标处理策略

#### 判断标准

如果用户输入目标过于宽泛，但仍能判断大概方向，可以认为是模糊目标。

例如：

我想变好

我要自律

我要提升自己

#### 处理方式

V1.0 可以采用两种策略。

#### 策略一：直接生成轻量通用任务

适合 V1.0，因为可以减少用户操作步骤。

例如用户输入：

我想变好

AI 可以生成：

{

"tasks": [

{

"title": "写下今天最想改善的一件事"

},

{

"title": "选择一个 30 分钟内能完成的小任务"

},

{

"title": "关闭干扰软件 30 分钟"

},

{

"title": "完成任务后记录一句感受"

}

]

}

#### 策略二：提示用户补充目标

适合后续版本。

例如提示：

你的目标有点宽泛，可以补充一下方向，例如学习、求职、健身或项目。

#### V1.0 推荐策略

V1.0 推荐优先采用策略一。

原因是第一版产品应降低用户使用门槛，不应频繁打断用户。

如果目标模糊但仍可拆解，AI 应尽量给出低压力、通用型行动任务。

### 9.6 过大目标处理策略

#### 判断标准

如果用户输入的是长期目标，短时间内无法完成，可以认为是过大目标。

例如：

我要成为程序员

我要考上研究生

我要一年瘦 30 斤

我要精通英语

#### 处理方式

AI 不应试图一次性规划完整长期路线。

AI 应将大目标压缩成“今天可以开始的第一步”。

#### 示例

用户输入：

我要成为程序员

AI 应返回：

{

"tasks": [

{

"title": "确定一个入门方向：前端或 Python"

},

{

"title": "安装对应开发工具"

},

{

"title": "观看 20 分钟入门教程"

},

{

"title": "跟着教程写一个最小示例"

},

{

"title": "记录今天不懂的 3 个问题"

}

]

}

#### 规则说明

AI 可以承认目标较大，但不应输出长篇解释。

AI 的重点是帮助用户开始行动。

### 9.7 过短输入处理策略

#### 判断标准

如果用户输入只有 1 到 2 个字，且无法判断具体目标，应认为输入过短。

例如：

学习

面试

减肥

#### 处理方式

系统可以提示用户输入更具体的目标。

提示文案建议：

请输入更具体的目标，例如：我要学习 Python。

#### 补充说明

对于“减肥”“面试”这类虽然很短但仍有明确方向的输入，系统也可以选择直接生成轻量任务。

V1.0 推荐规则为：

只有 1 个字或明显无意义：提示补充。

2 到 4 个字但能判断方向：允许生成任务。

超过 5 个字且表达清楚：直接生成任务。

### 9.8 无意义输入处理策略

#### 判断标准

如果用户输入无法识别目标，也无法判断行动方向，应视为无意义输入。

例如：

啊啊啊

11111

？？？

随便

#### 处理方式

系统不应调用 AI 生成任务。

应提示用户重新输入目标。

提示文案建议：

没有识别到明确目标，请输入类似“我要学习 Python”的内容。

#### 验收标准

无意义输入不会触发 AI 生成。

页面给出明确提示。

用户原输入可以保留，方便修改。

### 9.9 闲聊输入处理策略

#### 判断标准

如果用户输入内容明显是闲聊，而不是目标，应视为闲聊输入。

例如：

你好

你是谁

今天开心吗

#### 处理方式

AI Todo 不是聊天机器人，因此不进入闲聊模式。

系统应引导用户输入目标。

提示文案建议：

我是用来帮你拆解任务的。请输入一个目标，例如：我要准备面试。

#### 设计原因

V1.0 应避免产品定位变成通用聊天工具。

如果用户想聊天，可以使用其他 AI 工具。本产品只聚焦任务拆解。

### 9.10 高风险输入处理策略

#### 判断标准

如果用户输入涉及违法行为、自伤、伤害他人、危险行为等内容，应视为高风险输入。

例如：

我要报复别人

我要做违法的事

我想伤害自己

#### 处理方式

系统不应生成具体执行任务。

应给出安全提示，并鼓励用户寻求合适帮助。

提示文案应简短、明确，不提供可执行步骤。

#### 示例提示

这个目标可能会带来伤害或风险，我不能帮你拆解执行步骤。你可以换一个安全、积极的目标，例如“我想冷静下来”或“我想处理情绪”。

#### 补充说明

如果用户表达自伤倾向，产品应优先建议用户联系身边可信任的人或当地紧急帮助渠道。

V1.0 不做复杂心理干预功能，但必须避免生成危险行动建议。

### 9.11 任务数量决策策略

AI 生成任务数量不应固定死，而应根据目标复杂度调整。

#### 推荐规则

| **目标复杂度** | **示例** | **推荐任务数量** |
| --- | --- | --- |
| 简单目标 | 我要整理房间 | 3 到 4 条 |
| 中等目标 | 我要学习 Python | 4 到 6 条 |
| 复杂目标 | 我要做一个小项目 | 5 到 8 条 |
| 模糊目标 | 我想变好 | 3 到 5 条 |
| 过大目标 | 我要成为程序员 | 4 到 6 条，聚焦第一步 |

#### 决策原则

任务数量不是越多越好。

V1.0 应优先让用户感觉可以完成，而不是让用户感觉压力很大。

### 9.12 任务难度决策策略

AI 生成任务时，应按照从简单到稍难的顺序排列。

推荐顺序为：

准备动作

↓

理解动作

↓

执行动作

↓

练习动作

↓

总结动作

例如用户输入：

我要学习 Python

较好的顺序是：

安装 Python 开发环境

↓

学习变量和数据类型

↓

跟着教程写一个 print 示例

↓

完成 2 道变量练习题

↓

记录今天遇到的问题

不要一开始就生成：

开发一个完整项目

这样会增加小白压力。

### 9.13 时间范围决策策略

V1.0 默认生成“今天可执行”的任务。

即使用户输入长期目标，AI 也应拆成今天可以开始的行动。

#### 默认时间范围

今天

#### 不推荐

不推荐生成：

未来一年完整计划

三个月学习路线图

除非后续版本专门支持长期目标规划。

#### 推荐表达

今天先完成以下任务

虽然 V1.0 的 JSON 输出不需要返回这句话，但生成逻辑应遵循这一原则。

### 9.14 个性化决策策略

V1.0 暂不支持复杂个性化，因为第一版没有登录系统和长期历史数据。

但仍可以做基础个性化：

#### 基于用户输入个性化

如果用户输入包含时间信息，例如：

我只有 1 小时学习 Python

AI 应生成适合 1 小时完成的任务。

如果用户输入包含身份信息，例如：

我是小白，想学习 Python

AI 应生成更基础的任务。

如果用户输入包含场景信息，例如：

我明天有面试

AI 应生成更紧急、更聚焦的任务。

#### V1.0 个性化规则

AI 只根据当前输入内容做轻量个性化。

不读取长期用户画像。

不基于未明确提供的信息做复杂推断。

### 9.15 任务生成失败时的兜底策略

如果 AI 接口失败，系统应提供明确提示，而不是页面无反应。

推荐提示：

任务生成失败，请稍后重试。

如果 AI 返回格式异常，系统可以提示：

任务解析失败，请重新生成。

如果后续开发允许，也可以设计一个备用方案：

当 AI 失败时，根据常见目标生成固定模板任务。

例如：

学习类目标

求职类目标

项目类目标

健身类目标

但该能力属于增强功能，不属于 V1.0 必做。

### 9.16 决策策略总结

AI Todo V1.0 的 AI 决策策略可以总结为：

明确目标：直接拆解

模糊目标：轻量拆解

过大目标：拆成今天第一步

过短输入：提示补充

无意义输入：拒绝生成

闲聊输入：引导输入目标

高风险输入：不生成执行步骤

AI 的目标不是展示聪明，而是帮助用户行动。

所有决策都应围绕一个标准：

用户看完后，是否知道现在该做什么？

## 第十部分 AI 输出质量评估标准

### 10.1 章节目标

AI Todo V1.0 的核心价值来自 AI 生成的任务列表。

如果 AI 返回的任务过于空泛、重复、过多、过难，用户即使看到任务列表，也不会真正执行。因此，产品需要定义一套 AI 输出质量评估标准，用于判断 AI 返回内容是否符合产品要求。

本章用于规定 AI 输出内容的质量标准、合格条件、不合格示例和检查规则。

本章的目标不是让 AI 返回更长内容，而是让 AI 返回更适合用户行动的任务。

### 10.2 AI 输出质量总原则

AI Todo V1.0 的 AI 输出质量总原则是：

具体、轻量、可执行、低压力。

也就是说，AI 生成的任务必须让用户看完以后知道：

我现在可以马上做什么。

如果用户看完任务后仍然不知道怎么开始，说明 AI 输出不合格。

### 10.3 合格输出的基本标准

一组合格的 AI 任务结果，需要满足以下标准：

第一，任务与用户目标相关。

AI 生成的任务必须围绕用户输入目标，不应生成无关内容。

第二，任务足够具体。

任务不能停留在“努力”“坚持”“学习”“提升”等抽象词，而应该是具体动作。

第三，任务可以执行。

用户看完任务后，应能立即开始行动。

第四，任务数量适中。

V1.0 建议生成 3 到 8 条任务，避免过少或过多。

第五，任务难度合理。

任务应从简单到稍难排列，不应第一步就过难。

第六，任务适合当天完成。

V1.0 默认生成今日任务，不应生成长期计划。

第七，任务之间不重复。

每条任务都应承担不同作用，避免换句话重复表达同一件事。

第八，输出格式稳定。

AI 应按照系统要求返回结构化数据，方便前端解析和展示。

### 10.4 单条任务质量标准

每一条任务都应符合以下要求：

| **维度** | **合格标准** | **不合格表现** |
| --- | --- | --- |
| 相关性 | 与用户目标直接相关 | 与目标无关 |
| 具体性 | 有明确动作 | 表达空泛 |
| 可执行性 | 用户能马上开始 | 不知道怎么做 |
| 长度 | 10 到 30 个字左右 | 过短或过长 |
| 难度 | 适合小白或当前目标 | 难度过高 |
| 表达 | 尽量动词开头 | 名词堆砌 |
| 独立性 | 单条任务可单独理解 | 依赖不清 |

### 10.5 好任务示例

用户输入：

我要学习 Python

合格输出：

{

"tasks": [

{

"title": "安装 Python 开发环境"

},

{

"title": "学习变量和数据类型"

},

{

"title": "跟着教程写一个 print 示例"

},

{

"title": "完成 2 道变量练习题"

},

{

"title": "记录今天遇到的 1 个问题"

}

]

}

这组任务合格的原因是：

围绕 Python 学习目标

从安装环境开始，符合小白路径

每条任务都能执行

数量适中

难度逐步增加

今天可以完成

### 10.6 不合格任务示例

用户输入：

我要学习 Python

不合格输出：

{

"tasks": [

{

"title": "努力学习 Python"

},

{

"title": "全面掌握 Python"

},

{

"title": "提升编程能力"

},

{

"title": "坚持下去"

}

]

}

不合格原因：

“努力学习”没有具体动作

“全面掌握”目标过大

“提升编程能力”过于抽象

“坚持下去”属于空泛鼓励

用户看完仍然不知道马上该做什么

### 10.7 任务数量评估标准

AI 返回任务数量应根据目标复杂度调整。

#### 合格范围

V1.0 推荐任务数量为：

3 到 8 条

#### 不合格情况

少于 3 条：

可能导致用户感觉 AI 帮助不足。

超过 8 条：

可能导致用户产生压力，不利于行动。

#### 特殊说明

简单目标可以生成 3 到 4 条。

复杂目标可以生成 5 到 8 条。

但 V1.0 不建议超过 8 条。

### 10.8 任务难度评估标准

任务难度应遵循：

准备动作 → 基础理解 → 简单执行 → 练习巩固 → 总结反馈

例如用户输入：

我要准备面试

合格顺序：

阅读目标岗位 JD

整理岗位高频要求

准备 1 分钟自我介绍

复盘一个项目经历

模拟回答 3 个常见问题

不合格顺序：

直接参加模拟终面

背完所有面试题

一天投递 100 个岗位

完整掌握行业知识

不合格原因：

任务压力过大

难度不符合小白用户

缺少从易到难的过程

不适合当天完成

### 10.9 可执行性评估标准

判断任务是否可执行，可以看用户是否能回答三个问题：

我要做什么？

怎么开始？

今天能不能完成？

如果这三个问题都能回答，说明任务可执行。

例如：

合格任务：

阅读 3 个数据分析岗位 JD

用户知道要做什么：阅读岗位 JD。

用户知道怎么开始：打开招聘软件或岗位页面。

今天能不能完成：可以。

不合格任务：

提升求职竞争力

用户不知道具体做什么，也不知道怎么开始。

### 10.10 相关性评估标准

AI 生成任务必须围绕用户输入目标。

例如用户输入：

我要准备英语四级

合格任务：

背诵 30 个四级高频词

完成一篇四级阅读练习

听 15 分钟四级听力材料

整理 5 个错题原因

不合格任务：

学习 Python

整理房间

投递简历

规划健身计划

不合格原因：

与英语四级目标无关

用户会认为 AI 不理解自己

影响产品可信度

### 10.11 重复性评估标准

任务之间不能只是换一种说法表达同一件事。

例如用户输入：

我要减肥

不合格输出：

开始运动

进行锻炼

做一些训练

增加身体活动

这些任务本质重复，用户看不到清晰步骤。

合格输出：

步行 8000 步

喝 2000ml 水

减少一杯含糖饮料

完成 20 分钟力量训练

记录今天饮食情况

每条任务承担不同作用：

活动量

饮水

饮食控制

训练

记录反馈

### 10.12 输出格式评估标准

V1.0 推荐 AI 输出 JSON 格式，方便系统解析。

合格格式：

{

"tasks": [

{

"title": "安装 Python 开发环境"

},

{

"title": "学习变量和数据类型"

}

]

}

不合格格式：

当然可以！下面是你的任务：

1. 安装 Python

2. 学习变量

加油，你一定可以！

不合格原因：

包含多余自然语言

不方便程序稳定解析

容易导致前端展示异常

### 10.13 AI 输出自动检查规则

系统或开发实现中，可以对 AI 输出进行基础检查。

建议检查规则如下：

第一，检查是否包含 tasks 字段。

第二，检查 tasks 是否为数组。

第三，检查任务数量是否在 3 到 8 条之间。

第四，检查每条任务是否包含 title 字段。

第五，检查 title 是否为空。

第六，检查 title 是否过长。

第七，检查是否存在明显重复任务。

第八，检查是否包含多余解释内容。

如果检查失败，系统可以提示：

任务解析失败，请重新生成。

或者尝试重新调用 AI。

### 10.14 人工验收检查表

项目开发和测试阶段，可以使用以下清单人工检查 AI 输出质量。

| **检查项** | **是否通过** |
| --- | --- |
| 是否围绕用户目标生成？ | 是 / 否 |
| 是否生成 3 到 8 条任务？ | 是 / 否 |
| 每条任务是否具体？ | 是 / 否 |
| 每条任务是否可执行？ | 是 / 否 |
| 是否适合今天完成？ | 是 / 否 |
| 是否避免空泛鸡汤？ | 是 / 否 |
| 是否避免重复任务？ | 是 / 否 |
| 是否从简单到稍难排列？ | 是 / 否 |
| 是否符合 JSON 格式？ | 是 / 否 |
| 用户看完是否知道下一步？ | 是 / 否 |

如果任意关键项不通过，应调整 Prompt 或输出解析规则。

### 10.15 典型测试用例

为了验证 AI 输出质量，V1.0 至少需要测试以下输入：

#### 学习类

我要学习 Python

期望：

生成环境安装、基础概念、练习、记录问题等任务。

#### 求职类

生成阅读 JD、整理技能、准备自我介绍、复盘项目、模拟问题等任务。

#### 项目类

我要做一个 AI 待办清单项目

期望：

生成需求整理、页面草图、项目初始化、核心功能开发等任务。

#### 健身类

期望：

生成饮水、步行、饮食控制、训练、记录等任务。

#### 模糊类

生成轻量通用任务，例如确定一个小目标、完成一个 30 分钟任务、记录反馈。

#### 无效类

啊啊啊

期望：

不生成任务，提示用户输入明确目标。

### 10.16 AI 输出质量分级

为了后续优化，可以将 AI 输出质量分为三级。

#### A 级：优秀

特点：

任务非常具体

顺序合理

难度适中

与目标高度相关

用户看完可以立即行动

#### B 级：可用

特点：

任务基本相关

大部分任务可执行

少量任务略泛

不影响主流程使用

#### C 级：不合格

特点：

任务空泛

任务重复

与目标无关

任务过多或过难

格式无法解析

V1.0 应至少保证大多数常见输入达到 B 级，核心场景尽量达到 A 级。

### 10.17 输出质量优化方向

如果后续发现 AI 输出质量不稳定，可以从以下方向优化：

第一，优化系统 Prompt。

明确 AI 角色、输出格式、任务数量和禁止行为。

第二，增加示例。

在 Prompt 中提供好的任务示例和不好的任务示例。

第三，增加结构化输出约束。

要求 AI 严格返回 JSON，减少解析失败。

第四，增加结果校验。

系统对 AI 返回内容进行格式和质量检查。

第五，增加二次生成机制。

如果第一次输出不合格，可以自动重新请求 AI 生成。

### 10.18 本章总结

AI Todo V1.0 的 AI 输出质量，不以内容多为标准，而以是否能帮助用户行动为标准。

好的 AI 输出应该满足：

目标相关

任务具体

数量适中

难度合理

今天可做

格式稳定

最终判断标准只有一句话：

用户看完以后，是否知道自己现在应该做什么？

## 第十一部分 非功能需求与边界约束

### 11.1 章节目标

AI Todo V1.0 除了需要完成核心功能外，还需要满足基础的非功能需求。

非功能需求不是直接面向某一个按钮或页面功能，而是规定产品在性能、稳定性、安全性、兼容性、隐私保护和开发边界方面应满足的要求。

本章用于明确：

产品运行速度要求

页面兼容性要求

数据保存与隐私要求

AI 调用稳定性要求

异常情况处理要求

V1.0 不做什么

后续版本扩展边界

### 11.2 性能要求

AI Todo V1.0 应保证基础页面访问流畅，用户操作后能获得及时反馈。

#### 页面加载要求

首页应尽量保持轻量。

用户打开页面后，应快速看到产品名称、输入框和按钮。

建议目标：

首页主要内容在 3 秒内可见。

V1.0 不应引入过多复杂动画、大型图片或无关资源，避免影响加载速度。

#### AI 生成响应要求

用户点击“AI 拆分任务”按钮后，系统应立即进入加载状态。

即使 AI 生成需要几秒钟，用户也应该知道系统正在处理。

建议目标：

点击按钮后 0.5 秒内展示加载状态。

AI 返回时间受模型接口、网络情况影响，V1.0 不强制要求固定秒数，但需要保证用户等待期间页面有明确反馈。

#### 交互响应要求

用户点击任务勾选框后，任务状态应立即变化。

勾选操作不应明显卡顿。

建议目标：

任务勾选状态应在用户点击后立即更新。

### 11.3 稳定性要求

V1.0 应保证核心流程稳定运行。

核心流程包括：

输入目标 → AI 生成任务 → 展示任务 → 勾选完成 → 保存状态 → 刷新恢复

#### 页面稳定性

页面不应因为以下情况崩溃：

用户输入为空

用户输入过长

AI 调用失败

AI 返回格式异常

localStorage 数据异常

网络请求失败

用户连续点击按钮

#### 数据稳定性

任务生成后，用户已经看到的任务不应因为轻微异常被清空。

例如：

保存失败时，不应立即清空页面任务

AI 生成失败时，不应清空输入框

读取本地数据失败时，应回到空状态，而不是报错崩溃

#### 错误提示要求

系统出现错误时，应提供用户能理解的提示。

错误提示应避免技术化表达。

不推荐：

Error: JSON parse failed

推荐：

任务解析失败，请重新生成。

### 11.4 兼容性要求

AI Todo V1.0 应支持主流浏览器访问。

#### 电脑端

需要支持：

Chrome

Edge

Safari

Firefox

V1.0 不要求兼容非常老旧的浏览器版本。

#### 手机端

需要支持手机浏览器访问。

页面应适配常见手机屏幕。

手机端重点要求：

输入框不超出屏幕

按钮容易点击

任务文字不横向溢出

页面不出现明显错位

勾选框可以正常点击

#### 响应式要求

V1.0 不需要复杂移动端设计，但必须保证基础可用。

推荐布局：

电脑端：

居中卡片布局

手机端：

上下排列布局

### 11.5 数据保存要求

V1.0 推荐使用 localStorage 保存当前任务组。

#### 保存要求

系统需要保存：

用户输入的目标

AI 生成的任务列表

每条任务完成状态

创建时间

更新时间

#### 恢复要求

用户刷新页面后，应恢复上一次任务数据。

如果没有保存数据，则展示空状态。

如果本地保存数据损坏，系统应忽略异常数据，并展示空状态。

#### 数据丢失说明

由于 V1.0 使用本地存储，需要明确以下限制：

更换设备后无法看到原任务

更换浏览器后无法看到原任务

清除浏览器缓存可能导致任务丢失

无法进行云端同步

无法多人共享数据

这些限制属于 V1.0 的产品边界，不属于缺陷。

### 11.6 隐私要求

V1.0 不设计用户账号系统，因此不应主动收集用户身份信息。

#### 不应要求用户输入的信息

V1.0 不应要求用户输入：

身份证号

银行卡号

账号密码

家庭住址

手机验证码

私密文件内容

其他敏感个人信息

#### 用户输入目标的隐私说明

用户输入的目标可能包含个人计划，例如学习、求职、健身等内容。

如果任务拆解需要调用 AI 接口，则用户输入内容会被发送给 AI 服务进行处理。

因此，产品文案或后续设置页中应提醒用户：

请不要输入密码、证件号等敏感信息。

V1.0 可以先在输入框附近用轻量提示表达：

请勿输入密码、证件号等敏感信息。

#### 本地数据说明

如果后续版本加入云端保存和登录系统，需要单独补充隐私政策和数据安全说明。

### 11.7 安全要求

AI Todo V1.0 虽然是练手项目，但仍需要考虑基础安全问题。

#### API Key 安全

如果项目接入 AI API，API Key 不能暴露在前端代码中。

不允许：

在浏览器端直接写入 AI API Key。

推荐：

通过后端 API 路由调用 AI 服务。

这样可以避免用户在浏览器开发者工具中看到密钥。

#### 输入安全

用户输入内容需要做基础长度限制。

建议限制：

5 到 100 个字符

避免用户输入过长内容导致请求异常或成本失控。

#### 请求频率控制

V1.0 可以先做基础防重复点击。

当 AI 正在生成任务时，按钮应进入禁用状态，避免用户连续点击造成重复请求。

后续版本可以增加更完整的频率限制。

#### 高风险输入

如果用户输入涉及违法、自伤、伤害他人等内容，系统不应生成具体执行任务。

应提示用户换一个安全、积极的目标。

### 11.8 AI 成本控制要求

AI Todo 调用大模型会产生一定成本，因此 V1.0 需要避免无意义调用。

#### 避免无效请求

以下情况不应调用 AI：

输入为空

输入明显无意义

输入过短且无法判断目标

用户连续重复点击按钮

页面正在生成任务

#### 控制输入长度

限制用户输入长度，可以减少无意义消耗。

#### 控制输出长度

AI 输出只需要返回 3 到 8 条任务。

不应生成长篇解释。

推荐输出格式：

{

"tasks": [

{

"title": "安装 Python 开发环境"

}

]

}

#### 后续成本优化方向

后续版本可以增加：

每日生成次数限制

用户登录后按账号限制

常见目标模板缓存

AI 失败时使用本地模板兜底

这些能力不属于 V1.0 必做。

### 11.12 风险说明

V1.0 可能存在以下风险。

#### 风险一：AI 生成质量不稳定

不同输入可能导致 AI 输出质量不同。

应通过 Prompt 约束、格式校验和测试用例降低风险。

#### 风险二：用户输入过于模糊

用户输入太宽泛时，AI 可能生成通用任务。

V1.0 可以先接受该情况，后续通过追问机制优化。

#### 风险三：本地数据可能丢失

localStorage 受浏览器环境影响，用户清缓存后数据会消失。

V1.0 接受该限制，后续通过数据库解决。

#### 风险四：AI API 调用失败

网络、接口额度、服务异常都可能导致生成失败。

系统需要提供错误提示和重试入口。

#### 风险五：开发范围膨胀

如果过早加入登录、历史记录、提醒、日历等功能，项目容易变复杂。

V1.0 必须坚持 MVP 范围。

AI Todo V1.0 的非功能需求可以总结为：

页面要快

流程要稳

数据不乱

隐私要提醒

密钥不能泄露

异常不能崩溃

范围不能扩大

V1.0 的第一目标不是做一个大而全的产品，而是用安全、稳定、简单的方式跑通核心闭环：

输入目标 → AI 拆解 → 展示任务 → 勾选完成 → 刷新恢复

### 12.1 章节目标

AI Todo V1.0 需要采用分阶段开发方式，避免一次性开发过多功能导致项目失控。

### 14.2 项目执行原则

AI Todo V1.0 开发过程中，必须遵守以下原则。

#### 原则一：先完成 MVP，再考虑扩展

V1.0 只做最小可用版本。

核心闭环为：

输入目标 → AI 拆解任务 → 展示任务列表 → 勾选完成 → 本地保存 → 刷新恢复

任何不服务于该闭环的功能，默认不做。

#### 原则二：每次只开发一个阶段

Codex 每次只能执行一个 Phase。

不得一次性完成多个阶段。

原因是：

小白项目需要过程可控

方便阶段验收

方便及时发现问题

避免一次性生成大量不可维护代码

#### 原则三：不允许自由发挥新增功能

Codex 不得主动新增 PRD 中没有要求的功能。

例如，V1.0 不允许擅自加入：

登录注册

用户中心

数据库

会员系统

日历提醒

历史记录

复杂多页面

聊天机器人功能

如果确实认为需要新增功能，必须先提出建议，由项目负责人确认后才能执行。

#### 原则四：先假数据，后接 AI

开发顺序必须遵守：

先用假数据跑通页面和任务列表

↓

再接入真实 AI 接口

不得一开始就直接接 AI。

这样可以降低调试难度，避免小白同时面对 UI、状态、接口、密钥、网络等多个问题。

#### 原则五：API Key 不得暴露

如果项目接入 AI API，API Key 必须放在环境变量中。

不得出现在：

前端组件代码

浏览器请求参数

GitHub 仓库

页面源码

控制台日志

前端只能调用项目自己的后端 API 路由，由后端 API 再调用 AI 服务。

#### 原则六：每个阶段必须验收

Codex 完成一个阶段后，不能直接进入下一阶段。

必须经过：

Codex 完成开发

↓

项目负责人本地运行

↓

按验收标准测试

↓

Claude Code Review

↓

Codex 修复问题

↓

再次验收

↓

进入下一阶段

如果以下任意 P0 问题存在，则不能视为完成：

页面打不开

输入框无法输入

AI 无法生成任务

任务列表无法展示

任务无法勾选

刷新后任务丢失

API Key 暴露在前端

页面核心流程频繁报错
