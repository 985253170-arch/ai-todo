## 一、背景：为什么需要 AI 辅助执行

### 1.1 当前已完成闭环

AI Todo 已经完成了从目标到复盘的完整闭环：

```
目标 → AI 拆解 → 执行 → 记录 → 统计 → 复盘 → 智能调整
```

| 能力 | 版本 | 状态 |
|------|------|:--:|
| AI 根据目标拆解为 3-8 条可执行任务 | V1.0 | ✅ |
| 任务勾选/完成/清空/云端同步 | V1.0 | ✅ |
| 历史记录（游标分页） | Phase 12 | ✅ |
| 统计（Today/7Day/Total） | Phase 13 | ✅ |
| AI 复盘（反馈 + 难度建议） | Phase 14 | ✅ |
| 智能任务调整（5 级阈值） | Phase 15 | ✅ |
| Email+Password + OTP 登录 | V2.1 + V2.1B | ✅ |
| 页面路由结构（`/` `/login` `/app`） | V2.2A | ✅ |

### 1.2 当前缺口

AI 目前主要帮用户做两件事：

1. **拆任务**：把模糊目标拆成可执行的子任务
2. **复盘**：任务完成后给出反馈和建议

但有一个明显的空白地带：**用户在任务执行过程中，AI 没有提供帮助。**

用户看到任务列表后，仍然可能遇到：

- 不知道怎么开始第一条任务
- 任务看起来还是太大，不知道从哪里下手
- 没有模板可以参考
- 不知道完成标准是什么
- 卡在某个任务上，想换一条路但不知道怎么换
- 想要一个可以直接复制使用的半成品
- 想把任务变成更具体的、能立刻执行的下一步
- 想把这个任务转成给别人看的指令

### 1.3 升级方向

当前产品核心是：**AI 帮你拆任务 → 你自己执行 → AI 帮你复盘**。

升级后产品核心是：**AI 帮你拆任务 → AI 陪你执行每一步 → AI 帮你复盘**。

也就是说：

```
从 "AI 拆任务 + 监督完成"
升级为 "AI 辅助用户推进每一个任务"
```

这是 AI Todo 从"智能任务列表"升级为"AI 行动教练"的关键一步。

## 二、核心产品定义

### 2.1 一句话产品定义

> **AI 辅助执行不是固定按钮系统，而是任务级受限 AI 工作台。用户可以围绕当前任务自由描述需求，AI 根据任务内容和用户意图，生成具体可执行的步骤、模板、初稿、指令、清单或下一步建议。**

### 2.2 功能名称

功能名称定为：**任务级受限 AI 助手**

以下命名**不使用**：

| 不使用 | 原因 |
|--------|------|
| 固定辅助菜单 | 限制了用户表达自由的系统设计方向 |
| 全局聊天机器人 | 没有任务边界，与 ChatGPT 无差异 |
| 万能 ChatGPT | 没有产品定位，无法差异化 |
| AI 代办系统 | 暗示 AI 替你完成任务，违背产品边界 |
| 自动执行 Agent | 暗示全自动，违背"不替用户完成"原则 |
| AI 教练 | 虽然接近产品定位，但过于宏大，不适合 MVP |

### 2.3 它是什么

任务级受限 AI 助手是：

- **绑定在某个 taskId 下的轻量 AI 辅助面板**
- **只围绕当前任务提供帮助**
- **允许用户自由输入需求**
- **AI 自动识别意图**
- **返回短小、具体、可执行、可复制的结果**
- **最终仍由用户执行和确认完成**

### 2.4 它不是什么

| 它不是 | 说明 |
|--------|------|
| 通用聊天机器人 | 不能聊任何话题，只围绕当前任务 |
| 全局 AI 聊天页 | 不是独立页面，是任务卡片内的面板 |
| 自动代办工具 | 不会自动完成、自动提交、自动操作外部系统 |
| 固定按钮集合 | 不是 7 个固定辅助按钮 |
| 项目管理系统 | 不做甘特图、不做依赖关系、不做团队协作 |

### 2.5 核心原则

> **AI 负责铺路，用户负责迈步。**

- **自由输入是核心**。快捷入口只是辅助。
- **AI 分类在后端**。用户不需要选择 assistType。
- **每次输出都要给可执行结果**。不能只给建议不给具体内容。
- **对话不是目的，推进任务是目的**。不保留长期聊天记录。
- **AI 不替用户完成任务**。AI 提供方法、步骤、模板、初稿、检查清单、下一步建议；用户必须亲自判断、修改、执行、提交、确认完成。

## 三、产品边界：受限任务对话

### 3.1 为什么不是固定按钮菜单

**固定选项无法覆盖真实任务。**

下面这些固定选项：

- 怎么开始
- 拆小一点
- 给我 5 分钟版本
- 生成模板 / 初稿
- 我卡住了
- 完成标准
- 下一步做什么

它们有用，但太抽象、太有限。只能覆盖常见动作模式，不能覆盖所有真实场景。

真实用户可能会问：

- "帮我写一段给老板请假的话"
- "帮我把这个任务变成 Codex 指令"
- "帮我根据我现在的电脑报错判断下一步"
- "帮我写一个小红书标题"
- "帮我把今天的健身任务改成膝盖不疼的版本"
- "帮我准备明天面试的数据分析自我介绍"
- "帮我把这个任务改成 10 分钟内能做完"
- "帮我先生成一个可以直接复制到微信的版本"
- "帮我判断这一步到底算不算完成"

**固定按钮永远追不上用户真实目标的变化。** 所以后续 AI 辅助执行不能设计成"固定辅助按钮集合"。

### 3.2 为什么不是普通聊天机器人

原因有五个：

1. **没有任务边界**：普通聊天机器人不知道当前 goal 是什么、当前 task 是什么、用户已经完成了哪些任务。它只能根据对话历史来猜。
2. **没有完成导向**：普通聊天机器人不关心用户是否完成了任务。它会一直聊下去，而不是帮用户推进到"完成"。
3. **没有产品身份**：如果 AI Todo 的 AI 辅助只是一个通用聊天窗口，那和打开 ChatGPT 有什么区别？
4. **上下文浪费**：任务数据（goal、task、taskGroupId、完成状态）是产品积累的宝贵上下文，不用就是浪费。
5. **用户认知混乱**：用户不知道什么时候该用 AI 辅助、什么时候该用 ChatGPT。

### 3.3 受限任务对话 vs 无限聊天

| 不应该做 | 应该做 |
|----------|--------|
| 无限聊天机器人 | 受限任务对话 |
| 用户随便聊任何话题 | 只围绕当前任务 |
| AI 无限接话 | 目标是推进任务完成 |
| 没有任务边界 | 必须绑定 taskId |
| 没有完成导向 | 每次输出都要给可执行结果 |
| 自动帮用户完成外部操作 | 用户保留最终执行权 |
| 保留长期聊天记录 | 临时辅助空间，任务完成后可抛弃 |
| 用户可以无限追问发散 | 允许追问 1-2 轮，但始终围绕原任务 |
| 情绪陪聊 | 不接情绪、不做闲聊、不替代通用搜索 |
| 开放式回答 | 输出短小、具体、可执行、可复制 |

### 3.4 任务约束上下文

AI 辅助面板的每次请求，必须携带以下上下文：

| 上下文项 | 说明 |
|----------|------|
| `goal` | 当前目标是什么 |
| `task.title` | 当前任务标题 |
| `task.id` | 当前任务 ID |
| `taskGroupId` | 当前任务组 ID |
| 已完成任务列表 | 用户已经完成了哪些任务 |
| 当前任务完成状态 | 已完成 / 未完成 / 进行中 |
| `userRequest` | 用户现在请求什么帮助 |

这些上下文不是限制 AI 的能力，而是**聚焦 AI 的能力**。让 AI 在一个明确的任务范围内给出最精准的帮助。

### 3.5 最终边界

> **AI Todo 不替用户完成任务，而是帮助用户完成任务。**
>
> AI 可以提供方法、步骤、模板、初稿、检查清单、下一步建议；
> 但用户必须亲自判断、修改、执行、提交、确认完成。

具体来说：

| AI 可以做的 | AI 不能做的 |
|------------|------------|
| 写一段请假文案 | 帮你发邮件给老板 |
| 生成 Codex 修复指令 | 帮你提交代码 |
| 列出排查错误的步骤 | 帮你执行命令行 |
| 写小红书标题 | 帮你发布到小红书 |
| 提供健身动作替代方案 | 帮你做运动 |
| 写面试自我介绍 | 帮你参加面试 |
| 生成可复制到微信的版本 | 帮你发微信 |
| 判断任务是否达到完成标准 | 帮你勾选完成 |

## 四、Human-in-the-loop 完成边界：人在回路中的完成边界

### 4.1 为什么需要单独定义完成边界

AI 辅助执行的核心张力在于：AI 能力强了，用户就容易产生"让 AI 帮我全做了"的冲动；但产品边界决定了 AI 不能替用户完成最终动作。

因此，必须清晰定义：在任务推进的每个阶段，AI 做什么、用户做什么、谁对最终结果负责。

### 4.2 AI 可以与不应该

| AI 可以 | AI 不应该 |
|---------|----------|
| 辅助开始：给第一步具体行动 | 自动替用户投简历 |
| 生成步骤：把任务拆成可执行的小步骤 | 自动替用户发消息 |
| 生成模板：提供可直接修改的半成品 | 自动提交作业 |
| 生成初稿：写一段可以改的文本 | 自动勾选完成 |
| 生成检查清单：列出需要核对的事项 | 自动伪造经历 |
| 生成示例话术：写一段可参考的对话 | 自动代替用户做需要个人判断的决定 |
| 生成 Claude / Codex 指令：描述修复方案 | 自动完成第三方平台操作 |
| 生成面试回答框架：给结构和要点 | 自动修改任务完成状态 |
| 给完成标准：定义"做到什么程度算完成" | — |
| 给下一步建议：完成后接下来做什么 | — |

### 4.3 人机协作分工表

在任务推进的 7 个关键阶段，AI 和用户的角色边界如下：

| 阶段 | AI 角色 | 用户角色 |
|------|---------|---------|
| 想不清楚 | 帮你澄清：列出可能的理解 | 选择目标：确定自己真正想做的是什么 |
| 不会开始 | 给第一步：具体、可执行、不需要前置准备 | 执行第一步：亲自迈出第一步 |
| 不会写 | 给模板/初稿：可修改的半成品 | 修改成自己的：用自己的语言和情况调整 |
| 卡住了 | 给替代路径：列出其他可行方法 | 选择继续方式：决定走哪条路 |
| 不知道标准 | 给完成标准：清单或一句话标准 | 自己判断是否完成：对照标准自我评估 |
| 准备提交 | 给检查清单：最后要核对的事项 | 用户确认提交：亲自检查并提交 |
| 完成任务 | 记录和复盘：记录完成情况 + 给出复盘反馈 | 用户勾选完成：亲自勾选，确认结束 |

### 4.4 最终裁决原则

> **AI 辅助用户完成任务，但不替用户完成任务。**
>
> 最终判断、修改、执行、提交、确认完成，都必须由用户完成。

这不是因为 AI 做不到——而是因为产品定位决定了这条线必须清晰。一旦 AI 开始替用户做最终决定和最终操作，产品就从"行动教练"变成了"自动执行系统"。用户不再对自己的任务有掌控感，也不再需要为自己的完成负责。

**AI 负责铺路，用户负责迈步。路的尽头是什么，由用户自己决定。**

## 五、交互结构设计

### 5.1 三层交互结构总览

```
┌─────────────────────────────────────────┐
│           第一层：任务卡片入口             │
│   每个 TaskItem 右侧只有一个入口           │
│   "AI 帮助" 或 "帮我完成"                 │
├─────────────────────────────────────────┤
│           第二层：任务辅助面板             │
│   点击入口后展开的轻量面板                 │
│   包含：任务信息 + 自由输入 + 快捷入口      │
│        + AI 输出区 + 复制/重新生成         │
├─────────────────────────────────────────┤
│           第三层：动态建议                 │
│   2-3 个通用快捷入口（永远可用）            │
│   + 2-3 个 AI 动态建议（根据任务生成）      │
└─────────────────────────────────────────┘
```

### 5.2 第一层：任务卡片入口

每个任务卡片（TaskItem）只放**一个**主入口：

- "AI 帮助"
- 或 "帮我完成"

**不在任务卡片上堆 7 个固定按钮。** 原因：

| 原因 | 说明 |
|------|------|
| 任务卡片会变复杂 | 每条任务右边跟 7 个按钮，信息密度爆炸 |
| 手机端会拥挤 | 320px 宽度的手机屏幕放不下 7 个按钮 + 勾选框 + 任务文字 |
| 用户目标不可穷举 | 不管放多少个按钮，永远有用户需求在按钮之外 |
| 固定按钮会误导产品方向 | 产品团队会不断想"要不要再加一个按钮？"而不是"怎么让用户自由表达" |
| 固定按钮会让用户以为只能按固定模式求助 | 用户看到 7 个按钮会觉得"我只能选这 7 种帮助"，而不是"我可以描述任何需求" |

**一个入口，点击后展开面板，让用户在面板内自由描述需求。** 这个设计决策是整个交互结构的基石。

### 5.3 第二层：任务辅助面板

点击"AI 帮助"后，在当前任务卡片下方展开一个轻量面板（不是全屏页面，不是 Modal 弹窗）。

面板必须包含以下元素：

```
┌──────────────────────────────────────────────┐
│  📋 任务：[任务标题]                           │  ← 当前任务标题
│  🎯 目标：[所属 Goal]                          │  ← 当前 goal
│  📌 状态：未完成                               │  ← 完成状态
│                                              │
│  ┌──────────────────────────────────────────┐│
│  │  你希望 AI 怎么帮你完成这一步？            ││  ← 自由输入框
│  │                                          ││
│  │  [输入你的需求...]                        ││
│  └──────────────────────────────────────────┘│
│                                              │
│  快捷入口：                                   │
│  [帮我开始]  [变简单一点]  [生成可复制内容]     │  ← 2-3 个通用快捷入口
│  [生成 Codex 指令]  [列出排查步骤]             │  ← 2-3 个动态建议（AI 生成）
│                                              │
│  ─────────────────────────────────────────── │
│                                              │
│  AI 输出：                                    │  ← AI 输出结果区域
│  ┌──────────────────────────────────────────┐│
│  │  [AI 生成的文本/清单/步骤/模板...]         ││
│  │                                          ││
│  │  [📋 复制]  [🔄 重新生成]                 ││
│  └──────────────────────────────────────────┘│
│                                              │
│  [继续追问...]  ← 可选，允许追问一次            │
└──────────────────────────────────────────────┘
```

**面板设计原则**：

| 原则 | 说明 |
|------|------|
| 不是全屏聊天页 | 面板紧凑、轻量，展开在当前任务卡片下方 |
| 不是长期聊天记录 | 每次打开面板是新会话，不保留历史（可选允许追 1-2 轮） |
| 围绕当前任务的临时辅助空间 | 聚焦当前任务，不跨任务关联 |
| 移动端优先 | 面板宽度自适应，输入框在键盘弹起后仍可见 |
| 可关闭 | 用户点击面板外部或关闭按钮即可关闭 |
| 不遮挡勾选框 | 面板展开后，勾选框和任务文字仍然可见 |

### 5.4 第三层：动态建议

快捷入口不是固定死的 7 个按钮，而是分两类。

#### 5.4.1 第一类：通用快捷入口（2-3 个，永远可用）

| 快捷入口 | 对应 system intent | 说明 |
|----------|-------------------|------|
| "帮我开始" | `first_step` | 告诉用户这条任务的第一步做什么 |
| "变简单一点" | `reduce_difficulty` | 降低任务的复杂度或拆成更小的步骤 |
| "生成可复制内容" | `generate_copy` | 生成可以直接复制到其他地方的文本 |

这 2-3 个通用入口覆盖最频繁的求助场景，降低用户的输入成本。但它们只是辅助——用户不需要从中选择，可以直接在输入框里写自己的需求。

#### 5.4.2 第二类：动态快捷入口（2-3 个，AI 根据当前任务生成）— V2.5

> **此功能属于 V2.5 范围，V2.4 MVP 只使用前端固定通用快捷入口。** 此处描述完整交互设计供 V2.5 架构方案引用。

每次展开面板时，AI 根据当前任务内容，动态生成 2-3 个快捷入口（通过 `POST /api/task-assist/suggest`，V2.5 新增）。这些入口**不需要用户手动选择触发**——用户点击后，相当于用自然语言发送了对应的请求。

**示例 1**：任务 = "准备面试自我介绍"

动态建议：
- "生成 30 秒自我介绍"
- "改成更适合运营岗"
- "列出面试官可能追问"

**示例 2**：任务 = "修复登录按钮报错"

动态建议：
- "生成给 Codex 的修复指令"
- "列出排查步骤"
- "帮我写测试清单"

**示例 3**：任务 = "整理房间"

动态建议：
- "给我 5 分钟收拾法"
- "先从桌面开始"
- "列一个最小清单"

**动态建议的生成时机**（V2.5）：

每次展开面板时，客户端请求 `POST /api/task-assist/suggest`（V2.5 新增 API），后端根据 task 内容返回 2-3 条动态建议。响应缓存到面板关闭为止。V2.4 MVP 不调用此 API，面板仅显示 3 个前端固定通用快捷入口。

**动态建议不是菜单，是降低输入成本的快捷方式。** 用户点击动态建议 = 自动填充输入框为该建议文本，用户可以修改后再提交，也可以直接忽略。

## 六、意图识别设计

### 6.1 用户不需要选择 assistType

前端不应该强迫用户选择：

- start
- unblock
- simplify
- template
- checklist

这些可以作为**系统内部 intent**，而不是**用户前台菜单**。

用户输入自然语言，后端识别意图：

| 用户输入 | 后端识别 intent | outputType |
|----------|----------------|------------|
| "帮我写一个能直接发给 HR 的版本" | `generate_copy` | `copyable_text` |
| "我现在完全不想做，能不能变简单一点" | `reduce_difficulty` | `mini_action` |
| "这一步做到什么程度算完成" | `done_criteria` | `checklist` |
| "我不知道从哪里开始" | `first_step` | `step` |
| "给我一个可以直接用的模板" | `draft_or_template` | `template` |
| "卡住了，有没有别的办法" | `unblock` | `alternative_path` |
| "下一步做什么" | `next_action` | `step` |
| "帮我生成给 Codex 的修复指令" | `codex_instruction` | `codex_prompt` |
| "列出我要检查的东西" | `checklist` | `checklist` |
| "能再详细一点吗" | `explain_task` | `explanation` |

### 6.2 首批内部 Intent 定义

| Intent | 含义 | 输出类型 |
|--------|------|---------|
| `first_step` | 给第一步 | 具体的一步行动 |
| `reduce_difficulty` | 降低难度 | 更简单的替代步骤 |
| `five_min_version` | 5 分钟版本 | 极简版快速行动 |
| `generate_copy` | 生成可复制文本 | 可复制的文本内容 |
| `draft_or_template` | 生成模板 / 初稿 | 模板或初稿文本 |
| `unblock` | 卡住时给替代路径 | 替代方法或路径 |
| `done_criteria` | 完成标准 | 检查清单 |
| `next_action` | 下一步建议 | 下一步行动 |
| `codex_instruction` | 生成给 Codex 的执行指令 | 类 Codex 格式的指令 |
| `checklist` | 生成检查清单 | 清单列表 |
| `explain_task` | 解释任务怎么做 | 方法说明 |

> **intent 列表是系统内部结构，随产品迭代可扩展。用户不需要知道 intent 的存在。**

### 6.3 意图识别的实现方式

**不是训练专门的意图分类模型。** MVP 阶段通过与任务上下文一起发送给 LLM，让 LLM 同时完成意图识别和内容生成：

```
System Prompt 包含：
1. 你是 AI Todo 的任务级受限 AI 助手
2. 你的核心目标是帮助用户推进当前任务
3. 当前任务上下文（goal / task / 完成状态）
4. 内部 intent 列表及说明（供 LLM 理解用户需求类型）
5. 输出格式要求：短小、具体、可执行、可复制
6. 边界规则：不替用户完成、不闲聊、不发散
```

LLM 根据用户输入和任务上下文，自然理解用户意图，生成对应内容。不需要两阶段调用（先识别 intent → 再生成内容），一次调用完成。

## 七、API 方向设计

### 7.1 核心 API：POST /api/task-assist

这是 AI 辅助执行的核心 API。

#### 请求结构

```json
{
  "taskGroupId": "uuid-of-task-group",
  "taskId": "uuid-of-task",
  "userRequest": "帮我写一个可以直接发给HR的版本",
  "conversationContext": [
    {
      "role": "user",
      "content": "上一轮用户输入（可选）"
    },
    {
      "role": "assistant",
      "content": "上一轮AI结果（可选）"
    }
  ]
}
```

**注意**：
- `userRequest` 是用户自由输入的自然语言，不是 assistType 枚举值
- `conversationContext` 可选，仅在用户追问时携带（最多 2 轮）
- 不传 `deviceId`——与现有 session-aware API 一致，userId 由服务端 Supabase Auth session 获取

#### 响应结构

```json
{
  "success": true,
  "data": {
    "intent": "generate_copy",
    "outputType": "copyable_text",
    "content": "尊敬的领导：\n\n因个人身体不适，需请假一天...",
    "quickActions": [
      { "label": "再短一点", "request": "帮我再缩短到两句话" },
      { "label": "改成邮件格式", "request": "帮我改成正式邮件格式" }
    ]
  }
}
```

**响应字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `intent` | string | 系统内部识别的意图类型 |
| `outputType` | string | 输出内容类型（决定前端渲染方式） |
| `content` | string | AI 生成的 Markdown 文本内容 |
| `quickActions` | array | 可选的追问快捷操作（最多 2 个） |

#### 错误响应

```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "未找到对应任务"
  }
}
```

错误码：

| Error Code | 说明 |
|------------|------|
| `UNAUTHORIZED` | 未登录 |
| `INVALID_REQUEST` | 请求体格式错误 |
| `TASK_NOT_FOUND` | taskId 不存在 |
| `TASK_GROUP_NOT_FOUND` | taskGroupId 不存在 |
| `EMPTY_REQUEST` | userRequest 为空 |
| `AI_ASSIST_FAILED` | AI 调用失败 |
| `AI_RESPONSE_INVALID` | AI 响应格式错误 |
| `RATE_LIMITED` | 请求过于频繁 |

### 7.2 辅助 API：POST /api/task-assist/suggest（V2.5）

> **此 API 属于 V2.5 范围，V2.4 MVP 不实现。** 此处列出完整设计供 V2.5 架构方案引用。

用于生成动态快捷入口建议。

#### 请求结构

```json
{
  "taskGroupId": "uuid-of-task-group",
  "taskId": "uuid-of-task"
}
```

#### 响应结构

```json
{
  "success": true,
  "data": {
    "suggestions": [
      { "label": "生成 30 秒自我介绍", "request": "帮我生成 30 秒的自我介绍" },
      { "label": "改成更适合运营岗", "request": "帮我把自我介绍改成更适合运营岗位" },
      { "label": "列出面试官可能追问", "request": "列出面试官听完自我介绍后可能会追问的问题" }
    ]
  }
}
```

**注意**：
- 此 API 可缓存——同一个 task 的动态建议在一次会话中不变
- 如果 AI 调用失败，返回空 `suggestions: []`，前端只显示通用快捷入口
- 这是一个轻量 API，响应时间应该 < 2s

### 7.3 API 应该以什么作为主要输入

**必须是 `userRequest: string`（用户自由输入），不是 `assistType: enum`（固定类型选择）。**

原因：

1. **assistType 枚举会固化产品方向**。一旦 API 定义了 `assistType: "start" | "unblock" | "simplify" | ...`，前端就会围绕这个枚举设计 UI，最终演变成固定按钮菜单。
2. **枚举追不上用户需求**。用户说"帮我写请假文案"，这不属于 start / unblock / simplify / template 任何一个。如果后端只能接收枚举值，就无法处理这类请求。
3. **LLM 天然适合理解自然语言**。不需要把用户输入预先归类到固定桶里——LLM 可以同时完成意图理解和内容生成。

**所以：API 以 userRequest 为主要输入，intent 仅作为响应中的标签，不作为请求中的选择项。**

### 7.4 API 安全

- **必须登录才能调用**：复用现有 `getAuthenticatedUserId()` 机制
- **必须校验 taskId 属于当前用户**：防止用户通过修改 taskId 访问他人任务
- **rate limit**：建议同一用户每分钟最多 10 次调用
- **userRequest 长度限制**：最多 500 字符

#### 验收重点

- 跨任务上下文不会让 AI 输出变发散
- 条件触发不打扰用户（频率控制）
- AI 辅助后任务完成率提升可测量

### 9.4 V3.0：AI 教练 1.0

#### 定位

V3.0 是 AI Todo 的产品愿景版本——从"工具"升级为"教练"。

## 十、数据与存储边界

### 10.1 V2.4 MVP 不保存长期聊天

V2.4 是 AI 辅助执行的首次交付，核心目标是验证产品方向，**不做任何长期数据存储**：

| # | 约束 | 说明 |
|---|------|------|
| 1 | **不新增聊天历史表** | 数据库 schema 零变更，不创建 `chat_history`、`assist_conversations` 等任何聊天相关表 |
| 2 | **不保存完整多轮聊天正文** | 用户输入和 AI 回复不写入任何数据库表 |
| 3 | **面板关闭即清空** | TaskAssistPanel 关闭后，本次辅助的所有上下文（用户输入、AI 输出、追问记录）全部丢弃 |
| 4 | **conversationContext 仅用于当前面板内追问** | `conversationContext` 只在前端内存中暂存，用于支持面板内的 1 次追问；面板关闭后即释放，不进入长期存储 |
| 5 | **不做聊天历史搜索、导出、归档** | V2.4 不提供任何聊天记录查询、导出或归档功能 |

**一句话：V2.4 的 AI 辅助是"阅后即焚"的临时辅助空间。**

### 10.2 如果后续需要保存，只保存轻量辅助记录

V2.6 如果确实需要追踪辅助效果（例如评估 AI 辅助是否真的帮助了用户完成任务），**只能保存轻量辅助记录**，不能保存完整聊天正文。

**可以保存的轻量字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `taskId` | uuid | 关联的任务 ID |
| `taskGroupId` | uuid | 关联的任务组 ID |
| `intent` | string | AI 识别的意图类型（如 `first_step`、`generate_copy`） |
| `createdAt` | timestamp | 辅助记录创建时间 |
| `whetherCopied` | boolean | 用户是否复制了 AI 输出 |
| `whetherHelpful` | boolean | 用户是否标记为"有帮助"（可选，需前端配合） |
| `summary` | string（≤200 字符） | AI 输出的摘要（一行描述，不保存完整正文） |

**明确禁止保存的内容**：

| 禁止保存 | 原因 |
|----------|------|
| 完整多轮聊天正文 | 隐私风险 + 存储膨胀 + 产品方向退化 |
| 用户所有 `userRequest` 原文长期记录 | 用户输入可能包含敏感信息（如真实姓名、公司名、具体事件） |
| AI 的 `assistant` 完整回答长期记录 | 回答内容可能很长，且随 AI 模型升级而过时 |
| 跨任务长期聊天上下文 | 违反"受限任务对话"的产品边界，退化为聊天记录系统 |

### 10.3 原因

为什么必须严格控制数据存储边界：

| 原因 | 说明 |
|------|------|
| **降低数据库复杂度** | 不新增聊天表意味着不需要设计表结构、索引、迁移脚本、归档策略。V2.4 零数据库变更，实现成本最低 |
| **避免隐私风险** | 用户可能向 AI 描述真实的个人情况（"帮我写请假条"、"帮我准备面试"）。这些内容一旦长期存储，就变成隐私数据，需要合规处理、数据加密、用户删除权等 |
| **避免产品变成聊天记录系统** | 一旦开始保存聊天记录，产品团队就会被要求做"聊天历史搜索"、"聊天导出"、"聊天归档"，产品方向从"任务推进器"退化为"聊天记录管理工具" |
| **保持任务导向** | AI 辅助的目的是推进当前任务，不是积累对话。每次打开面板都是新的开始，聚焦当前任务 |
| **降低 Review 和实现风险** | 零数据库变更意味着：不需要 migration、不需要回滚方案、不需要数据迁移、不需要 Review schema 设计 |

### 10.4 版本边界

| 版本 | 数据库变更 | 聊天存储 | 说明 |
|:---:|:---:|:---:|------|
| **V2.4** | 零变更 | 不保存 | AI 辅助"阅后即焚"，面板关闭即清空 |
| **V2.5** | 零变更 | 不保存 | 可增强动态建议和 intent 识别，但仍不做长期聊天历史 |
| **V2.6** | 如需保存，仅轻量记录 | 仅轻量辅助记录 | 不保存完整聊天正文。如确认不需要辅助效果追踪，V2.6 也可以零变更 |
| **V3.0** | 待定 | 仅轻量辅助记录 | 即使进入行动教练系统，也**不能变成完整聊天记录系统**。核心仍是任务导向，不是对话导向 |

> **底线：AI Todo 永远不会成为聊天记录系统。辅助对话是临时的、围绕任务的、完成即丢弃的。**

## 十一、明确不能做的事情

### 11.1 产品层面不能做

| # | 不能做 | 原因 |
|---|--------|------|
| 1 | **不做自动执行 Agent** | AI Todo 不替用户完成任务。不能自动发邮件、自动提交代码、自动发消息。这是产品边界 |
| 2 | **不做固定辅助按钮菜单** | 固定选项无法覆盖真实用户需求。产品方向应该是自由输入 + AI 理解 |
| 3 | **不做全局 AI 聊天页** | 一旦做了全页面聊天，产品就从"任务辅助"变成了"ChatGPT 套壳"。差异化消失 |
| 4 | **不做情绪陪聊** | AI Todo 是行动教练，不是情感陪伴。不做"我今天好累" → "抱抱你"这类对话 |
| 5 | **不做通用搜索** | 不做"帮我搜索一下 React 18 的新特性"。可以用浏览器搜的东西不需要 AI Todo |
| 6 | **不做 AI 代办** | 不做"AI 自动帮你把任务标记为完成"。用户必须亲自判断和勾选 |
| 7 | **不做团队协作 AI 辅助** | AI Todo 是单用户工具。不做"帮我把这个任务分配给张三" |
| 8 | **不做长期聊天记录** | 辅助对话是临时的、围绕任务的。不做聊天历史搜索、导出、归档 |

### 11.2 技术层面不能做

| # | 不能做 | 原因 |
|---|--------|------|
| 1 | **不做 Streaming（V2.4 MVP）** | MVP 用普通 JSON 响应，降低实现复杂度。V2.5+ 可重新评估 |
| 2 | **不做独立 AI 聊天数据库** | 不新增聊天记录表。MVP 不保存对话历史 |
| 3 | **不做前端本地模型** | 不使用 WebLLM / transformers.js。AI 调用全走服务端 |
| 4 | **不做离线 AI 辅助** | 需要网络 |
| 5 | **不新增第三方 AI SDK** | 复用现有 `callAIService`（OpenAI-compatible DeepSeek API） |

### 11.3 交互层面不能做

| # | 不能做 | 原因 |
|---|--------|------|
| 1 | **不做任务卡片上的多个 AI 按钮** | 每个任务只有一个 AI 入口。多个按钮会拥挤并误导用户 |
| 2 | **不做全屏 AI 聊天页** | AI 辅助在面板内完成，不跳转到独立页面 |
| 3 | **不做 AI 辅助浮窗** | 不用 floating action button / 悬浮球。AI 入口属于任务卡片 |
| 4 | **不做自动弹出 AI 面板** | 不自动打断用户。AI 辅助始终由用户主动触发 |

## 十二、和 V2.2B/C/D UI 升级的关系

### 12.1 关系说明

AI 辅助执行（V2.4+）和 V2.2B/C/D UI 升级是**两个独立的产品维度**：

| 维度 | V2.2B/C/D UI 升级 | AI 辅助执行 V2.4+ |
|------|-------------------|-------------------|
| **维度** | 视觉与体验层 | 功能与智能层 |
| **改什么** | 组件样式（className + CSS 变量） | 新增组件 + 新增 API + 新增 Prompt |
| **不改什么** | hooks / API / lib / 数据库 | 现有 hooks / API / lib / 数据库 |
| **目标** | 让已有功能更好看、更顺手 | 新增一个产品能力 |
| **依赖** | 仅依赖 V2.2A 路由结构 | 依赖 V2.2A 路由结构 + V2.2C 组件结构 |

### 12.2 为什么 V2.4 放在 V2.2 / V2.3 之后

| 原因 | 说明 |
|------|------|
| **产品基础先稳固** | V2.2 让页面结构清晰、视觉统一。V2.3 让账号安全。这些基础不稳固时加 AI 辅助，产品质量会打折扣 |
| **组件结构先成型** | V2.2C 会重新设计 TaskItem 的样式。如果 V2.4 的"AI 帮助"按钮在 TaskItem 改版前加上去，改版时可能需要重新设计按钮位置和样式。V2.2C 可以直接预留入口 |
| **用户习惯先建立** | 用户先习惯三页面结构（/ /login /app）和新的 UI 风格，再加 AI 辅助功能时不会觉得"产品变化太大" |
| **AI 辅助的入口依赖 TaskItem** | V2.4 的"AI 帮助"按钮放在 TaskItem 上。V2.2C 会重新设计 TaskItem 卡片样式，V2.4 的入口设计应基于 V2.2C 的成品 |

### 12.3 V2.2C 应该为 V2.4 预留什么

V2.2C 在重新设计 TaskItem 时，应该**在 TaskItem 右侧预留一个操作区**：

- 预留一个 icon-button 位置（当前可能是空的或放其他操作）
- 不要求 V2.2C 实现这个按钮——只是视觉上预留空间
- V2.4 可以直接在这个预留位放置"AI 帮助"入口

这是 V2.2C 和 V2.4 之间唯一的衔接点。除此之外，V2.2B/C/D 和 V2.4 完全独立。

### 12.4 执行顺序

```
V2.2A（路由结构）✅
    │
    ▼
V2.2B（登录页设计）
    │
    ▼
V2.2C（主工作台 UI 美化）  ← 在 TaskItem 右侧预留操作区
    │
    ▼
V2.2D（移动端优化）
    │
    ▼
V2.3（安全增强）
    │
    ▼
V2.4（AI 辅助执行 MVP）  ← 在这里实现"AI 帮助"入口 + TaskAssistPanel
    │
    ▼
V2.5 → V2.6 → V3.0
```

## 十四、风险与注意事项

### 14.1 风险矩阵

| # | 风险 | 等级 | 影响 | 缓解措施 |
|---|------|:---:|------|---------|
| 1 | **自由输入导致 AI 输出质量不可控** | **P1** | 用户输入"帮我写一篇论文"，AI 无法在受限任务上下文中提供有效帮助 | System Prompt 明确边界：如果用户请求超出任务范围，AI 应礼貌说明并引导回任务 |
| 2 | **用户期望 AI 替自己完成任务** | **P1** | 用户说"帮我做这个"，AI 给出步骤后用户不满"你为什么不帮我做完" | 产品文案和 AI 输出结尾统一加"以上建议供参考，最终由你判断和完成" |
| 3 | **AI 辅助增加 latency，影响体验** | **P2** | `/api/task-assist` 调用 DeepSeek API 需要 2-5s，面板等待时间过长 | MVP 用普通 JSON 响应 + loading spinner；不做 streaming（复杂度和延迟不是正比） |
| 4 | **TaskAssistPanel 使 TaskItem 组件变复杂** | **P2** | TaskItem 从简单的卡片变成带面板、带状态管理的复杂组件 | TaskAssistPanel 是独立组件，TaskItem 只管理面板的展开/关闭状态 |
| 5 | **动态建议 API 调用失败影响面板体验** | **P2** | suggest API 失败 → 面板无法显示动态快捷入口 | 静默回退：suggest 失败时只显示 3 个通用快捷入口，不弹错误提示 |
| 6 | **用户滥用 AI 辅助（无限制调用）** | **P2** | 用户在一个任务上反复调用 AI 辅助，消耗 API 额度 | rate limit（每分钟 10 次）+ 每次面板打开后最多追问 1 轮（MVP） |
| 7 | **产品方向被误解为"ChatGPT 套壳"** | **P2** | 外部评价"这不就是 ChatGPT 吗" | 产品定位文档明确差异化价值；UI 设计强调任务绑定和完成导向；不做全页聊天 |

### 14.2 产品方向防退化

**以下行为必须避免，否则产品会退化为"ChatGPT 套壳"**：

| 退化行为 | 为什么危险 |
|----------|-----------|
| 把 AI 辅助面板做成全屏聊天页 | 失去任务绑定感，用户感知变为"聊天机器人" |
| 去掉 taskId / taskGroupId 绑定 | 失去任务上下文，AI 变成通用回答 |
| 允许多轮无限对话 | 失去完成导向，AI 会一直聊下去 |
| 增加聊天历史功能 | 用户开始把辅助面板当聊天工具，而不是任务推进器 |
| AI 输出不强调"用户最终执行" | 用户期望 AI 替自己完成，产品边界模糊 |

### 14.3 AI 安全

| 维度 | 措施 |
|------|------|
| **Prompt Injection** | userRequest 长度限制 500 字符；System Prompt 中使用明确的结构化输出指令 |
| **内容安全** | 复用现有 `input-validator.ts` 的 `containsHighRiskPatterns`；AI 输出不包含外部链接 |
| **数据隔离** | taskId 必须通过服务端校验归属当前 user_id |
| **Token 消耗** | userRequest 限制 500 字符；AI 输出限制 2000 字符；conversationContext 最多 2 轮 |

### 15.2 关键决策摘要

| # | 决策 | 结论 |
|---|------|------|
| 1 | 功能名称 | **任务级受限 AI 助手** |

### 15.3 一句话总结

> **AI Todo 后续不是变成一个能聊天的 ChatGPT，而是在每个任务旁边放一个"知道你是谁、知道你要做什么、知道怎么帮你推进"的受限 AI 助手。自由输入是核心，任务绑定是灵魂，完成导向是底线。AI 负责铺路，用户负责迈步。**
